Introduction & Overview

Why ML moved from research to mainstream, and what the lifecycle achieves

📘 Definition — ML Lifecycle
The ML lifecycle is the cyclic iterative process with instructions and best practices to use across defined phases while developing an ML workload. It adds clarity and structure for making a machine learning project successful.

Machine learning has evolved from research and development to the mainstream, driven by the exponential growth of data sources, generative AI, and scalable cloud-based compute resources. ML workloads enable algorithms to learn from data through iterative and continuous cycles.

Unlike traditional software, ML systems require continuous retraining. A model's accuracy degrades over time as real-world data drifts from training data. ⭐ Exam Tip
The ML lifecycle is described as cyclic and iterative — not linear or sequential. The phases can have feedback loops that interrupt and cycle back across the lifecycle. This is the most commonly tested conceptual point.
⚠️ Common Misconception
Many students think "deploy = done." In ML, deployment is just the beginning. The model must be actively monitored, and retraining must happen when performance degrades. Forgetting this is how you get a model that worked great at launch and fails silently 6 months later.

Three Levels of ML Software

Every ML system manages three intertwined assets: Data, Model, and Code

📘 Core Framework
Every ML-based software needs to manage three main assets: Data, Model, and Code. A change to one ripples through the others — this entanglement is why ML engineering is harder than data engineering or modelling alone.

Data Engineering

The initial step in any data science workflow is to acquire and prepare the data to be analyzed. Responsible for: data acquisition, cleaning, validation, and feature engineering.

🔬 Research Context (bitsnotes.com, 2026)
The data pipeline for ML spans 6 stages: Collection → Storage → EDA → Cleaning → ETL → Delivery. AWS services like S3 (storage), Glue (ETL), and Athena (querying) map directly to these stages in production systems.

Key principle: "Garbage In, Garbage Out" — the ML model is only as good as the data. Most time on an ML project is spent constructing datasets, cleaning, and transforming data.

Model Engineering

The core of the ML workflow — writing and executing ML algorithms to obtain an ML model. The model engineering pipeline is usually utilized by a data science team and includes operations leading to a final deployable model.

Key activities: feature engineering, algorithm selection, training, hyperparameter tuning, evaluation, and model packaging.

Code / Deployment Engineering

The final stage of delivering an ML project — integrating the ML model into the final product. This is often the most neglected layer but equally critical for production success.

Key activities: containerization (Docker), CI/CD pipelines, API creation, monitoring integration, and version control.

In industry, the three levels map to three operations disciplines: DataOps (data quality & pipelines), ModelOps (model training & registry), and DevOps (code deployment & infrastructure). Together they form MLOps — the practice of applying DevOps principles to ML systems. Companies like Google, Microsoft, and AWS have built entire platforms (Vertex AI, Azure ML, SageMaker) around this framework.

⚠️ Data Snooping Bias
If you select a particular ML model using the test set, you are guilty of data snooping. This leads to over-optimistic model selection that will not perform well in production. Set aside the test set at the very beginning and never look at it until final evaluation.
🔬 Research — Data Sources in Practice (bitsnotes.com, 2026)
Modern ingestion architectures use: streaming (Apache Kafka, AWS Kinesis for real-time data), batch ETL (AWS Glue for scheduled processing), and APIs (REST hooks from third-party data providers). IoT sensors, social networks, and time-series databases are common ML data sources.
🔍 Step 2: Exploration & Validation▼

Includes data profiling to obtain information about content and structure. Output is a set of metadata (max, min, avg of values). Data validation operations are user-defined error detection functions that scan the dataset for errors.

Attribute Profiling

  • Name, Number of Records, Data Type (categorical, numerical, int/float, text)
  • Numerical measures (min, max, avg, median)
  • Missing value ratio = absent values / total records
  • Type of distribution (Gaussian, uniform, logarithmic)
💡 Tool Recommendation — Exploratory Data Analysis
Using Jupyter notebooks is a good way to keep records of data exploration and experimentation. In cloud contexts, AWS SageMaker provides managed notebooks. For big data, Amazon Athena can run SQL-based EDA at petabyte scale without downloading data.

Example — Address Validation: Are address components consistent? Is the correct postal code associated with the address? Are there missing values?

⚠️ Validation vs. Exploration
Validation is formal (automated rules checking for schema violations, range violations, null checks). Exploration is analytical (understanding distributions, spotting anomalies visually). Both happen in this step. Validation guards the training pipeline — data problems caught here don't poison the model.
🧹 Step 3: Data Wrangling (Cleaning)▼

Data preparation step where we programmatically wrangle data — reformatting or restructuring particular attributes that might change the form of the data's schema. Recommend writing scripts/functions for all data transformations in the pipeline to reuse on future data.

Key Transformations

💡 Industry Practice — Data Wrangling Tools
Tools like AWS Glue DataBrew, Pandas, and Apache Spark are standard. AWS Glue ETL jobs handle large-scale cleaning at petabyte scale. For interactive cleaning, SageMaker Data Wrangler provides a no-code/low-code visual interface. (Source: bitsnotes.com, 2026)
✂️ Step 4: Data Splitting▼

Splitting the data into training (80%), validation, and test datasets to be used during the core machine learning stages to produce the ML model.

⭐ Exam Tip — Split Ratios
The typical ratio is 80% training / 10% validation / 10% test, though 70/15/15 is also common. The test set must NEVER be used during model development — only for final "Model Acceptance Test." Data leakage occurs when test data leaks into training; remove duplicates before splitting to avoid it.
⚠️ Data Leakage
Data leakage occurs when information from the hold-out test dataset leaks into the training data. This makes model performance look better than it will be in production. One way to avoid this: remove duplicates before splitting the data.
🔬 Industry Research — Athena Partitioning as a Data Pipeline Technique (bitsnotes.com, 2026)
In AWS-based ML pipelines, horizontal partitioning (by date/month) and vertical partitioning (by categorical column) dramatically reduce query costs and time. Example benchmark: querying 9.32 GB full dataset took ~29 seconds and cost ~$0.047. After date partitioning (January only), data scanned dropped to 815 MB (91% reduction), time to ~7 seconds. After further vertical partitioning by payment_type, scanning dropped to 71 MB and ~1.8 seconds. Design principle: partition on columns that appear most in WHERE clauses.

📋 Phase 1 & 2: Business Goal + ML Problem Framing

Business Goal Identification▼

The most important phase of the ML lifecycle. An organization must have a clear idea of the problem to be solved and the business value to be gained.

Business Considerations

  • Understand business requirements
  • Align affected stakeholders
  • Form a business question
  • Identify critical, must-have features
  • Consider new business processes that might emerge
  • Identify how business value can be measured using business metrics

Cost and Performance Optimization

  • Evaluate cost of data acquisition, training, inference, and wrong predictions
  • Evaluate whether external data sources might improve model performance
  • Establish pathways to production
  • Review how to handle ML-generated errors
⭐ Homework Alert
From the lecture: Find out what "Definition of Done" and "Acceptance Criteria" mean in software/ML projects. These are important project management concepts for ML.
ML Problem Framing▼

The business problem is framed as an ML problem: what is observed (inputs) and what should be predicted (label or target variable).

Example — Manufacturing Company

Goal: Maximize profits. Several ML approaches: (1) Forecast sales demand for existing products, (2) Forecast required input materials to reduce stock, (3) Predict sales for new products. Each frames the same business goal differently as an ML problem.

Framing Steps

  1. Define criteria for a successful outcome
  2. Establish observable, quantifiable performance metric (e.g., accuracy)
  3. Define relationship between technical metric and business outcome
  4. Get stakeholder agreement on metrics
  5. Formulate as: inputs → desired outputs → performance metric
  6. Map technical outcome to business outcome

Data Processing Deep Dive

Preprocessing, Feature Engineering, and the science of preparing ML data

🔧 Data Preprocessing Operations

Data Cleaning — Remove Outliers, Duplicates, Impute Missing

Detect duplicates, replace inaccurate or irrelevant data, correct missing data using imputation techniques that minimize bias.

💡 Classroom Example
Consider 40 Grade 4 students calculating average age: (1) Same student entered 3 times = duplicate, (2) Age entered as 100 instead of 10 = outlier, (3) 2 students not entered = missing values. Each scenario corrupts the result differently.

Partitioning — Train / Validate / Test Split

To prevent overfitting and evaluate models accurately, randomly split data into train, validate, and test sets.

⭐ Exam Tip
Usual rule: 80% train, 10% validate, 10% test. Data leakage = when test data info leaks into training data. Prevention: remove duplicates BEFORE splitting.

Scaling — Normalization and Standardization

Normalization: scales values to [0, 1] range. Used when features must have equal importance with different ranges.

Standardization: transforms to mean=0, std=1. Better for handling outliers.

📘 When to Use Each
Use normalization for neural networks and distance-based algorithms (KNN, SVM). Use standardization when outliers are present or when algorithm assumes Gaussian distribution (Linear Regression, Logistic Regression, PCA).

Bias Detection and Mitigation

Biases are imbalances in prediction accuracy across different groups (e.g., age, income bracket). Can come from data (historical inequities) or algorithm (optimization objective not capturing fairness).

⚠️ Real-World Bias Examples
Amazon's 2018 hiring AI was biased against women (trained on historical male-dominated hiring data). Facial recognition systems have higher error rates for darker-skinned individuals. These are direct consequences of biased training data.

Data Augmentation

Artificially increases training data by synthesizing new data from existing data. Common in image classification (flipping, rotation, cropping), NLP (synonym replacement, back-translation), and time-series (noise injection).

Benefits: reduces overfitting, improves generalization to unseen data.

⚙️ Feature Engineering

📘 What is a Feature?
Every unique attribute of the data is considered a feature (also known as an attribute). Feature Engineering is the process of selecting and transforming variables when creating a predictive model.
Feature Creation▼

Creating new features from existing data to assist with better predictions.

Examples: one-hot encoding (convert categories to binary vectors), binning (convert continuous to categorical), splitting (extract parts of strings), calculated features (e.g., BMI from height+weight).

Feature Transformation & Imputation▼

Replacing missing features or invalid features. Techniques include: Cartesian products of features, non-linear transformations (binning numeric variables into categories), and creating domain-specific features (e.g., "day of week" from a timestamp for retail models).

Feature Extraction (Dimensionality Reduction)▼

Reducing the amount of data to be processed using dimensionality reduction techniques, while maintaining original data characteristics.

TechniqueFull NameUse Case
PCAPrincipal Component AnalysisLinear dimensionality reduction, uncorrelated features
ICAIndependent Component AnalysisSeparating independent sources (e.g., audio signal separation)
LDALinear Discriminant AnalysisSupervised reduction, maximize class separability
Feature Selection▼

Selecting a subset of extracted features that is relevant and minimizes model error rate. Tools: feature importance score (tree-based models) and correlation matrix (remove highly correlated redundant features).

🏋️ Model Training & Tuning

Algorithm Selection▼

Train many ML models from different categories using standard parameters. Factors: success metrics, model explainability, compute requirements (training/prediction time, memory).

💡 Algorithm Categories to Try
Linear regression, logistic regression, k-means, naive Bayes, SVM, Random Forest. For each: use N-fold cross-validation, compute mean and std of performance metric on N folds. Select top 3–5 most promising models, preferring models that make different types of errors (for ensembling).
Distributed Training (Data & Model Parallelism)▼
TypeWhat is SplitUse Case
Data ParallelismTraining dataset split into mini-batches across nodes. Each node trains on fraction of data.Standard large-scale training; model fits in one GPU
Model ParallelismModel itself split across multiple instances/nodesExtremely large models (GPT-scale) that don't fit in single GPU memory
🔬 Research — Real Costs of Distributed Training (2024–2025)
Training GPT-3 (175B params) cost ~$4.6M. Training LLMs is growing in cost — NLP models use billions of parameters and this number keeps increasing. High CO₂ emissions due to heavy compute makes large-scale training a sustainability concern. Custom hardware-aware HPO techniques are needed to co-optimize for accuracy AND hardware efficiency.
Hyperparameter Tuning▼

Hyperparameters are settings that control the behavior of the ML algorithm (not learned from data). Must be set before training begins.

Examples: learning rate, number of epochs, hidden layers, hidden units, activation functions.

⭐ Exam Tip — HPO Search Methods
Random search is preferred over grid search for hyperparameter optimization. Also note: data transformation choices ARE hyperparameters. HPO task size grows exponentially with each new hyperparameter (adds a new search dimension).
⚠️ HPO Challenge
HPO is very expensive and resource-heavy in practice, especially for deep learning. At AirBnB: team started with a complex deep learning model and faced complexity, many development cycles, and high hardware requirements. Solution: start simpler, iterate.
Debugging & Profiling▼

A training job can have problems: system bottlenecks, overfitting, saturated activation functions, vanishing gradients. A debugger provides visibility through monitoring, recording, and analyzing data. It captures state at periodic intervals.

✅ Model Evaluation & Testing

Validate the trained model to ensure it meets original business objectives before serving in production. Use the hold-back test dataset to estimate generalization error via the final "Model Acceptance Test."

⭐ Exam Tip — Model Verification Challenges
Booking.com realized required business value was NOT achieved after deploying 150 models. Challenge: defining right KPIs beyond just model performance metrics — must understand engineering AND business angles. Full-scale real-world testing is often impossible for safety/security/scale reasons → simulated testing with assumptions.

📦 Step 4: Model Packaging

Exporting the final ML model into a specific format (e.g., PMML, PFA, or ONNX) that describes the model to be consumed by the business application. The ML model should work outside of the model-training environment — be executable as an independent asset.

Model Deployment

Deploying the pipeline for automated retraining AND providing prediction API

📘 Definition — Model Serving
Model Serving is the process of deploying the ML model in a production environment. Deploying an ML system includes two aspects: (1) deploying the pipeline for automated retraining, (2) providing an API for prediction on unseen data.
💡 What Inference Requires
Inference (getting predictions from a deployed model) requires: a model, an interpreter for execution, and input data.

🐳 Deployment as Docker Containers

No standard, open solution to ML model deployment exists. However, containerization is the de-facto standard for delivery. ML model inference is stateless, lightweight, and idempotent → ideal for containers.

Docker-based Deployment▼

Package the whole ML tech stack (dependencies) and model prediction code into a Docker container. Orchestration via Kubernetes or AWS Fargate. Prediction exposed through REST API (e.g., Flask application).

# Simplified Docker deployment structure
FROM python:3.10-slim
WORKDIR /app
COPY requirements.txt .
RUN pip install -r requirements.txt
COPY model.pkl .
COPY app.py .
EXPOSE 8080
CMD ["python", "app.py"]
Serverless Functions (Cloud-native)▼

Cloud vendors provide ML platforms: Amazon SageMaker, Google Cloud AI Platform, Azure ML Studio, IBM Watson ML. Serverless options: AWS Lambda, Google App Engine, Azure Functions.

Code + dependencies packaged into .zip files with single entry point function. Managed by cloud provider.

⚠️ Serverless Constraint
Pay attention to artifact size limits on serverless deployments. AWS Lambda has a 250MB deployment package limit (unzipped), which can be exceeded by large ML libraries like TensorFlow.

Monitoring & Drift Detection

Keeping the model alive after deployment

💡 Why Monitoring is Critical
"The biggest failures come after deployment." Teams celebrate a great eval score and stop watching; drift then degrades the model in silence. Monitoring + automated retraining isn't optional — it's the lifecycle. (bansal-ai.in, 2026)

🔍 Types of Drift

Drift TypeDefinitionExampleDetection Method
Data DriftSignificant changes to the input data distribution compared to training dataCustomer demographics shift over timeStatistical tests (KS test, PSI)
Concept DriftProperties of the target variable change — the relationship between inputs and outputs shiftsConsumer behavior changes after COVID; fraudsters change tacticsMonitor prediction accuracy over time
Feature DriftIndividual feature distributions changeA sensor starts returning slightly different range valuesFeature-level monitoring dashboards
📘 Model Decay / Model Staleness
When a deployed model remains constant but real-world data changes, it becomes stale. This phenomenon is called 'model decay' and should be carefully monitored. It's why offline learning (static models) requires scheduled retraining.

📋 What to Monitor

Issues Detected in Monitoring Phase▼
  • Data Quality — Schema changes, null increases, value range violations
  • Model Quality — Accuracy, precision, recall degradation
  • Bias Drift — Model becomes biased over time for certain demographic groups
  • Feature Attribution Drift — Feature importance changes (SHAP values)
  • Prediction Bias — Systematic over- or under-prediction
  • Outlier Detection — Deploy outlier detector in production
💡 Model Explainability in Monitoring
Monitoring system uses explainability (e.g., SHAP — SHapley Additive exPlanations) to evaluate whether the model predictions can be trusted, not just whether they're accurate. A model might be accurate on average but rely on spurious correlations that break under distribution shift.

🔄 Model Update Pipeline

If the alarm manager identifies violations, it launches the model update pipeline for a re-train. Re-training at intervals minimizes the risk of data and concept drift. A scheduler can initiate re-training at business-defined intervals. Data preparation, CI/CD/CT, and feature pipelines are active during this process.

⭐ Exam Tip — Monitoring Scope
The monitoring system captures data, compares to training set, defines rules to detect issues, and sends alerts. This process repeats on a defined schedule, when initiated by an event, or by human intervention. The four issues detected: data quality, model quality, bias drift, feature attribution drift.

Model Serialization Formats

Exporting trained models for deployment and interoperability

📘 Why Serialization Matters
To achieve a distributable format, the ML model must be present and executable as an independent asset. For example, we might want to use a Scikit-learn model in a Spark job — this means ML models must work outside the model-training environment.

🌐 Language-Agnostic Formats

ONNX — Open Neural Network eXchange ⭐ Most Important▼

ML framework independent file format. Created to allow any ML tool to share a single model format. Supported by Microsoft, Facebook, Amazon, and many others.

Once serialized in ONNX format → consumed by onnx-enabled runtime libraries (inference engines) → make predictions.

⭐ Exam Tip
ONNX is the modern standard for model interoperability. If a question asks about cross-framework model sharing, ONNX is the answer.
PMML — Predictive Model Markup Language▼

XML-based format with extension .pmml. Standardized by the Data Mining Group (DMG). Describes model and pipeline in XML. Limitation: does not support all ML algorithms; limited in open-source tools due to licensing issues.

PFA — Portable Format for Analytics▼

Designed as a replacement for PMML. JSON-formatted string that describes an executable "scoring engine." Each engine has well-defined input, output, and functions. Requires a PFA-enabled environment to run.

📐 Architecture Patterns

Forecast Pattern (Academic / Kaggle)▼

Widely spread in academic research and data science education (Kaggle, DataCamp). Take an available dataset, train a model, run on historical data, make predictions.

⚠️ Not for Production
The Forecast pattern is the easiest way to create an ML system but is not common in industry for production systems (e.g., mobile applications). It doesn't have a feedback loop or monitoring.
Web-Service Pattern (Most Common in Industry)▼

The most commonly described deployment architecture for ML models. Web service takes input data and outputs a prediction. Model trained offline but serves real-live prediction data. Model stays constant until re-trained and re-deployed.

⭐ Exam Tip
Web-service is offline training + real-time prediction. The model doesn't update at runtime — only during scheduled retraining cycles.
Online Learning / Real-time Streaming Analytics▼

The most dynamic way to embed ML in a production system. The system learns about new data on the fly — ML model is incrementally re-trained with new data. This continually re-trained model is instantly available as a web service.

💡 Naming Note
"Online learning" is a confusing name because core training is usually NOT performed on the live system. Better to call it incremental learning. Works well with lambda architecture in big data systems.
⚠️ Critical Drawback
If bad data enters the system, the ML model AND whole system performance will increasingly decline. This is why monitoring and data validation are especially critical for streaming architectures.
AutoML Pattern▼

A sophisticated version of online learning. Promises training ML models with minimal effort and without ML expertise. User provides data; AutoML system automatically selects an ML algorithm and configures it.

🔬 AutoML in Practice (2024–2025)
Major AutoML platforms: Google AutoML, AWS SageMaker Autopilot, Azure AutoML, H2O.ai Driverless AI. Trade-off: AutoML reduces expertise requirements but produces "black box" models that can be harder to debug and explain. Emerging: LLM-driven AutoML using generative AI code tools.

Model Serving Patterns

How to package and expose ML models to the world

Model-as-a-Service (MaaS)▼

Common pattern for wrapping an ML model as an independent service. Wrap the model and interpreter within a dedicated web service. Applications request predictions via REST API or gRPC service.

💡 Classroom Example
"Do you use Gemini or Copilot?" — These are Model-as-a-Service. You send a request via API, the model processes it in the cloud, and returns a response. You don't know (or need to know) how the model runs.
Model-as-a-Dependency▼

The most straightforward way to package an ML model. A packaged ML model is considered as a dependency within the software application — like an SDK or library.

💡 Classroom Example
Think of it like import tensorflow or using ML.Model — the model is bundled with your code and shipped as part of the application. Used in mobile apps (Core ML on iOS).
Precompute Serving▼

Tightly related to the Forecast workflow. Use an already trained model to precompute predictions for incoming batch of data. Resulting predictions are persisted in the database. For any input request → query the database to get the prediction result (not the model).

💡 Use Case
Product recommendations on e-commerce: compute personalized recommendations for all users every night and store in a DB. At request time, just fetch from DB — extremely fast, no model serving latency.
Model-on-Demand (Message Broker Architecture)▼

Similar to Model-as-Dependency but available at runtime via a message-broker architecture.

Components:

  • Broker Component: Contains event channels (message queues). A process writes prediction-requests to input queue.
  • Event Processor Component: Contains model serving runtime + ML model. Reads requests from queue, sends to model, writes predictions to output queue.

Queued prediction results are pushed to the service that initiated the prediction request.

Federated Learning (Hybrid Serving) ⭐ Advanced Topic▼

Also known as hybrid serving. As many models as users exist, in addition to one held on a server. Starts with a unique model on the server (trained once with real-world data, sets initial model for each user).

How It Works:

  1. Server sends current model to all devices
  2. Each device trains on its OWN local data (never shared)
  3. Devices send only model weights/updates back to server (not raw data)
  4. Server aggregates updates → creates new global model
  5. New model sent back to all devices as new initial model
💡 Classroom Example (B. Sunil, 30-05-2026)
The spell checker on your Google keyboard. When you type "good" → suggests "good morning." The model trained on your local typing patterns without your private text ever leaving your device.
BenefitsConstraints
Personal data never leaves deviceMobile devices are less powerful
Train accurate models without storing personal data in cloudTraining data distributed across millions of devices not always available
Privacy-preserving by designRequires TensorFlow Federated (TFF) — lightweight TensorFlow
⭐ Exam Tip — Federated vs. Reinforcement Learning
Federated Learning: Multiple devices train models locally; share only model weights with central server. Privacy-preserving, distributed.

Reinforcement Learning: Agent learns by receiving immediate feedback from environment (reward/penalty signals). About learning through trial and error.

These are completely different concepts — a common exam trap.

📘 The Three Ops in ML
  • DataOps: Focuses on data quality, governance, and pipelines — gathering, cleansing, and prepping data
  • ModelOps: Focuses on model training, registry, and versioning — turning data into trained models
  • DevOps: Focuses on deployment infrastructure — deploying models into real-world environments via Inference API

🔄 The MLOps Cycle

DataOps → ModelOps: From Training Set to Trained Model▼

The process starts with DataOps — data is gathered, cleansed, and prepared, resulting in a comprehensive training set. This training set forms the crucial bridge to ModelOps. The prepared data is then employed for training and fine-tuning models.

ModelOps → DevOps: Using the Trained Model in Production API▼

The refined model (product of DataOps + ModelOps collaboration) transitions into DevOps. The model is deployed into a real-world environment via an Inference API, translating ML capabilities into practical applications.

DevOps → DataOps: Feedback and Retraining for Quality▼

Track model performance, identifying and addressing drift, bias, and accuracy issues. Detected anomalies prompt a loop back to DataOps for training set adjustments → model retraining via ModelOps → redeployment into Inference API via DevOps.

💡 This is Why It's a Loop
Every production anomaly detected by DevOps monitoring flows back to DataOps. This feedback loop is what makes ML systems self-improving over time — if the loop is properly implemented and automated.
🔬 MLOps Maturity Levels (Industry Standard, 2024–2025)
Google defines 3 MLOps levels: Level 0 (manual, no automation — common in academia), Level 1 (ML pipeline automation — automated training), Level 2 (CI/CD pipeline automation — automated model deployment). Most companies operate at Level 0 or 1. True Level 2 MLOps with full automation is rare and complex. Tools: MLflow, Kubeflow, Vertex AI, SageMaker Pipelines, Azure ML.

Deployment Failure Case Studies

Real challenges in deploying ML models to production

💡 The Hard Reality
There are significant differences between what works in academic settings and what is required by real-world systems. Certain bottlenecks and invalidated assumptions should always be expected. Primary focus: "How difficult is it to use?"

📊 Phase: Data Management

ChallengeDescriptionReal Example
Data CollectionDiscovering what & where data exists. Finding sources and understanding structure.Twitter: Follows "single responsibility principle" — every service responsible for single operation. At large scale, impossible to track what entity data is stored by which service and in which form.
Data PreprocessingJoining data from different schemas, conventions, storage methods into a single ML-suitable dataset.Firebird: Data joining was a massive challenge — had to join 12 datasets, each with different schemas and storage approaches.
Data AugmentationReal-world data often unlabeled. Labeling challenge due to limited expert access, absence of high-variance data, high volume.Manual labeling at scale requires hundreds of annotators; crowdsourcing introduces noise.
Data AnalysisToo few tools enabling efficient execution of data mining and profiling tasks at scale.Custom tooling often required; off-the-shelf tools don't fit specific ML problem needs.

🤖 Phase: Model Learning

ChallengeDescriptionReal Example
Model SelectionComplexity, many development cycles, high hardware requirementsAirBnB: Team started with complex deep learning model. Faced: high complexity, many dev cycles, high hardware requirements. Solution: start simpler.
Model Training CostNLP models using billions of parameters; high CO₂ emissionsOverall cost of training NLP models is only growing. GPT-3 training cost ~$4.6M in compute.
Hyperparameter OptimizationHPO size grows exponentially; very expensive for deep learningFor mobile devices: need customized hardware-aware optimization to co-optimize accuracy AND hardware efficiency.

✅ Phase: Model Verification

ChallengeDescriptionReal Example
Requirement EncodingDefining right KPIs — not just model performance but business valueBooking.com: Realized required business value was NOT achieved after deploying 150 models. Challenge: understanding both engineering AND business angles.
Test-based VerificationFull-scale real-world testing often impossible for safety/security/scale reasonsSolution: simulated testing with assumptions — but simulations definitely differ from real-world scenarios.
Formal VerificationIncreased regulatory control; model risk frameworks required for business decision-makingBanking/healthcare ML models require extensive test suites to understand model behavior.

🚀 Phase: Model Deployment

ChallengeDescriptionReal Example
IntegrationReuse of models; maintaining separate but similar models creates engineering overheadPinterest: Three models used similar embeddings but maintained separately → every embedding change had to be multiplied by three.
MonitoringAll monitoring tools require considerable investigation and implementation. Out-of-the-box tooling doesn't fit specific needs.The final ML solutions are so sensitive to problem specifics that custom monitoring tools are almost always required.
Updating / Concept DriftModels must be updated as the world changesFraud detection models become obsolete as fraudsters adapt. Recommendation models decay as user tastes evolve.
⭐ Exam Tip — Four Deployment Failure Types
Know these four technical failure patterns: Abstraction boundary erosion (clean separations break), Correction cascades (fixing one model breaks another), Pipeline jungles (tangled unmaintainable pipelines), Concept drift (world changes, model doesn't). The Pinterest, Booking.com, Twitter, Firebird, and AirBnB examples are all exam-testable.
