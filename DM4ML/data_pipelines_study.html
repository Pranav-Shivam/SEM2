Introduction & Why Data Pipelines?

Session 4 · Dr. D. Venkata Subramanian, BITS Pilani | DM4ML

📚 Core Definition
A Data Pipeline is a series of steps that allow data from one system to move and become useful in another system — particularly analytics, data science, or AI/ML systems. It works by pulling data from the source, applying rules for transformation and processing, then pushing data to its destination.

⚠️ Warning — The "Data Outlives Code" Problem
From the lecture scenarios on databases: when you deploy a new application version, you replace the old code in minutes. But the data written by the old code version may remain in the database for years. A value written 5 years ago by V1.0 may be read today by V5.0 — both forward and backward compatibility must be planned for.

Modes of Data Flow

Three fundamental ways data moves between systems

📚 Overview
Data flows between systems in three primary modes: Through Databases, Through Services, and Through Message Passing. Each mode has different coupling characteristics, latency profiles, and compatibility requirements.

Data Flow Through Databases

One process writes data to a shared database; another process reads from it. The database acts as an intermediary — without the two processes needing to know about each other.

Three Scenarios (from lecture)

ScenarioDescriptionCompatibility Requirement
Scenario 1Process 1 encodes (writes) → DB → Process 2 decodes (reads)Encoding/decoding contract between two processes
Scenario 2Single process V1.01 writes, V1.02 reads laterBackward compatibility — newer code must read older data
Scenario 3Multiple processes (1,2,3,4) all access the same DB concurrentlyForward + backward compatibility both required
🎯 Exam Trap
Students confuse Backward and Forward Compatibility. Backward Compatibility: newer code can read data written by older code. Forward Compatibility: older code can read data written by newer code (and must ignore unknown fields gracefully). Both are required in a live rolling upgrade scenario.

Schema Evolution

When a new field is added to a record schema, newer code writes it, but older code reading the record must keep the new field intact even if it cannot interpret it. The database appears to use a single unified schema even though underlying records may have been encoded with different historical schema versions.

Archival Storage

Database snapshots are typically encoded with the latest schema when taken, even if the source records were written at different schema versions. This ensures consistent analytical queries on the snapshot.

Data Flow Through Services

Processes communicate over a network. The server exposes an API; clients make requests to that API.

Client-Server Architecture▼

Servers expose APIs; clients connect and request data. Web browsers make GET requests for HTML/CSS/JS; POST for submitting data. The server's response to programmatic clients is typically JSON, not HTML.

💡 Insight
Because web standards (HTTP, URLs, SSL, HTML) are agreed upon universally, any browser can access any website — this is the power of standardization over proprietary interfaces.
Service Oriented Architecture (SOA)▼

Large applications are decomposed into smaller services by area of functionality. A server can itself be a client to another service. Key design goal: services are independently deployable and evolvable — one team can release without coordinating with others.

Implication: old and new versions of services run simultaneously during rolling deployments. Data encoding across versions must be compatible.

Remote Procedure Calls (RPC)▼

RPC makes a remote network request look like a local function call — location transparency. However this is fundamentally flawed: a network call has unpredictable latency, can fail, and may be retried — none of which apply to local function calls. Modern RPC frameworks (gRPC with Protocol Buffers, Thrift, Avro RPC) are more explicit about this distinction and add service discovery.

🎯 Exam Tip
The key critique of RPC: don't try to make remote calls look like local calls. Focus of modern RPC is intra-organization, intra-datacenter communication where custom binary encoding (Protocol Buffers) outperforms JSON over REST.

Asynchronous Message Passing

📚 Definition
Asynchronous message-passing systems sit between RPC (low latency) and databases (persistent storage). A message broker intermediary stores messages temporarily. Also called message queue or message-oriented middleware.

How It Works

  1. A producer process sends a message to a named queue or topic
  2. The broker stores it temporarily
  3. One or more consumers subscribe and receive the message

Advantages Over Direct RPC

AdvantageExplanation
BufferingActs as buffer if recipient is unavailable or overloaded → improves reliability
Auto-redeliverCan redeliver to crashed processes → prevents message loss
DecouplingSender doesn't need IP/port of recipient
Fan-outOne message can reach multiple consumers
Loose couplingSender publishes without caring who consumes
💡 Insight — Why Asynchronous?

The sender does not wait for the message to be delivered — it sends and forgets. This is fundamentally different from RPC where the caller blocks until a response is received. Message-passing is one-way; if a response is needed, it goes on a separate channel.

🔬 Research Addition — Confluent/Kafka, 2024

Apache Kafka has become the de facto standard for streaming data pipelines, capable of handling millions of events per second with sub-10ms latency. Kafka is used by over 80% of Fortune 100 companies. Its key design innovation is treating a message queue as a durable, replayable log — messages are not deleted after consumption but retained for a configurable period, enabling consumers to replay historical events. This turns Kafka from a simple queue into a system of record for event-driven architectures. (Confluent State of Data Streaming, 2024)

Modern Data Stack (MDS)

From legacy infrastructure to cloud-native, modular data platforms

📚 Definition
The Modern Data Stack (MDS) refers to integrated, cloud-based tools and technologies that enable the collection, ingestion, storage, cleaning, transformation, analysis, and governance of data. It is a cloud-native, modular solution designed to streamline automation, optimize costs, and accelerate insights — empowering self-service analytics and AI applications.

Problems with the Traditional Stack

Long Turnaround Time▼

On-premise companies bear all infrastructure costs. Deeply interconnected systems mean a minor change can break other parts. Finding logical coupling requires many work-hours before any improvement can be made.

Expensive Journey to Insights▼

Report generation is manual — manually generated, cleaned, and transferred to Excel. Leads to errors, time lost from critical tasks, inability to scale. Analysts cannot efficiently perform their roles due to complexity.

Slow Response to New Information▼

As the company grows, so does data and compute needs. Scaling on-premises is very costly. Data pipelines can take hours. A TDS requires slow ETL operations before new data can conform to the data model. A new data update can take weeks of refactoring before insights appear — by then, the opportunity has passed.

🎯 Exam Summary
TDS is summarized as: "Relies on rigid, on-premises infrastructure that can struggle with scalability, flexibility, and real-time processing." This is a direct quote from the lecture to memorize.

MDS Key Characteristics

✅ Enable Self-Service
  • Easily discover and analyze data
  • Understand column descriptions, history, lineage
  • Derive insights with minimal IT dependency
✅ Agile Data Management
  • Availability: Data lake/warehouse separates compute from storage — large data at low cost
  • Elasticity: Auto-scale on-demand (e.g., Friday afternoon peak → scale up, scale down later)
✅ Flexible & Pay-as-you-go
  • Easy to set up — no lengthy sales cycles, just login and go
  • Pay as you go — no million-dollar licensing fees
  • Plug and play — open standards & APIs, no vendor lock-in
🔬 FinOps Principle — Source: Reis & Housley, 2022
Cloud spending is inherently dynamic. FinOps — collaborative financial management between engineering, finance, and business — ensures data projects deliver maximum value. Monitor spending continuously; adopt hard limits with graceful failure modes. "Cost attacks" via excessive S3 downloads are a real threat.

Data Pipelines

What they are, where they came from, their purpose and challenges

📚 Definition (Lecture)
A data pipeline is a series of steps that allow data from one system to move and become useful in another system — particularly analytics, data science, or AI/ML systems. It works by pulling data from the source, applying transformation/processing rules, then pushing data to its destination.

Origin

Data pipelines emerged from the big data community. Data lakes created a "feeding frenzy" among developers, data scientists, and analysts who for the first time gained unfettered access to a nearly limitless supply of raw data from multiple internal and external systems. This created a wide diversity of workflows — or data pipelines — to process that data (clean, filter, aggregate, move, load) for a variety of purposes.

The Situation Problem

⚠️ The Current Painful Reality

The development of data and analytics pipelines is "still a handcrafted and largely non-repeatable process with minimal reuse, managed by individuals working in isolation with different tools and approaches." This results in:

  • A plodding development environment that can't keep pace with data-driven business demands
  • An error-prone operational environment that is slow to respond to change requests
  • Business and IT drawing swords and going to war instead of partnering ("Drawing Swords" metaphor from lecture)

Benefits of Well-Designed Data Pipelines

🔓 Self-Service Data

Created ad hoc by data scientists — breaks the IT bottleneck. People can test ideas, fail fast, and innovate faster.

⚡ Real-Time Analytics

Streaming data pipelines deliver continuous data to real-time applications and analytics — not just batch reports.

☁️ Cloud Acceleration

Enables cloud migration. Cloud computing offers processing speeds, cost-effectiveness, and bursting capacity impossible with on-premises — enabling NLP, sentiment analysis, image processing.

Challenges

⚒️ Always Under Construction▼

Building and debugging pipelines takes time. Schema alignment, setting sources/destinations, checking work, finding errors — all back and forth until going live, by which time business requirements may have changed again. This is why data engineers have such a massive backlog.

🔄 Out of Order / Data Drift▼

Even a small change to a row or table can mean hours of rework, updating each stage, debugging, and redeploying. Pipelines often go offline for updates. Unexpected, unplanned, unrelenting changes are called "data drift". Hidden breakages from unplanned changes can take months to uncover.

🏗️ Build It and They Will Come▼

Pipelines are built for specific frameworks, processors, and platforms. Changing any infrastructure technology to take advantage of cost savings or optimizations can mean weeks or months of rebuilding before deployment.

🔬 Research Addition — Gartner / Stitch Data, 2024

Industry research shows data engineers spend 40–60% of their time on pipeline maintenance rather than new development — the key driver behind the DataOps movement. The concept of "data drift" has been formalized: data drift occurs when the statistical properties or schema of data change over time, degrading pipeline reliability and ML model accuracy. Tools like Great Expectations and Monte Carlo Data provide automated drift detection as part of data observability platforms. (Gartner Data & Analytics Summit, 2024)

Batch vs. Stream Processing

Batch Processing Pipeline

Loads "batches" of data into a repository during set time intervals — typically scheduled during off-peak business hours.

💡 Insight

Batch processing was a critical step in building reliable, scalable data infrastructures. Batch jobs form a workflow of sequenced commands where the output of one command becomes the input of the next. Optimal when there's no immediate need for real-time analysis (e.g., monthly accounting, end-of-day reports).

Stream Processing Pipeline

Also known as event-driven architectures — continuously processes events generated by IoT devices, sensors, or user interactions in an application. Events are processed, analyzed, then stored in databases or sent downstream.

⚠️ Trade-off

Streaming systems have lower latency than batch systems, but are not as reliable — messages can be unintentionally dropped or spend a long time in the queue. When data absolutely cannot be lost (financial transactions), use exactly-once semantics (Kafka's transactional API) or implement idempotent consumers.

🎯 Exam Scenario

The lecture CDC scenario: an e-commerce inventory system must replicate to a backup database with no service interruption and minimum data loss. The answer: Transaction Log-based CDC (binlog reading via Debezium → Kafka). The RDBMS generates event logs which can be used for replication. Cloud-based NoSQL systems send events (trigger CDC). Dataflow Model & Unified Batch + Streaming

One central problem: managing batch and stream processing requires unifying multiple code paths.

ArchitectureDescriptionIssue
Lambda ArchitectureSeparate batch layer + speed (stream) layer + serving layerTwo code paths to maintain (batch logic ≠ stream logic)
Kappa ArchitectureUnified streaming layer only — no separate batch layerStill needs different tools for real-time stats vs batch aggregation
Dataflow ModelGoogle's model: view all data as events; aggregation over windowsElegant solution — batch is just bounded streaming
💡 Key Insight — "Batch as a Special Case of Streaming"

Google's Dataflow model (implemented as Apache Beam) views ALL data as events:

  • Real-time streams = unbounded event streams
  • Data batches = bounded event streams (the batch boundary provides a natural time window)
  • Real-time and batch processing happen in the same system using nearly identical code
  • Windows can be tumbling (non-overlapping) or sliding (overlapping)

Apache Flink and Apache Spark have adopted this approach. This philosophy is now pervasive in modern streaming systems.

🔬 Research Addition — Google / Apache Beam, 2024

The Dataflow programming model introduced by Google (Akidau et al., 2015) is now the foundation of Apache Beam, which provides a unified API that runs on multiple execution engines (Flink, Spark, Dataflow, Samza). In 2024, Apache Flink 2.0's "unified batch and streaming" has essentially made the Lambda Architecture obsolete for new greenfield projects. Flink processes billions of events per day at companies like Alibaba and Netflix. (Apache Beam blog, 2024)

Micro-Pipelines

📚 Definition
To manage complexity, individual stages are divided into micro-pipelines — ensuring changes can be made to each step without disrupting the entire flow. Each micro-pipeline is a small, independently deployable unit of data processing logic.

Steps in a Micro-Pipeline

1. Requirement▼

Agile methods (Scrum, Kanban) elicit requirements iteratively from stakeholders. A requirement could be integrating a new data source in an ETL process or adding a new field to a database table.

2. Development▼

In data and analytics, development doesn't necessarily mean programming — it can comprise tasks like changing a data model or reconfiguring a dashboard. Requires a high level of automation (e.g., auto-generating DB queries when a model changes).

3. Testing / Tuning▼

Validate the pipeline step against real data samples. Test for schema compliance, null values, referential integrity, and performance benchmarks. Tools: Great Expectations (unit testing for data), dbt tests, Apache Deequ.

4. Production / Deployment▼

Deploy the micro-pipeline to production. Monitoring and alerting are configured. The deployed pipeline feeds into the execution layer. Rollback capability is essential — if a deployed step breaks downstream systems, you must revert quickly.

🔬 Research Addition — DataOps Manifesto / Gartner, 2024

The DataOps Manifesto (datakitchen.io) defines 18 principles for industrializing data pipeline development. Core principle: "Data pipelines are software — treat them with software engineering discipline." This means version control (Git for dbt models), CI/CD (automated testing before deployment), and observability (monitoring pipeline SLAs). Gartner predicts that by 2026, 75% of large enterprises will adopt a DataOps automation platform. (Gartner, 2024)

Data Pipeline Tools & Platforms

Choosing the right tool for the right job

Key Considerations When Choosing Tools

🎯 Decision Framework (Lecture)
  1. Are we grabbing data from one place to put it somewhere else? Or do we need to transform it to fit downstream analytics requirements?
  2. Is our data environment stable and fully under control? Or dynamic, pulling from systems outside our control?
  3. Will the pipeline move data once (short-term project)? Or must it be operationalized to handle data flows over time?

Three Tool Categories

Data Ingestion & Loading Tools

These tools make pipelines easy to set up, build, and deploy — solving the "under construction" problem. But only if data scientists can do the prep work later.

⚠️ Limitations
  • Only support the most basic transformations — work best for simply copying data
  • Pipelines are rigid and embedded with the specifics of data structure and semantics
  • When data drift happens, they must be rebuilt and redeployed from scratch
