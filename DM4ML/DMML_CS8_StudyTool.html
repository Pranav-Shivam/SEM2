Data Quality Dimensions
Core measurement attributes of data — assess, interpret, improve individually
📘 Definition

Data quality dimensions are measurement attributes of data that can be individually assessed, interpreted, and improved. Their aggregated scores represent overall data quality in a specific context and indicate fitness of data for use.

💡 Healthcare Domain Examples
  • Accuracy: Wrong blood group recorded (A+ instead of O−) → risk of wrong transfusion
  • Completeness: Allergy info missing → risk of adverse drug reaction
  • Consistency: Patient weight 68 kg in EMR but 75 kg in lab → incorrect dosage
  • Timeliness: ICU vitals updated 2 hours late → delayed intervention
  • Validity: Heart rate 400 bpm or invalid ICD code → violates business rules
  • Uniqueness: Duplicate profiles ("Rajesh Kumar" vs "R. Kumar") → fragmented history
  • Integrity: Prescription references nonexistent doctor ID → compliance issue
  • Traceability: Medication changed with no audit trail → unclear accountability
  • Reliability: Multiple systems show different SpO₂ readings → untrustworthy data
  • Accessibility: MRI scans not retrievable during ER emergency → delayed diagnosis

The 10 Core Dimensions

1. Accuracy

Measures how well data represents real-world entities and whether it can be validated against trusted sources. Critical for predictive analytics and customer segmentation. Inaccuracies cause flawed forecasts, misaligned marketing, compliance risks.

⚡ Exam Tip

Accuracy ≠ Validity. Accuracy = matches real world. Validity = conforms to format rules. A date "2025-02-30" is valid format but inaccurate (Feb has no 30th).

2. Completeness

Whether all required data values are present. Missing data = unreliable analytics. Signals weaknesses in data collection or system integration. Fix: automated alerts for missing fields, third-party data enrichment, governance policies.

3. Consistency

Data must be consistent across systems and sources. Example: customer phone number differs between CRM and order management. Inconsistency causes audit failures and misinterpretation of financial results. Fix: centralized governance frameworks and data integration tools.

4. Timeliness

Data available when needed, reflecting current state. Critical in finance, healthcare, e-commerce. Example: real-time stock prices for trading. Fix: regular data refreshes, real-time feeds, event-driven architectures, streaming analytics, SLAs for data delivery.

5. Validity

Whether data conforms to predefined rules, formats, and standards. Beyond format compliance: birth date should not be future, product codes should match catalog. Enforce via rules during data entry, automated anomaly detection, industry-aligned standards.

6. Uniqueness

Each record must be distinct and non-duplicated. Duplicates cause inflated metrics, distorted analytics, wasted resources. Often arises from migrations, manual errors, or lack of integration. Fix: data matching algorithms, strict identity policies, quality dashboards.

7. Integrity

Data being accurate, consistent, and reliable over time. Integrity can degrade as data is added, modified, deleted. Requires ongoing validation rules + referential integrity checks (changes in one place reflected in all other places).

⚡ Exam Tip

Integrity encompasses accuracy + consistency + reliability over time. Think of it as "long-term data health."

8. Lineage / Traceability

Lineage = where data came from and how it evolved. Traceability = ability to verify accuracy through audit trail. Regulations require clear documentation (e.g., BCBS 239 for financial services — risk data aggregation and reporting practices).

9. Reliability

Accurately reflects reality AND ensures continuous availability/uptime = business confidence. Example: sales performance reporting needs accurate metrics delivered on time.

⚠️ Common Mistake

Assuming reliability is purely a technical problem. Non-technical factors — organizational processes, governance, culture — play a major part.

10. Accessibility

Data must be easy for authorized users to find, retrieve, understand, and use. Involves: clear organization (catalogs), consistent access methods (APIs), affordability, and often multi-language support.

Measuring Quality — Ratio + Threshold Pattern

🔬 Source: learning.bansal-ai.in (2026)

Most dimensions reduce to a ratio against a threshold. This turns "data quality" into concrete, automatable tests.

💡 Worked Example — Completeness Check

An email column has 9,200 non-null values out of 10,000 rows.

Completeness = 9,200 / 10,000 = 92%

Threshold set at ≥ 98% → FAIL → Pipeline alerts/blocks. Bad data stopped before training.

Completenessnon-null / total≥ 98%

⚡ Exam Tip

Quality as "ratio + threshold" is the key formula pattern. Any dimension can be automated this way — completeness, validity, uniqueness each become a measurable percentage compared to an acceptance threshold.

Profiling vs Validation

🔬 Source: learning.bansal-ai.in (2026)

Profiling = explores data to understand it (distributions, nulls, ranges, distinct values) — learns the baseline.
Validation = checks data against explicit expectations (rules, schema, thresholds) and flags/blocks violations.
Principle: Profile to set expectations; validate continuously to enforce them. "Examine, then enforce."

🔬 Research Addition — Industry Impact (Gartner/IBM, 2024)

Organizations lose $12.9 million/year on average due to poor data quality (Gartner). IBM estimated the US economy loses $3.1 trillion/year to bad data. Modern tools: Monte Carlo (data observability), Atlan (active metadata platform), Great Expectations (validation framework).

Data Quality Dimensions — ML Extension
Additional dimensions critical for machine learning data readiness
📘 Definition

Beyond the 10 core dimensions, ML workloads require data that is informative, has good coverage, reflects real inputs, is unbiased, avoids feedback loops, has consistent labels, and is big enough to generalize.

Informative → maps to Completeness/Usage

Contains enough information for modeling. Example: predicting if customer buys product — need both product properties AND customer purchase history. With only product properties + location/name, predictions are same for all users from same location.

⚠️ Warning

Model may derive gender/ethnicity from name and make biased predictions — technically "working" but ethically problematic.

Coverage → maps to Completeness/Coverage

Data must cover what you want the model to do. Example: classifying 1000 topics — need sufficient examples per topic. If only 1-2 docs per topic, algorithm may use unique IDs as differentiators → won't generalize.

⚡ Exam Tip

Insufficient coverage per class is a common reason models fail in production but appear to work in testing.

Real Inputs → maps to Accuracy

Training data must reflect production inputs. Example: car recognition trained only on daytime photos — fails on night pictures in production. Distribution of inputs at training time must match serving time.

Unbiased → maps to Integrity

Bias can exist in both training data and production data. UI can be a source of bias. Example: predicting news popularity using click rate — top-of-page placement inflates clicks regardless of content quality (position bias).

Not a Feedback Loop → maps to Integrity

Don't use model's own predictions as training labels. Examples:

  • Can't train gender-prediction model, then use its predictions to label new training data
  • Email importance model highlights messages → users click highlighted ones → using those clicks as "important" signal is circular
⚠️ Common Mistake

Feedback loops are subtle and self-reinforcing. Model amplifies its own biases over time.

Consistent Labels → maps to Consistency

Inconsistent labeling sources: different criteria per labeler, class definitions evolving over time, misinterpreting user motives (user ignored recommendation because they already knew the story, not because they're uninterested).

Big Enough → maps to Completeness

Must allow generalization. Sometimes data has low predictive power — no amount helps. But often, going from thousands to millions/hundreds of millions dramatically improves accuracy. Can't know how much is needed before starting.

🔬 Research — Andrew Ng's Data-Centric AI (2021–present)

Improving 1,000 labels often outperforms adding 10,000 noisy examples. Quality over quantity for small/medium datasets.

Cost

Getting unlabeled data can be expensive, but labeling is the most expensive — especially manual labeling for specific problems.

💡 Clever Solution

Google's reCAPTCHA: outsources labeling to random users for free. Solves two problems: reduces spam + provides cheap labeled data to Google.

Noise

Random process corrupting examples independently: blurry images, lost text formatting, background audio noise, missing poll attributes.

  • Missing attributes → data imputation techniques
  • Blurred images → image deblurring algorithms
  • Audio noise → algorithmic suppression

Small datasets (thousands): Noise causes overfitting — algorithm models the noise itself.

Big data: Noise typically "averaged out" over multiple examples.

Low Predictive Power

Example: Predicting if listener likes a song. Data: artist name, title, lyrics, playlist status. Problem: musical preferences depend on arrangement, instruments, sound effects, tone — none extractable from text data.

⚡ Exam Tip

Low predictive power is often not considered until fruitless energy is spent training. Hard to distinguish: is model not expressive enough, or does data lack information?

Outliers

Examples dissimilar to majority, measured by distance metrics (e.g., Euclidean). Shallow algorithms especially sensitive: linear/logistic regression, AdaBoost.

⚠️ Debate

Deleting outliers is not scientifically sound for small datasets. In big data, outliers typically don't influence model significantly. Exclusion may be justified if holdout performance improves.

Bias & Fairness
12 types of bias and identifying them in ML data
📘 Definition

Bias in data is an inconsistency with the phenomenon that data represents. Can occur for multiple non-mutually-exclusive reasons.

🔬 Source: bitsnotes.com — The Scale Analogy

Bias is systematic, not random. Think of a weighing scale that always reads 2 kg heavy. Every measurement is wrong in the same direction — that's bias. Random noise would scatter readings above and below; bias shifts them consistently. In ML: biased data doesn't produce random errors, it produces consistently wrong predictions for specific groups or scenarios.

12 Types of Bias

1. Omitted Variable Bias

Featurized data lacks a necessary feature for accurate prediction. Key feature wasn't available, so important information was missing.

2. Sponsorship / Funding Bias

Data from sponsored agency is biased. Sponsored news agencies suppress bad news about sponsor, exaggerate achievements → suboptimal model.

3. Prejudice / Stereotype Bias

Often in historical data. Photo archives show men at work/outdoors, women at home. Word2vec: king − man + woman ≈ queen, but programmer − man + woman ≈ homemaker.

🔬 Research — Amazon Hiring Tool (Reuters, 2018)

Amazon's recruiting tool trained on 10 years of resumes (predominantly male). Penalized "women's" in text, downgraded all-women's colleges. Amazon scrapped the tool.

4. Systematic Value Distortion

Constant errors from measurement devices. Training on low-quality camera with yellowish white → fails with high-quality camera that sees true white.

5. Experimenter Bias

Tendency to search/interpret/favor info that affirms prior beliefs. In ML: each dataset example from survey answers by a particular person.

6. Labeling Bias

Labels assigned by biased process/person. Some labelers read fully vs. scanning keyphrases. Brain pays attention to specific domains → biased labels. Some skip uninteresting documents → underrepresented classes.

7. Reporting Bias

Documented frequency doesn't match real-world frequency. People record unusual/memorable events, assume ordinary needs no recording. Book reviews skew extreme — people with mild opinions don't submit.

8. Automation Bias

Favoring automated results over non-automated, irrespective of error rates. Example: "groundbreaking" defect model had 15% lower precision and recall than human inspectors.

9. Selection Bias (3 subtypes)

Coverage bias: Non-representative selection. Model trained on buyers only, ignoring competing-product buyers.

Non-response / Participation bias: Competing product buyers 80% more likely to refuse survey → underrepresented.

Sampling bias: First 200 email respondents chosen instead of random → more enthusiastic than average.

10. Group Attribution Bias

In-group bias: Preference for your own group. Engineers favor applicants from their own academy.

Out-group homogeneity: Stereotyping non-group members. "All non-CS-academy applicants lack expertise."

11. Implicit Bias

Assumptions from personal experience that don't generalize. Head shake = "no" in some cultures but "yes" in others.

12. Confirmation Bias

Unconsciously processing data to affirm pre-existing beliefs. ML practitioner who feared toy poodles discarded evidence of docility in small dogs.

🔬 Research — COMPAS Recidivism (ProPublica, 2016)

COMPAS tool used in US courts: false positive rate for Black defendants nearly 2× that for white defendants (45% vs 24%). Illustrates the impossibility theorem — cannot simultaneously satisfy all fairness metrics.

Entered into force Aug 2024. Risk tiers: unacceptable (banned), high-risk (hiring, credit, justice — requires bias audits), limited risk (disclosure), minimal risk. Penalties: up to €35M or 7% global turnover.

⚠️ Common Misconception

"Removing protected attributes (race, gender) eliminates bias." WRONG. Proxy variables (zip code, name patterns) encode protected attributes. This is "fairness through unawareness" — widely considered insufficient.

Data Leakage
Target leakage — contamination affecting multiple ML lifecycle stages
📘 Definition

Data leakage (target leakage) = unintentional introduction of information about the target that should not be available. Training on contaminated data → overly optimistic performance expectations.

🔬 Source: bitsnotes.com — The Food Analogies for Leakage

Idli-Chutney Analogy: Imagine tasting idli with chutney during training, then being asked to identify the taste of plain idli alone in production. The chutney (leaked info) made it taste great — but it was never the idli alone. Your model learned the "chutney" (leaked feature), not the actual signal.

Aloo Paratha Analogy: If someone always serves aloo paratha with pickle, and you learn to predict "aloo paratha" by tasting pickle — you'll fail when the pickle isn't there. The pickle is the feature from the future or the target-derived feature that won't exist at inference time.

3 Causes of Leakage

Target = f(feature)
→
Feature hides target
→
Feature from future

1. Target Being a Function of a Feature

Example 1: Predicting GDP. Dataset has Population and GDP per capita columns. Population × GDP per capita = GDP. Model perfectly predicts by multiplying two columns.

Example 2: Predicting yearly salary when monthly salary is a feature. Monthly salary alone predicts perfectly. In production, monthly salary won't be available.

2. Feature Hiding the Target

Predicting customer gender — "Group" column contains demographic value encoding gender+age info. The target is "hidden" inside a feature.

💡 Exception

If Group values are predictions from another less-accurate model, using them to build a stronger model = legitimate model stacking.

3. Feature Coming from the Future

Predicting loan repayment — using "Late Payment Reminders" as feature. In production, this value is always 0 (model is used BEFORE credit is given). Model learned to predict "No" when reminders ≥ 1 → useless in production.

⚡ Exam Tip

Feature from future is HARDEST to catch. Requires clear understanding of business context and when the model will be used.

🔬 Research — Kaggle Leaky Competitions

KDD Cup 2014: Target leakage via an inadvertent feature. Competitors who exploited the leak dominated. COVID-19 X-ray models (2020-21): Models learned hospital-specific markers (text annotations, positioning artifacts) instead of pathology — a textbook confounding metadata leakage (Nature Machine Intelligence, 2021).

Prevention

  • Strict temporal splits (no random splitting for time-series)
  • Feature computation audits — no future information used
  • Adversarial validation: train model to distinguish train from test. If AUC >> 0.5, leakage likely
⚠️ Misconception

"Cross-validation prevents leakage." CV only prevents train-test contamination. Does NOT catch feature leakage where feature itself encodes target. Preprocessing (scaling, imputation) before CV split also leaks info.

Data Skew & Drift
Why models decay in production — schema skew, distribution skew, concept drift
📘 Definition

Model decay/drift = ML models in production experience reduced performance over time, not only from bad data but from constantly evolving datasets and profiles. Speed varies: years in some models, days in others.

Data Drift — Schema Skew

Training and serving data don't conform to same schema. Caused by changes in upstream process.

  • Inconsistent features: new feature added to serving data
  • Inconsistent feature types: integer in training → real number in serving
  • Inconsistent feature domains: categorical value disappears or numerical range changes
Data Drift — Distribution Skew

Distribution of feature values significantly different between training and serving. Can result from wrong training dataset choice or natural trends (real estate prices change, fashion popularity shifts).

Concept Drift

Interpretation of data has changed. Mapping of input features to labels no longer valid, or novel class/range has appeared. Result of change in the process being modeled or evolution of understanding.

💡 Example

User content preferences change over aging or new discoveries → historical training examples start HURTING performance instead of helping.

Training-Serving Skew

Discrepancy between model's feature engineering code during training vs deployment. Causes: different data handling in training/serving pipelines, data changes between phases, feedback loops, bugs, computational resource mismatches.

Fix: Reuse same feature engineering code for training and serving. Use feature stores.

🔬 Research — Feature Stores (2024-2025)

Feast (open-source, v0.38+): registry + offline/online stores. Tecton (commercial, by Feast creators): real-time feature computation, used by Plaid, Coinbase. Uber's Michelangelo feature store reduced model debugging time ~50%. Databricks Feature Store: integrated with MLflow.

⚠️ Misconception

"Good offline evaluation = good production performance." Training-serving skew means offline metrics can be misleadingly optimistic. Even small differences in feature computation (timezone handling, null imputation) cause meaningful drops.

Data Drift Detection Metrics
PSI, KL Divergence — measuring and detecting drift
📘 Definition

Measuring data drift for large datasets is hard. System must log all incoming features, predictions, and ground truth. Three approaches: model-based, statistical, algorithm-based.

Detection Approaches

Supervised: Statistical Process Control, Sequential Analysis, Adaptive Windowing

Unsupervised: Clustering/novelty detection, Feature distribution monitoring

Model-dependent: MD3 (Margin Density Drift Detection)

  • PSI (Population Stability Index)
  • KL Divergence (Kullback-Leibler)

Creating dedicated models specifically to detect drift in data distributions.

PSI (Population Stability Index)

📘 Formula
PSI = Σ (Actual% − Expected%) × ln(Actual% / Expected%)

Measures how much distribution changes between training dataset and new dataset.

PSI ValueInterpretationAction
< 0.1No significant changeNo action needed
0.1 – 0.25Moderate shiftInvestigate
≥ 0.25Significant shiftAction required

Applications

  • Credit Scoring: Ensure model still works after economic changes
  • Risk Assessment: Check if borrower income shifts affect lending models
  • Customer Segmentation: Verify segments still relevant over time
  • Fraud Detection: Spot changes in fraudulent behavior patterns

⚠️ PSI Limitation

PSI can miss multivariate drift — features individually stable but joint distribution shifts. Also sensitive to binning strategy for continuous variables.

KL Divergence (Kullback-Leibler)

📘 Definition

Relative entropy — quantifies difference between one probability distribution and a reference distribution. Applied to each feature independently (not a covariant measure).

Applications

  • Model evaluation: measuring prediction-to-label distance
  • Variational Inference: minimizing D_KL in VAEs
  • Reinforcement Learning: TRPO uses KL constraints
  • Language Modeling: comparing token predictions to ground truth

KL Divergence Worked Example

🔬 Source: bitsnotes.com

Consider a categorical feature "payment_method" with 3 values:

Payment MethodTraining P(x)Current Q(x)P(x) × ln(P/Q)
Cash0.500.300.50 × ln(0.50/0.30) = 0.2554
Card0.300.400.30 × ln(0.30/0.40) = −0.0863
UPI0.200.300.20 × ln(0.20/0.30) = −0.0811
KL(P||Q)0.088 nats
⚡ Exam Tip

KL = 0 means identical distributions. Higher = more divergence. Note: if we computed KL(Q||P) instead, we'd get a DIFFERENT value — that's the asymmetry problem. For this reason, Jensen-Shannon Divergence (average of both directions) is often preferred.

⚠️ KL Limitation

KL divergence is undefined when Q(x) = 0 but P(x) > 0 (requires smoothing). It is asymmetric: KL(P||Q) ≠ KL(Q||P). Jensen-Shannon Divergence (symmetric) often preferred in practice.

ML Data Readiness 1 — Data Profiling
Analyzing datasets systematically before ML projects begin
📘 Definition

Data profiling = systematically analyzing datasets to evaluate quality, structure, and content. Identifies anomalies, missing values, patterns. Supports data governance, standardization, and compliance.

⚡ When to Profile

Right at the BEGINNING of a data project. Evaluate viability early. Catching problems early saves significant time. Spot data transformations, identify unexpected behavior, predict ETL issues.

Techniques

Summary Statistics

Count, max, min, mean values.

Data Types

Categorical vs continuous, patterns. Simple types: strings, numbers, timestamps. Complex: XML, JSON.

Data Values

Identify characteristics/patterns: addresses, cities, ID strings. Assess against business rules (e.g., only South American students eligible for specific funding).

Distributions

Categorical: counts per category. Numerical: histograms, skewness, modes, outliers.

Inclusion Dependencies, Keys, and Functional Dependencies

Find if values in one field are subset of another → adds dimensions to dataset.

Visualize Numerical Relationships

Pair plots, cross-correlation heat maps, correlation tables. Quick overview of relationships between data assets.

ML Data Readiness 2 — Filter, Sampling & Validation
Getting data ready before training begins

Filter / Selection / Sampling

Not all created data is ingested — filtering/selection step is necessary. Sampling reduces cost of intermediate processing and training.

⚠️ Trade-off

ML systems generally perform better with more data. Any reduction may impact quality. Must measure quality cost of sampling vs savings. Some detail loss is unavoidable.

Data Validation

📘 Definition

Checking accuracy and quality of source data before training a new model version. Ensures anomalies in incremental data aren't silently ignored. Checks statistics match expectations (feature distribution, number of categories, etc.).

Key Validation Questions

  • Are there anomalies/errors in incremental data?
  • Are training-time assumptions violated during serving?
  • Significant differences between training and serving data?
  • Differences between successive training data additions?
⚡ Exam Tip

Validation output must be informative enough for data engineers to take action and have high precision (too many false alarms = lost credibility).

Challenges

Few columns = simple. Many columns = humongous task. Tracking/comparing metrics across historical datasets for each column needs recurring data scientist time. Modern applications should auto-refresh validation rules.

Great Expectations

Core idea: "We should test data, not just code." Expectations prescribe expected state of data, checked in-place (no data movement).

  • Exhaustive list of Expectations (pre-defined + templated)
  • Supports data contracts with automated checks and human-readable summaries
  • Connects to databases, data warehouses, catalogs (Airflow, Meltano, Dagster)
  • Flexible storage: S3, Azure Blob, GCS, PostgreSQL, filesystem

Soda Core

Open-source Python library with CLI. Uses SodaCL (YAML-compatible DSL). Connects to data sources and workflows.

  • Find insufficient data by running checks inside data sources
  • Programmatic or orchestrated scans (Airflow, Dagster, dbt Core)
  • Pass YAML config + Soda checks → run validation

Deequ

AWS Labs open-source, built on Spark. Good for testing massive data. Anything in a Spark DataFrame can be tested.

Automated Data Quality Testing in Pipelines
Integrating validation across the Medallion architecture

Data sources produce incorrect data. Transformation rules need updates over time. Quality validation starts during initial development — check correct patterns in each layer (Medallion architecture), correct data types, proper documentation.

The Quality Gate Pattern

🔬 Source: learning.bansal-ai.in

Validation runs at ingestion and between stages. Define expectations → run on every batch → on failure, stop or quarantine data rather than let it reach the model. The gate stands between bad data and the model.

Pre-Ingestion vs Pre-Training Validation
Strategic placement of validation in ML pipelines
📘 Key Principle

Data quality issues can be introduced anywhere. Identify hotspots and deploy validation strategically. Following shift-left principle: first check ingestion, then each transformation step before training.

Pre-Ingestion Validation

Example: Card transaction validated via API to bank. Card info validated AND verified. Result sent back immediately (valid/invalid). If invalid → client alerted before ingestion, failed payment logged.

🔬 Source: bitsnotes.com

After data is ingested, additional validation runs:

Header File & Audit Table Patterns

💡 Validation Design Patterns (bitsnotes.com)

Header File Pattern: Each data batch arrives with a header file containing expected record count, checksum, and schema version. Ingestion compares actual data against header before accepting.

Audit Table Pattern: Every validation result (pass/fail, metric values, timestamp) is logged to an audit table. Enables historical trend analysis and compliance reporting.

Data Governance Context

🔬 Source: bitsnotes.com — Governance Frameworks

Pre-ingestion validation must align with governance frameworks: GDPR (EU data privacy), HIPAA (US healthcare), PII handling rules (personally identifiable information), PCI DSS (payment card industry security). Data contracts serve as "handshake agreements" between data producers and consumers — formalizing schema, quality expectations, and SLAs before data exchange.

⚡ Exam Tip
