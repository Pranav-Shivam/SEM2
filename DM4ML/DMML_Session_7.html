Session 6–7 · Core Framework

CRISP-DM Lifecycle

The Cross Industry Standard Process for Data Mining — the de facto framework for every ML project since 1999.

📘 Definition

CRISP-DM is a six-stage iterative methodology for data mining and ML projects. Published in 1999, it remains the most widely adopted framework. Every stage can loop back — it is a cycle, not a waterfall. IBM SPSS, TFX, SageMaker, and MLflow all map to its stages.

⟲ iterative
1. Business Understanding — "Deciding what kind of house you need"▼

Set objectives, assess feasibility, identify risks, produce a project plan. The most skipped yet most critical stage. Key question: What decision will this model inform?

Deliverables: Project objectives, success criteria, resource inventory, risk assessment, project plan.

⚡ Exam Tip

Professor warned: "Engineers love building models. But a model that predicts the wrong thing is worthless. Always start with: what decision will this model inform?"

2. Data Understanding — "Surveying the land"▼

Collect initial data, describe shape (mean, median, range, types), explore patterns, verify quality. Produces four reports: data collection, data description, data exploration, data quality.

Key rule: Understanding comes BEFORE preparation. You cannot clean data you don't understand.

3. Data Preparation — "Laying the foundation"▼

Select, clean, construct derived attributes (e.g., "car age" from manufacturing date), integrate multiple sources, reformat. The most time-consuming stage — typically 60–80% of project time.

⚠️ Warning

"If the foundation is crooked, the walls will crack — no model can fix bad data." Poor preparation is the #1 cause of ML project failure.

4. Modeling — "Building the walls"▼

Select technique, document assumptions, generate test design, build model with chosen parameters (batch size, learning rate), assess, revise. Often multiple techniques are tried.

5. Evaluation — "Inspection against business criteria"▼

Check results against business success criteria (not just technical metrics). Three tasks: Evaluate Results → Review Process → Determine Next Steps.

⚡ Key Distinction

Modeling assessment = technical fit. Evaluation = business value. A technically perfect model solving the wrong problem fails Evaluation.

6. Deployment — "Handing over the keys"▼

Plan deployment, monitoring, and maintenance. Produce final report. Post-mortem review. Unlike a house, an ML project is never "done" — models degrade, data drifts, you cycle back.

Industry Scenario: Churn Prediction

🏭 Full CRISP-DM Walkthrough — Telecom Churn

Business: Predict customers leaving in 30 days. Recall ≥ 85%. FN cost=$500, FP cost=$20.

Data: 12 months of records, 15% missing tenure, 7 inconsistent payment categories.

Prep: Impute tenure with median, standardise payment to 4 categories, create avg_charge_to_tenure_ratio.

Model: XGBoost after grid search. 88% recall. Projected ROI: $2.3M/year.

Deploy: REST API, daily batch scoring, dashboard tracking recall, precision, drift.

CRISP-DM vs Modern Variants

Original 1999 standard. Six iterative stages. Tool-agnostic and industry-agnostic. Still most-cited framework in academic ML literature. IBM SPSS Modeler officially supports it.

CRISP-ML(Q) — 2021 quality-focused update. Adds a Quality Assurance layer to every stage. Requires measurable KPIs ("time savings per user per session"), feasibility confirmation, and formalised risk registers.

🔬 Research — ml-ops.org (2021)

Helpful addition: define a non-ML heuristic benchmark at Business Understanding. "Will this ML approach clearly outperform a simple rule-based system?" If not, skip ML.

1. Treating CRISP-DM as one-pass. It is iterative — plan for loops back.

3. The professor's principle: "Stay tool-agnostic and model-agnostic — the same data can be viewed differently."

Session 6–7 · Business Layer

Problem Framing & Success Criteria

Before a single line of code: quantify goals, costs, and metrics.

📘 The Four Framing Questions

1. What is the goal? (Classify / Predict / Cluster / Rank)
2. What does the data cost? (acquisition, storage, labeling, compute)
3. What does a wrong prediction cost? (Type I vs II asymmetry)
4. What quantitative metric proves success?

Qualitative vs Quantitative Metrics

TypeDefinitionML ExampleVerdict
QualitativeDescriptive, subjective"The model seems accurate"❌ Not actionable
QuantitativeMeasurable, verifiable"F1 ≥ 0.85 on holdout test set"✅ Required

Cost Asymmetry: Type I vs Type II

⚠️ Type I Error — False Positive

Model says YES, reality says NO. Fraud detection: block a legitimate transaction. Cost = customer trust. Precision measures this error's frequency.

⚠️ Type II Error — False Negative

Model says NO, reality says YES. Fraud detection: approve a fraudulent transaction. Cost = direct financial loss. Recall measures this error's frequency.

Full Example: Gold Price Prediction

🏭 Framing from Scratch

Goal: Predict tomorrow's gold price within ±2% of actual closing price.

Data cost: Historical API $200/mo + news sentiment $500/mo = $8,400/year.

Error cost: 5% miss on $1M position = $50,000 loss per incident.

Success metric: RMSE ≤ $35/oz on 90-day rolling test window. Auto-retrain if threshold breached.

Metric Selection Guide

When to use Accuracy vs F1 vs AUC-ROC vs RMSE▼
MetricBest ForAvoid When
AccuracyBalanced classes, equal error costsImbalanced data — misleading
PrecisionFP cost high (spam, legal)FN cost high (cancer screening)
RecallFN cost high (cancer, fraud)FP cost high (spam blocking)
F1Imbalanced, balanced FP/FN costVery asymmetric costs
AUC-ROCRanking quality, threshold-independentHeavily imbalanced multi-class
RMSE / MAERegression (continuous output)Classification tasks
⚡ Exam Trap

A model achieving 99% accuracy on data where 99% of samples are Class A is worthless — it just always predicts Class A. Always check class balance before choosing accuracy. Align metric choice with stakeholders before building.

🔬 Research — CRISP-ML(Q), ml-ops.org (2021)

Formalises this further: define measurable economic success criteria alongside ML and business criteria. KPIs must be verifiable. Establish a non-ML heuristic baseline — "can a rule-based system reach 70%? If so, ML must clearly beat it to justify the cost."

Confusion Matrix & Error Types

The 2×2 table that reveals what accuracy hides — asymmetric error costs.

📘 Definition

A confusion matrix compares actual class labels against predicted labels, decomposing errors into TP, FP, FN, TN. It exposes the cost asymmetry that accuracy alone conceals.

Error Type Definitions

⚠️ Type I Error — False Positive

Predicts positive, reality is negative. The false alarm. Mnemonic: Type I = "I falsely accuse." Example: airport scanner beeps for a metalless person.

⚠️ Type II Error — False Negative

Predicts negative, reality is positive. The missed detection. Mnemonic: Type II = "II missed it." Example: scanner stays silent for someone with metal.

Derived Metrics

Disease Screening Case Study (10,000 patients)

🏭 Why Accuracy Lies on Imbalanced Data

Setup: 100 sick (1%), 9,900 healthy. Model: 95% sensitivity, 90% specificity.
TP=95, FN=5, FP=990, TN=8,910

Accuracy = (95+8910)/10000 = 90.05% — looks great!

Precision = 95/(95+990) = 8.8% — terrible! 91.2% of positive predictions are wrong.

Recall = 95/(95+5) = 95% — good for catching disease.

Of 1,085 positive predictions, only 95 were truly sick. 990 healthy people get unnecessary follow-ups.

⚡ Exam Tips

1. Professor warned: accuracy is deceptive for imbalanced data. Always compute P, R, F1.
2. Type I vs II confusion: memorise the mnemonics.
3. In regulated industries, confusion matrix breakdowns prove the model doesn't systematically harm protected groups.

Session 6–7 · Architecture

The ML Data Pipeline

Five stages, one feedback loop — the backbone of every MLOps system.

📘 Definition

An ML data pipeline is a five-stage automated workflow: raw data → feature store → training → deployment → monitoring, with a feedback loop from Stage 5 back to Stage 1. It is a cycle, not a waterfall.

⟲ to Stage 1
Stage 1: Collect / Pre-process / Engineer▼

Raw data arrives from databases, logs, APIs, files. Clean, impute, transform, engineer features. After engineering, check feature correlations — pairs at ≥0.95 correlation add noise without signal, drop one.

Stage 2: Feature Store — Online & Offline Stores▼

Centralised catalog preventing duplicate feature engineering. Two layers:

StoreLatencyUse CaseTechnology
Online<5msReal-time inference servingRedis, DynamoDB, Cassandra
OfflineMinutes+Batch training, backfillS3/Parquet, BigQuery, Hive
🔬 Research — Introl Blog, Databricks (2025)

Uber Michelangelo: 10,000 features across 1,000 models. Spotify: 100 billion events/day into features. Netflix: petabytes across tiered stores. Without a feature store → training-serving skew.

Stage 3: Train / Tune / Evaluate▼

Drop correlated features, train models, tune hyperparameters (grid search / random / Bayesian), evaluate against holdout data. Output: trained model artifact + evaluation report → sent to Model Registry.

Stage 4: Deploy▼

Push trained model to production. Patterns: REST API (real-time), batch scoring (scheduled bulk), blue/green deployment (zero-downtime). Docker containers + Kubernetes = standard production infrastructure.

Stage 5: Monitor + Feedback Loop▼

Watch for drift. Feedback loop: drift detected → diagnose → collect new data → retrain → redeploy → monitor. What to monitor: prediction distribution, feature distribution, business KPIs, and ground-truth labels when available.

⚠️ Critical

Deploying without monitoring is flying blind. You won't know the model is broken until users complain.

E-commerce Pipeline (Full Example)

🏭 Production Pipeline Walkthrough

Collect: User clickstream (10M events/day), 50K SKUs, 100K purchases/day.
Store: Online=Redis <5ms. Offline=S3/Parquet 90-day history.
Train: Two-tower neural network. Weekly full retrain + daily incremental update.
Deploy: TensorFlow Serving on Kubernetes. Blue/green deployment.
Monitor: Track CTR, coverage, embedding drift. Auto-investigate if CTR drops >5% WoW.

⚠️ Three Cardinal Mistakes

1. No feature store. Teams engineer features differently → training-serving skew.

2. Correlated features not removed. ≥0.95 correlation = noise without signal.

3. No feedback loop. No monitoring = flying blind.

🏭 Industry Registry Tools

MLflow Model Registry — open-source, most widely used. Amazon SageMaker Model Registry — native AWS, approval workflows. Google Vertex AI — GCP native with AutoML integration. All implement the same concept the professor described.

⚠️ Common Mistakes

1. Saving only model weights, not metadata. A .pkl without params, lineage, and metrics is useless 6 months later.

3. No stage governance. Model should never jump from "just trained" to "production" without review.

⚡ Exam Tip

The professor's scenario: two engineers collaborating. The registry creates shared truth — audit which data and code produced any model. Exam may ask you to trace the full lineage chain: data → code → hyperparameters → artifact → metrics.

Model Drift, Feedback Loop & Monitoring

The world changes. Models that don't notice this silently rot in production.

📘 Definitions

Model Drift: Performance degradation caused by changes in production data vs. training data.
Data Drift: P(X) changes — input feature distribution shifts.
Concept Drift: P(Y|X) changes — same features now predict different outcomes.

Fraud Detection Case Study

🏭 Real Drift Scenario — 4-Hour Resolution

Context: Model trained Q1 2025. Q2: fraudsters shift to small "test" transactions before large fraud.

Data drift: Average transaction amount drops 30%.

Concept drift: False negative rate rises from 2% → 8%.

Alarm fires: FN crosses 5% threshold.

Response: Lineage tracker confirms model v2.1 trained Q1. Retraining scheduler collects Q2 data, relabels, retrains as v2.2. Blue/green swap. FN restored to 1.8%.

Total time: 4 hours. Without monitoring: weeks undetected → estimated $500K in missed fraud.

Monitoring Tool Landscape (2025–2026)

🔬 Research — Evidently AI, kgt.solutions (2025–2026)

Evidently AI: Leading open-source library (20M+ downloads). KS-test, PSI, Jensen-Shannon divergence. Integrates with MLflow, Grafana, Airflow.
Arize AI: Fastest-growing managed platform, real-time alerting at scale.
Fiddler AI: Purpose-built for regulated industries (compliance + explainability).
Stat: 85% of ML models fail in production, most silently. MLOps market: 35.5% CAGR → $19.55B by 2032.

⚠️ Critical Mistakes

1. Only monitoring system health (CPU, latency). A model can return predictions at 200ms while being completely wrong about them.
2. No drift threshold. "Seems worse" is not actionable. Define: "If F1 drops below 0.80 for 3 consecutive days, trigger retraining."
3. Retraining on corrupted data. If data is corrupted (not just shifted), retraining makes it worse. Diagnose root cause first.

⚡ Exam Tip — The Four Feedback Loop Components

Professor mentioned these four explicitly: Alarm Manager (fires when metric threshold breached), Lineage Tracker (confirms which model is running and on what data), Retraining Scheduler (orchestrates collection + pipeline), Model Update Pipeline (evaluates, versions, deploys new model).

Session 6–7 · Data Engineering

Data Pre-processing Techniques

Impute → deduplicate → split → normalise. The order matters critically.

📘 Correct Pre-processing Order

1. Impute missing values → 2. Remove outliers → 3. Remove duplicates → 4. Conform formats → 5. Split stratified → 6. Normalise (fit on train only)

1. Data Imputation

📘 Definition

Imputation replaces missing values with a substitute to prevent row loss. Common strategies: mean (normal dist), median (skewed), mode (categorical), KNN (model-based).

# Mean imputation: x_missing = (1/n-k) × Σ x_observed
from sklearn.impute import SimpleImputer
imputer = SimpleImputer(strategy='median')
X_train['tenure'] = imputer.fit_transform(X_train[['tenure']])
# ALWAYS fit imputer on training data only, then transform all splits

2. Duplicate & Outlier Removal

⚠️ Most Important Rule: Deduplicate BEFORE Splitting

A duplicate landing in both train and test = data leakage. Evaluation metrics become deceptively optimistic — the model is tested on data it partially trained on. The professor stated this as the most emphatic pre-processing warning.

3. Data Splitting

SplitTrainValTestBest For
60-20-2060%20%20%Medium datasets with hyperparameter tuning
80-2080%—20%No tuning needed, simpler models
80-10-1080%10%10%Large datasets
90-1090%—10%Very limited data

Random: Rows randomly assigned. May not preserve class proportions. Dangerous for imbalanced data.

Stratified: Each split mirrors class distribution. If 5% churn, each fold has 5% churners. Always use for imbalanced classes.

Validation set: tunes hyperparameters. Test set: touched ONLY ONCE for final evaluation. Iterating on test set converts it into a de-facto validation set and inflates reported performance.

4. Normalisation

📘 Z-Score Standardisation

z = (x − μ) / σ. Centers at 0, unit variance. Best for roughly normal distributions. No bound on output range.

📘 Min-Max Normalisation

x' = (x − x_min)/(x_max − x_min). Scales to [0,1]. Sensitive to outliers — one extreme value skews all others.

⚠️ Fit Scaler on TRAINING Data Only

Fitting on the full dataset (including test) lets test set statistics influence normalisation = data leakage. Use training mean/std to normalise both training AND test. Never recompute on test data.

5. Overfitting vs. Underfitting

ConditionAnalogySymptomFix
UnderfittingAdult in 5-year-old's shirtHigh train + test errorMore complex model, more features
Best FitRight-sized shirtLow train + test error—
Overfitting5-year-old in adult's shirtLow train error, HIGH test errorRegularisation (L1/L2), more data, simpler model

Full Worked Example

🏭 Churn Dataset Pre-processing (10,000 rows)

1. Impute: tenure 800 missing → median (24 mo); monthly_charges 50 missing → mean ($64.70).
2. Outlier: tenure=900 (75 years, impossible) → remove.
3. Deduplicate: 120 exact dupes → remove. Now 9,879 rows.
4. Conform: "Credit Card"/"CC" → standardise to "credit_card".
5. Stratified split 70-15-15: churn rate=26.5% preserved in each fold.
6. Z-score: monthly_charges → μ=0, σ=1. Fit on train only, apply to all splits.

Session 7 · Data Collection

Data Collection & Ingestion

The pipeline's entry point — where most quality problems start.

📘 Core Principle

Ingestion receives data from external sources into the pipeline. The source is outside your control and changes without warning. Validate at ingestion to catch problems immediately — not three stages later.

Source Classification

TypeExamplesIngestion Pattern
DatabasesMySQL, PostgreSQL, MongoDBReplicas / CDC (transaction logs)
APIsREST, GraphQL endpointsPolling (pull) or webhooks (push)
FilesCSV, JSON, Parquet, ExcelObject storage (S3, GCS, ADLS)
StreamsKafka, Kinesis, Pub/SubReal-time event-by-event consumer
OwnershipDefinitionTrustCompliance
First-partyYour own systemsHighestFull control
Second-partyPartner's dataMediumNegotiated access
Third-partyPurchased / aggregatedLowestGDPR/CCPA risk

Structured: Fixed schema, rows/columns (relational DBs, CSV). Easiest to process.
Semi-structured: Flexible schema, self-describing (JSON, XML). Common in APIs.
Unstructured: No schema (images, audio, free text). Hardest — requires preprocessing before feature extraction.

Ingestion Patterns Compared

PatternDefinitionLatencyBest For
BatchBulk transfer on schedule (hourly/daily)HoursReporting, model training, historical backfill
Micro-batchVery frequent small batchesSecondsNear-real-time without full streaming overhead
StreamingEvent-by-event, continuousMillisecondsFraud detection, real-time recommendations

Push vs Pull

📘 Pull Ingestion

You fetch on a schedule. Poll API hourly, query DB on cron. Simpler. You control timing. Risk: miss events between polls. Best for 3rd-party APIs and non-streaming databases.

📘 Push Ingestion

Source sends to you. Webhooks, Kafka event streams. Lower latency. Source drives timing. Requires robust receiving infrastructure. Best for clickstreams, real-time transactions.

Change Data Capture (CDC)

🏭 CDC: The Backbone of Near-Real-Time Pipelines

Reads the database transaction log (Postgres WAL, MySQL binlog) to capture INSERT, UPDATE, and DELETE events as they occur — no full table scans needed. Streaming + CDC = near-real-time feature freshness at low compute cost.

-- Incremental extraction with high-water mark (simpler alternative)
SELECT * FROM source_db.orders
WHERE updated_at > '2025-03-15 00:00:00'   -- last checkpoint
  AND updated_at <= '2025-03-16 00:00:00'; -- current run boundary
-- ⚠️ LIMITATION: misses hard deletes — use CDC if deletes matter!

What Goes Wrong at Ingestion

⚠️ Four Common Ingestion Failures

1. Schema change: Source column renamed without notice → downstream transforms break silently.
2. Format drift: Date changes from YYYY-MM-DD to MM/DD/YYYY → NaT/null downstream.
3. Duplicates / late records: Kafka processes message twice; late-arriving event bypasses join window.
4. API rate limits/outages: 3rd-party returns 429 → ingestion stops silently, model trains on stale data.
Solution: Data contracts — formal schema agreements enforced at ingestion. Fail loudly, immediately.

🔬 Research — datavidhya.com, estuary.dev (2025–2026)

Modern tools (Estuary Flow) unify batch + streaming with sub-100ms latency. Schema versioning built-in. Incremental extraction fails silently on hard deletes — a common exam scenario. CDC is the correct solution.

⚡ Session 7 Core Principle

"Validate at the loading dock — catch source changes immediately." This is the concept of data contracts: formal producer-consumer schema agreements. A renamed column = broken contract. Catching it at ingestion = 1 hour. Catching it after model training = 3 days.
