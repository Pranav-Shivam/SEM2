Modern Data Infrastructure — Overview

Session 5 · Prof. Sunil Bhutada · The foundational pipeline: Source → Ingest → Transform → Serve

📘 Core Definition

Modern Data Infrastructure is a layered, interoperable stack of specialised tools that automates the movement, transformation, and serving of data — from raw source systems to analytics, ML models, and downstream applications. It replaces brittle, hand-coded scripts with managed, observable, reproducible pipelines.

The Data Pipeline Architecture

Every data system can be reduced to three primitives: input → process → output. In data engineering this expands to a five-layer stack:

🌍 Real-World Connection

A modern e-commerce company like Flipkart has hundreds of such pipelines running simultaneously: clickstream → Kafka → Spark → Redshift → recommendation engine. Each pipeline runs on a schedule orchestrated by Apache Airflow.

Three Axes of Categorisation

1 · By Centralisation ▼

Centralised — Single repository (data warehouse). Easy governance, single source of truth. Risk: bottleneck at scale.

Decentralised — Each business unit owns its data. Agile but risks silos and inconsistent definitions.

Hybrid — Shared master data centrally, domain-specific data locally. Most common in large enterprises.

Federated — Multi-party data sharing with each party retaining sovereignty. Common in research consortia and healthcare networks.

2 · By Storage Paradigm ▼
  • Data Warehouse — Structured, cleaned OLAP; Star/Snowflake schema; schema-on-write
  • Data Lake — Raw, all formats; schema-on-read; ELT pattern; cheap object storage
  • Data Lakehouse — Lake + ACID transactions + governance (Delta Lake, Iceberg, Hudi)
  • Data Mart — Filtered subset of a warehouse for one department
3 · By Governance Model ▼

Data Mesh — Domain ownership + data as product + self-serve platform + federated governance. Introduced by Zhamak Dehghani (2019).

Data Fabric — Unified virtualisation layer across heterogeneous sources. Integrates lakes, warehouses, SaaS via CDC and APIs without moving data.

Key Drivers of Modernisation

The professor highlighted three major forces in the past two decades:

  • Rise of major public cloud providers — AWS, GCP, Azure made it easy to deploy and scale without owning hardware
  • Columnar databases at scale — Redshift (2013), BigQuery (2010), Snowflake (2012) enabled cheap analytical queries on petabytes
  • Drop in storage costs — S3 costs ~$0.023/GB/month; archival tiers as low as $0.004/GB/month (Glacier Deep Archive)
⚠️ Exam Tip

The professor explicitly said the pipeline architecture is: Sources → Ingestion → Transformation → Serving → (Data Analytics / ML). Know this flow and be able to label Reverse ETL as the backward direction from Warehouse → Apps.

🔬 Research Addition · Source: Gartner 2024

According to Gartner's 2024 Data Management survey, 72% of enterprises now run a hybrid data architecture (both a cloud data warehouse AND a data lake). The trend toward Data Lakehouses is accelerating — Databricks reported 9,000+ organisations using Delta Lake by end of 2024.

Data Ingestion & ETL/ELT

Getting data in — the interface, format, and tooling choices that matter most

📘 Definition

Data Ingestion is the process of obtaining and importing data into a system for immediate use or storage. The data engineer's first question is always how to get the data — not who owns it. The interface and format matter more than the source.

Interface Options

The professor covered six types of interfaces through which data can be obtained:

Database Connection ▼

Direct JDBC/ODBC connection to PostgreSQL, MySQL, Oracle. Read with SQL queries. Suitable for relational, well-structured data. Requires network access and credentials.

REST API ▼

HTTP calls returning JSON or XML. Common for SaaS tools (Salesforce, HubSpot, Google Analytics). Requires handling pagination, rate limits, and auth tokens.

Streaming Platform ▼

Apache Kafka, AWS Kinesis, Google Pub/Sub. Continuous data flow. Requires a consumer group and offset management. Real-time use cases.

File / Object Storage ▼

CSV, JSON, Parquet files in S3 buckets, SFTP servers, or network shares. Batch processing. Very common for third-party data dumps.

HDFS / HBase ▼

Hadoop distributed file system or HBase NoSQL. Used when data already lives in an on-premise big data cluster.

Data Warehouse / Lake ▼

Reading from an existing Snowflake, Redshift, or BigQuery table — common when building a secondary pipeline or cross-team data sharing.

Data Formats

SQL table rows with defined columns — MySQL, PostgreSQL, Oracle. Fixed schema. Easy to query with SQL. High veracity.

Also includes Fixed-Width Format (FWF) — older mainframe data with columns at fixed character positions. Common in banking and government.

JSON — default format from REST APIs. Nested, flexible, but verbose. Requires parsing.

XML — older format used in SOAP services and enterprise systems (SAP, banking). More verbose than JSON.

CSV — comma-separated values. Simple, widely supported. No schema enforcement — type inference required.

Log files — structured text output from applications. Often require regex parsing.

Images, audio, video, free text — no predefined schema. Require specialised tools (computer vision, NLP, ASR).

Memory aid: TAVI — Text, Audio, Video, Image — the four types of unstructured data.

Apache Kafka stream output — records with key, value, topic, partition, offset, timestamp. Consumed by stream processors (Flink, Spark Streaming).

The professor noted Kafka supports long data retention, enabling historical replay — the foundation of Kappa architecture.

Reverse ETL

When the warehouse talks back — pushing clean data back into business tools

📘 Definition

Reverse ETL is the process of moving cleaned, modelled data OUT of a data warehouse (Snowflake, BigQuery) BACK INTO operational business tools — Salesforce, HubSpot, Zendesk, Slack — so that sales, marketing, and support teams can act on warehouse-quality data inside the tools they already use.

Warehouse

The professor used a software engineering analogy: Forward Engineering goes from requirements → code. Reverse Engineering goes from code → back to requirements. Similarly, ETL = apps → warehouse; Reverse ETL = warehouse → back to apps.

Reverse ETL (backward)Sales, Marketing, SupportActivate warehouse data in tools they use dailyCensus, Hightouch

Use Case Examples

Sales Team: Lead Scoring in Salesforce ▼

A data engineer pulls customer data from a CRM, trains a lead-scoring ML model, and stores the scores in Snowflake. Without Reverse ETL, the sales team must use a separate dashboard. With Reverse ETL, the lead scores are pushed back into Salesforce — the SDR sees "Score: 87 / 100" directly on the contact record without ever opening Snowflake.

Marketing: Audience Segmentation in HubSpot ▼

Warehouse query identifies "users who purchased in the last 30 days, age 25–35, mobile users." This segment is pushed to HubSpot as a contact list → email campaign is triggered automatically. No CSV export required.

Ad Tech: Cost-Per-Click Optimisation ▼

The professor's example: data from an ad-tech platform is ingested into the warehouse. Statistical models compute optimal cost-per-click (CPC) bids. These optimised bids are fed BACK to the ad-tech platform via Reverse ETL — closing the loop.

🔬 Research Addition · Source: Census.io State of Reverse ETL 2024

Reverse ETL adoption grew 3× from 2022–2024. Top use cases: (1) CRM enrichment — 68% of teams, (2) Marketing personalisation — 54%, (3) Customer success alerting — 41%. Top tools in 2024: Census, Hightouch, and Polytomic.

⚠️ Exam Tip

The professor taught this at the START of the session as a conceptual framing. It will appear in exams as: "What is Reverse ETL and how does it differ from ETL?" Answer: ETL = apps → warehouse; Reverse ETL = warehouse → apps. Users: ETL = analysts/engineers; Reverse ETL = sales/marketing/support.

⚠️ Common Mistake

Students often confuse Reverse ETL with data replication or backup. It is specifically about pushing processed, modelled data (not raw data) back into operational tools (not into another data store). The key word is "activation."

Data Transformation & Modeling

Converting raw ingested data into analytics-ready and ML-ready formats

📘 Definition

Transformation is the "T" in ETL or ELT — the broad set of operations that convert raw source data into a target format optimised for analysis or ML. It ranges from simple time-zone conversion to complex multi-table aggregations with business logic filters.

Data Modeling is a more specific type of transformation that structures data into formats (like Star Schema) optimised for analytical queries. Data models are usually represented as tables in a data warehouse.

Transformation: Simple vs. Complex

Timezone conversion: Converting a timestamp stored as IST (Indian Standard Time) to CST (US Central Standard Time) for a global analytics report.

Column renaming: cust_id → customer_id to match warehouse naming conventions.

Type casting: Converting string "2024-01-15" to a proper DATE type.

Null handling: Replacing NULLs with 0 or 'Unknown' based on business rules.

Multi-table aggregation: JOINing five tables, applying CASE logic, GROUP BY with HAVING filters, and computing derived metrics.

Example: Computing "Customer Lifetime Value" requires joining orders, returns, discounts, and customer tables, then applying business-defined LTV formula.

Sessionisation: Grouping raw clickstream events into "sessions" based on 30-minute inactivity windows — requires window functions and lag/lead operations.

PII (Personally Identifiable Information) protection is a key concern. Transformations can include:

  • Hashing: email@domain.com → SHA256 hash — no one can reverse it
  • Masking: 4532-xxxx-xxxx-7890 — show only last 4 digits
  • Tokenisation: Replace with a random token that maps back only in a secure vault

The professor noted basic ingestion tools provide SOME transformation capability, but for complex PII handling and business logic, dedicated tools like dbt are needed.

dbt — The Transformation Tool

🌍 Industry Tool: dbt (data build tool)

The professor mentioned dbt as "a wonderful tool" for transformations. In modern data stacks, dbt has become the standard for ELT transformations — you write SQL models, dbt compiles them into the right dialect, runs them in your warehouse, and manages dependencies between models. It also runs tests, generates documentation, and tracks data lineage.

2024 stat: dbt has 40,000+ active users and is used by Spotify, Airbnb, JetBlue, and most Fortune 500 data teams.

Storage Types: Block, Object & File

Three fundamental paradigms — how data sits on disk and how it's accessed

📘 Three Storage Paradigms
  • Block Storage — Splits data into fixed-size blocks with unique IDs; retrieved via specific protocols. No hierarchy.
  • Object Storage — Stores data as self-contained objects with metadata and a unique key. No hierarchy. Immutable once written.
  • File Storage — Saves data as files in hierarchical folders. Accessed via path. Traditional, most familiar.

The "faster and more beautiful" quote referred to the near-silence of SSD-based systems vs. the spinning noise of HDD arrays.

Block Storage Deep Dive

RAID — Redundant Array of Independent Disks ▼

An array of multiple disks that together appear as one block device to the OS. Provides redundancy (RAID 1/5/6) and/or performance (RAID 0/10) by distributing and replicating data across drives.

SAN — Storage Area Network ▼

Virtualised block storage devices served over a dedicated high-speed network. Creates a pool of storage that can be assigned to servers as virtual disks. Common in on-premise enterprise data centres.

AWS EBS — Elastic Block Store ▼

Cloud SAN equivalent for EC2. Stores data separate from the VM but in the same availability zone for low latency. Data persists when the EC2 instance stops. Automatically replicates to at least 2 host machines. Suitable for databases requiring high durability.

Object Storage Deep Dive

📘 Key Properties of Object Storage
  • Key-value store — each object has a unique key (like a URL)
  • Immutable — cannot append or update in place; must rewrite the full object
  • Stores metadata alongside data (content-type, size, custom attributes)
  • No hierarchical directory structure (the "/" in S3 keys is just a naming convention)
  • Supports random reads via byte-range requests
  • Serverless — no cluster to manage; virtually unlimited scale
🔬 Research Addition · Source: AWS S3 Documentation 2024

Amazon S3 stores more than 300 trillion objects and regularly peaks at more than 100 million requests per second. S3 Standard offers 99.999999999% (11 nines) durability. Data is stored across at least 3 Availability Zones. Cost: ~$0.023/GB/month for standard; $0.004/GB/month for Glacier Deep Archive.

File Storage Deep Dive

File storage uses a hierarchical tree of directories. Characteristics of a file:

  • Finite length (stream of bytes)
  • Supports append operations
  • Supports random access reads/writes
  • Organised in a directory hierarchy (path-based access)

Types: Local Disk (NTFS on Windows, EXT4 on Linux), NAS (Network Attached Storage over LAN), Cloud File Systems (AWS EFS — Elastic File System for multi-VM shared access).

⚠️ Exam Tip

The professor specifically asked the class this: "Which is faster — seeking two blocks on the same track vs. two blocks on separate tracks?" Answer: Same track = no seek time. Different tracks = requires a seek (head movement). This is why HDFS uses 128MB blocks — to maximise sequential reads and minimise seeks.

Storage Systems & HDFS

Distributed vs. single machine, consistency models, and the Hadoop filesystem

📘 Key Concepts

Distributed Storage — Data spread across multiple servers for redundancy, scalability, and fault tolerance. Examples: HDFS, S3, Apache Spark storage.

Consistency Models: Strong consistency — every read sees the latest write (guaranteed, but slower). Eventual consistency — reads will EVENTUALLY see the latest write (faster, but risk of stale reads).

Single Machine vs. Distributed Storage

The professor used a family analogy: "When you're single, a one-room house is enough. When your family grows, you need more space." Similarly, as data grows beyond one server's capacity, distributed storage is necessary.

🌍 Why Distributed Storage?
  • Scale: Single servers max out (typically 256TB with NVMe). Petabyte-scale data requires dozens of nodes.
  • Redundancy: If one server fails, data on other servers remains available.
  • Performance: Parallel reads/writes across multiple nodes = higher throughput.

HDFS Architecture

📘 HDFS Core Components
  • NameNode (Master) — Stores metadata only: file-to-block mappings, directory tree, permissions. Lives in RAM for fast access. Does NOT store actual data. The "card catalogue."
  • DataNodes (Slaves/Workers) — Store actual data blocks on local disk. Send heartbeats every 3 seconds. Send full block reports every hour.
  • Secondary NameNode — NOT a backup/failover! Periodically merges the edit log with the FS image to prevent the edit log from growing too large. A "helper" node.

HDFS Block Design

Block Size: 128MB default — Why so large? ▼

Traditional filesystem blocks are 4KB. HDFS uses 128MB (configurable to 256MB) because:

  • Big data workloads do sequential reads of huge files — large blocks reduce the number of seek operations
  • Fewer blocks = less metadata in the NameNode's RAM
  • Reduces overhead of establishing many small TCP connections to DataNodes
Replication Factor: 3 — How does failure recovery work? ▼

Each block is stored on 3 different DataNodes (ideally on different racks for rack-level fault tolerance).

When a DataNode goes silent (no heartbeat for ~10 minutes), the NameNode detects it and automatically triggers re-replication — finding nodes with only 2 copies and asking them to copy to a healthy 3rd node. This is transparent to the user.

Result: You can lose ~33% of your cluster and not lose any data. At 1,000 nodes with 3% annual failure rate (~30 failures/year), you'd have a node fail roughly every 12 days — HDFS handles all of them automatically.

Data Locality — The most important HDFS optimisation ▼

When a MapReduce or Spark job processes a file, the scheduler TRIES to run the compute task on the SAME node that holds the data block. Moving computation to data is vastly cheaper than moving terabytes of data across the network to computation.

This is why Hadoop was designed as a "shared-nothing" architecture — each node has its own storage and compute, and they rarely need to talk to each other.

⚠️ Common Misconception

  • Also alive: Amazon EMR continues to run Hadoop/Hive/Spark workloads at massive scale

Caching: RAM-Based Storage & Redis

Ultra-fast data access patterns — Memcached, Redis, and the cache hit/miss cycle

📘 Definition

Cache / Memory-Based Storage — Storage that uses RAM instead of disk for ultra-low-latency data access. RAM offers ~50,000 MB/s bandwidth vs. ~3,500 MB/s for SSD. The tradeoff: RAM is volatile (data lost on power-off) and expensive. Used to serve hot data quickly and reduce load on backend databases.

Redis Cache Hit / Miss Cycle

The professor walked through this exact 3-step logic:

Step 1: Check Redis Cache
→ Cache HIT? Return data immediately (sub-millisecond)
Step 2: Cache MISS?
→ Query the SQL/PostgreSQL persistent database
Step 3: Prime the Cache
→ Store the fetched data in Redis for next time
🌍 Real-World Example

Twitter uses Redis to cache the timelines of users with <20,000 followers. When you open Twitter, your timeline is served from Redis in <1ms. For users with millions of followers (celebrities), timelines are computed differently. This pattern of "cache for common cases" is universal in high-traffic web services.

Redis is used by 60% of the Fortune 100 companies. Typical cache hit rates in production: 80–95% for well-configured caches. A 90% cache hit rate means 90% of requests never touch the database — reducing DB load by 10×. Average Redis read latency: 0.1–0.3ms vs. 10–100ms for a PostgreSQL query.

Workflow Orchestration

Managing N × N pipelines — why you can't just use Python scripts and cron jobs

📘 Definition

Workflow Orchestration is the automated management, scheduling, monitoring, and error-handling of a data pipeline's tasks and their dependencies. An orchestrator runs each task when its upstream dependencies are complete, retries on failure, backfills missed runs, and sends alerts. It treats a pipeline as a DAG (Directed Acyclic Graph) of tasks.

The Problem Without Orchestration

The professor asked: "What if this has to run multiple times? How long can you manage all of this with Python and stored procedures?"

⚠️ The Fragile Script Problem

As the number of data sources grows from 1 to N, and each source requires its own ingestion script + transformation logic, maintaining interconnected hard-coded Python scripts + complex SQL procedures becomes:

  • ❌ Fragile — one script failing silently breaks downstream scripts
  • ❌ Error-prone — no retry logic, no alerting
  • ❌ Hard to schedule — cron jobs can't handle inter-task dependencies
  • ❌ Unobservable — no centralised view of what ran, what failed, what's pending

What is a DAG?

📘 Directed Acyclic Graph (DAG)

A DAG is a set of nodes (tasks) connected by directed edges (dependencies) with no cycles (a task cannot depend on itself or create a circular dependency).

In Apache Airflow, you define a DAG in Python: each task is an operator (BashOperator, PythonOperator, SparkSubmitOperator), and you wire them with >> to declare order.

# Example Airflow DAG
from airflow import DAG
from airflow.operators.python import PythonOperator
from datetime import datetime

with DAG('daily_etl', schedule='@daily', start_date=datetime(2024,1,1)) as dag:
    extract_task = PythonOperator(task_id='extract', python_callable=extract_fn)
    transform_task = PythonOperator(task_id='transform', python_callable=transform_fn)
    load_task = PythonOperator(task_id='load', python_callable=load_fn)

    # Declare dependency chain
    extract_task >> transform_task >> load_task
🌍 Analogy from the Lecture

The professor described the orchestrator as a "project manager for data." It knows the task dependencies, kicks each off at the right time, chases failures with retries, and alerts you — so the pipeline runs itself daily without manual babysitting.

DataOps & the CTCD Pipeline

DevOps thinking for data — automate, version, test, monitor the full data lifecycle

📘 Definition

DataOps is an agile, process-oriented methodology that applies DevOps and Agile principles to data analytics and ML workflows. It automates the entire data lifecycle: ingestion, transformation, testing, deployment, and monitoring — so changes ship quickly AND safely.

CTCD (the professor's emphasis) stands for Continuous Training and Continuous Deployment — the ML-specific extension of CI/CD. Not just deploying code but continuously retraining models and deploying updated model artifacts.

DevOps vs. DataOps — Key Differences

Main riskBroken codeBroken data (silent, hard to detect)

The Four Pillars of DataOps

1 · Automation / CI-CD (Continuous Integration & Delivery) ▼

Every code change to a data pipeline triggers an automated build → test → deploy cycle. Tools: GitHub Actions, Jenkins, GitLab CI. For ML: MLflow, Kubeflow Pipelines, SageMaker Pipelines.

Benefit: No manual deployments. A dbt model change by a junior engineer is automatically tested and deployed to production — safely.

2 · Version Control (Code + Data + Schemas) ▼

Code versioning (Git) — standard. Data versioning — newer challenge. Tools: DVC (Data Version Control), Delta Lake time travel, Iceberg snapshots.

Schema versioning — Confluent Schema Registry (Kafka), AWS Glue Schema Registry. Ensures producers and consumers agree on the data contract.

The professor's point: DevOps only versions code. DataOps must also version data and schemas because "data outlives code."

3 · Automated Testing (on Values, not just Logic) ▼

Unit tests check logic: "does this function return the right value for this input?" Data tests check values: "is the number of rows in today's pipeline between 100,000 and 200,000? Are nulls in 'customer_id' column <0.1%? Is 'age' always between 0 and 120?"

Tools: Great Expectations, dbt tests, Soda Core. These are data quality guardrails that run after every pipeline execution.

4 · Monitoring & Observability ▼

Four core signals to monitor in every data pipeline:

  • Freshness — Is data up to date? "Last updated: 2 hours ago" when it should update every 15 min = problem.
  • Volume — Right number of rows? Today's table has 0 rows when it normally has 1 million = upstream failure.
  • Schema — Have column names or types changed unexpectedly? A column renamed upstream without notice breaks all downstream queries.
  • Quality — Are values in expected ranges? Null rates within tolerance? Referential integrity maintained?

Tools: Monte Carlo, Bigeye, Anomalo (dedicated data observability), or Datadog + custom monitors.

Cloud Foundation & Security

🌍 Cloud Foundation in DataOps

The professor mentioned cloud foundation and security foundation as underpinning DataOps. Key principles:

  • IAM (Identity and Access Management) — Who can access which data? Principle of least privilege.
  • Encryption — At rest (S3-SSE, Snowflake encryption) and in transit (TLS 1.3).
  • VPC / Network Security — Data pipelines run inside private subnets. Data never travels over public internet.
  • Audit Logging — Every data access is logged for compliance (GDPR, HIPAA, SOC 2).

Query Federation

What is Query Federation? ▼

Query federation pulls data from multiple sources (data lakes, RDBMSs, data warehouses) via a distributed query engine — without centralising data in an OLAP system first. Tools: Trino, Presto, Starburst (managed).

  • Enables ad hoc exploratory analysis across heterogeneous sources
  • Provides read-only access — no risk of modifying production source systems
  • Best for compliance-critical situations (GDPR, HIPAA)
  • Watch out: federated queries touching live OLTP systems can over-load them — always validate resource impact

ML Infrastructure Stack

The layered stack that every data science & ML project needs — from data warehouse to model development

📘 The Eight-Layer ML Stack

Regardless of domain, all data science/ML projects share a common infrastructure pattern. The layers (bottom = most generic, top = most ML-specific):

8. Model Development — Finding a mathematical model that transforms features into outputs. The core ML expertise layer.
7. Feature Engineering — Raw input → sequence of data transformations → model-ready features. Designed by data scientists.
6. Model Operations — Deploy, monitor, and validate all model versions from prototype to production. Tracks metadata about every execution.
5. Versioning — Multiple model versions run side by side for A/B testing. Must isolate algorithms, input data, output data, and pipelines.
4. Architecture — Defines what is computed and how — algorithms, data pipelines, deployment strategy. This is software engineering.
3. Job Scheduler — Keeps data flowing and machines running uninterruptedly. Challenge: stay up 24/7, manage concurrent machines.
2. Compute Resources — On-demand CPU/GPU that blindly executes algorithms. Does not dictate what is computed.
1. Data Warehouse — Bottommost layer. Stores data. Key challenge: discoverability + efficient access + high durability.

The professor showed two inverted triangles: "How much a data scientist cares" decreases as you go DOWN the stack (they care most about Model Development). Infrastructure complements the interests of a data scientist or ML engineer — the stack is designed so ML practitioners can focus on the top layers.

Layer Details

Feature Engineering — Why it matters ▼

Raw input data is rarely fed directly into a model. It goes through a sequence of transformations (normalisation, encoding, window aggregations, imputation). Designing this sequence is a major part of the data scientist's work. Infrastructure challenge: make feature computation efficient and reusable (→ Feature Stores like Feast, Tecton).

Model Operations — What it tracks ▼

Model Operations tracks metadata about every model execution from prototype to production. Imagine 4 versions of an application running simultaneously, each built by a separate data scientist. Model Ops prevents chaos by: deploying models consistently, monitoring their performance metrics over time, and validating that production models haven't drifted. Tools: MLflow, Weights & Biases, SageMaker Model Monitor.

Versioning — Why models need it too ▼

Unlike software (where version B simply replaces A), ML model versions often need to run simultaneously (A/B test: 10% traffic to new model, 90% to old). Evaluating which version is better is non-trivial — it requires statistical tests, not just syntactic correctness. Must isolate: the algorithm code, the training data snapshot, the feature transformation pipeline, and the serving endpoint.

⚠️ Exam Tip

The professor said "Model Development occupies only a tiny part of the end-to-end machinery." This is a common exam point: data scientists often underestimate how much infrastructure effort surrounds the model itself. The stack is designed so model development CAN be quick — because infrastructure handles everything else.

Pipeline Automation

What pipeline automation covers ▼
  • ETL/ELT workflow execution on schedule or trigger
  • Streaming jobs running continuously on Kafka Streams / Flink
  • Trigger-based transfers (e.g., file lands in S3 → Lambda triggers ingestion)
  • Self-healing workflows: automatically rerun failed tasks with backoff strategy (e.g., retry after 5 min, then 15 min, then 60 min)
  • Schema-change detection: upstream schema change triggers automated update of downstream dbt models + re-run of dbt tests + deployment
Event-driven pipelines vs. scheduled pipelines ▼

Scheduled: runs at a fixed time (daily 2AM). Simple. Works when data arrival time is predictable.

Event-driven: runs when something happens (a file arrives, a message hits a queue, a metric threshold is breached). Tools: AWS EventBridge, Azure Event Grid, Kafka Streams. More complex but more responsive.

Data Security — The Three States

🔒 Data At Rest ▼

Data stored in a medium (S3, HDFS, database). Protection: full-disk or object-level encryption. Examples: AWS S3 SSE (Server-Side Encryption), Snowflake AES-256 encryption at rest, LUKS full-disk encryption on Linux.

🔄 Data In Transit ▼

Data actively moving between systems (database → pipeline → warehouse). Protection: message encryption — TLS 1.3 for HTTPS endpoints, mTLS for service-to-service communication, Kafka with SSL/TLS for streaming data in flight.

👤 Data In Use ▼

Data being actively read/processed by an application or user. Protection: Role-Based Access Control (RBAC) — each user/role can only access the data they need. Principle of least privilege. Examples: Snowflake column-level security, AWS IAM policies, Databricks Unity Catalog row-level filters.

Security Practices in Data Engineering

  • IAM (Identity & Access Management) — Who can access which data? Least privilege principle.
  • VPC / Network Security — Data pipelines run inside private subnets. Data never travels over public internet.
  • Audit Logging — Every data access logged for compliance (GDPR, HIPAA, SOC 2).
  • PII Masking — Personally Identifiable Information hashed/masked during transformation.
  • Secrets Management — API keys, DB passwords stored in AWS Secrets Manager, HashiCorp Vault — never hardcoded.
⚠️ Exam Tip

The professor described data security through three states: Data at Rest (encryption of storage medium), Data in Transit (message/TLS encryption), Data in Use (Role-Based Access Control). Know this framework and one protection mechanism for each state.

CT/CD — Continuous Training & Continuous Deployment

The ML-specific extension of DevOps — automating the full loop from code change to model serving

⚠️ Critical Exam Point — CTCD ≠ CI/CD

The professor explicitly stated: "It is not CI/CD — it is CT/CD: Continuous Training and Continuous Deployment." This distinction is important. CI/CD deploys code. CT/CD also continuously retrains models on new data and deploys updated model artifacts.

The Four Phases

Phase 1 · Experimental Development ▼

The ML engineer tests a new idea (e.g., a new feature transformation). They modify the code base, train a model locally or in a sandbox, and validate that the new approach yields higher performance. Output: a new version of the codebase that needs to enter the master repo.

Phase 2 · Continuous Integration (CI) ▼

The engineer creates a Pull Request (PR). This automatically triggers:

  • Unit tests (standard CI)
  • Instantiation of the automated training pipeline → retrains the model with the new code
  • Integration tests or test cases on the trained model
  • Push of the model artifact to the Model Registry

A second engineer manually reviews the PR, validates performance metrics, and approves the merge.

Phase 3 · Continuous Deployment (CD) ▼

Activating deployment triggers a canary deployment: route 5–10% of traffic to the new model. Run an A/B test to compare the new model against the production model. After satisfactory results (statistically significant improvement), promote the new model to replace the old production model.

Phase 4 · Continuous Training (CT) ▼

As soon as a model enters the model registry, it starts deteriorating — data distribution shifts over time (data drift / concept drift). CT activates recurring training: e.g., each day the model is fine-tuned with new training data of that day, re-deployed, and the serving pipeline is rerouted to the updated model. Fully automated.

Case Study: DoorDash ML Platform

A real-world example of the full modern data infrastructure stack — from fragmented pipelines to unified ML platform

📘 Company Background

DoorDash is the largest food delivery service in the US, connecting consumers with merchants via independent contractors called "Dashers." It processes millions of delivery decisions per day — each requiring real-time ML predictions (ETA, surge pricing, restaurant wait times).

The Problem

DoorDash's earlier ML workflows were:

  • ❌ Fragmented — analytics, batch ML, and real-time ML ran on completely separate stacks
  • ❌ Low observability — data quality issues went undetected, causing incorrect ETAs
  • ❌ Slow deployment — weeks to deploy a new model
  • ❌ Frequent data drift — models degraded without automatic retraining triggers
  • ❌ Manual pipeline debugging — no self-service tooling for data scientists

Goal: A unified, reliable, production-grade ML data platform.

Data Pipeline Architecture at DoorDash ▼

Source: Kafka Event Stream (live delivery events) + Historical databases

Ingestion: Realtime Feature Aggregator (streaming) + Historical Aggregator (batch ETL)

Storage: Feature Store (Redis, low-latency) + Feature Warehouse (Snowflake, batch analytics) + Prediction Logs (Snowflake)

Transformation: Historical aggregations compute complex window features (e.g., 3-month store rating trends)

Serving: Prediction Service serves real-time ML predictions to the DoorDash app. Model Training Pipeline handles CT/CD.

CT/CD in Action at DoorDash ▼

The Model Training Pipeline is the CT/CD system. The pipeline: (1) reads training data from Feature Warehouse + Prediction Logs, (2) trains new model, (3) validates against quality thresholds, (4) pushes to Model Store. Production models are refreshed automatically as new data accumulates. A/B testing is baked in via model ID overrides in Prediction Service.

Security & Audit at DoorDash ▼

Only the Model Training Pipeline has write access to the Model Store — generating a complete audit trail of every model change. This is a direct application of the "principle of least privilege" in data security. No engineer can directly push a model to production; it must go through the pipeline.

🌍 Key Takeaway

DoorDash's platform is a textbook implementation of everything covered in Session 5: data pipeline architecture (Source → Ingest → Store → Serve), the ML Infrastructure Stack (Feature Store = Storage Layer, Model Store = Versioning Layer, Prediction Service = Serving Layer), CT/CD pipelines, and the security principle of only authorised pipelines writing to production stores.
