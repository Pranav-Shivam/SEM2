Key concept from TAVI framework (BitsNotes)
💡 TAVI Framework

Modern data comes in 4 modalities: Text · Audio · Video · Image. Data management must handle all four. Historically only numbers were stored; deep learning (CNNs, Transformers) now makes all modalities processable.

⭐ GIGO Principle (from BitsNotes)

Garbage In, Garbage Out — No matter how advanced your ML model, poor-quality training data produces a permanently degraded model that makes wrong predictions on all future transactions. This is why Data Management is the foundation of ML success.

DIKW Hierarchy

Data → Information → Knowledge → Wisdom · Click each level to expand

📖 Definition — Prof. Bhutada

The pyramid represents increasing levels of refinement and abstraction. The y-axis is "more refined and abstract"; the x-axis represents correctness. Each level filters noise from the level below it.

🔮 Wisdom

Wisdom — Distilled and integrated knowledge

Demonstrates high-level understanding. Goes beyond knowing facts to exercising judgment.

Lecture example: "If you don't feel well, see a doctor."

BitsNotes example: "If I want lowest running cost, choose Honda. If I want performance, choose Apache."

🧠 Knowledge

Knowledge — Justified true belief

Information that can be acted upon. Reveals patterns, rules, and generalizations.

Lecture example: "If temperature above 100°F → you most likely have a fever."

BitsNotes example: "My bike gives 25 km/l mileage. Honda: 30 km/l. Apache: 20 km/l."

Explicit vs Tacit Knowledge (BitsNotes)

Explicit: Textbooks, manuals, formulas — codified and transferable.
Tacit: A doctor's diagnostic intuition, a DBA's instinct for query optimization — hard to document. Organizations lose tacit knowledge when employees leave (brain drain).

📊 Information

Information — Data organized and presented in a particular manner

Answers who, what, where, when. Data is structured, categorized, and described.

Lecture example: "Hourly body temperature: 98.6°F, 99.5°F, 100.3°F, 101°F..."

BitsNotes example: "I traveled 50 km, spent 200 INR, used 2 liters of petrol."

🔢 Data (raw material) Raw, unprocessed. No inherent meaning alone.

Data — The raw material of information

Raw, unprocessed facts, symbols, or observations. Data is always mediated — it represents something in the real world but has no meaning without context.

Lecture example: "98.6°F, 99.5°F, 100.3°F, 101°F, ..."

BitsNotes example: The number 1995 alone — could be a year, a price, a temperature reading, a room number.

Formats of Data

Structured · Unstructured · Semi-structured · (Quasi-structured homework)

📖 Schema

A blueprint or structure that defines how data is organized, stored, and related. It specifies the rules, format, and constraints of the data.

📊

Structured

Follows a predefined schema. Rows and columns.

Uses schema-on-write: validates data structure before storage. Faster reads, but schema changes require careful migration.

Example: Excel/CSV tables, RDBMS tables. RDBMS uses ER (Entity Relationship) model.

Advantages: Easy for data analysis; faster to retrieve (binary yes/no answer); 100% accurate results.

Disadvantages: Complex if schema changes; restricting to commit to predefined schema; definite storage format required.

Used for DBMS/RDBMS systems.

📝

Unstructured

No schema. No fixed format.

Examples: News articles, audio/video files, plain text, images, handwritten notes.

Advantages: Wide range of storage options; lots of data with many interpretations; adaptive to schema changes.

Disadvantages: Difficult for analysis; slower to retrieve information.

Used for search engines — gives approximate results (precision/recall metrics).

📧

Semi-structured

Combination of both. Evolving schema.

Example: Emails — From/To/Subject fields are structured; body is free-form unstructured.

Advantages: Evolving schema (unlike fixed structured or non-existent unstructured); highly interoperable.

Disadvantages: Tricky to have a routine for validation and analysis.

Data Models

5 types — click each to deep dive. Research-enriched: ACID vs BASE, scaling, 2025 use cases.

🔮 Research — ACID vs BASE (integrate.io / AWS / Medium 2024)

ACID (SQL): Atomicity — tx fully completes or rolls back. Consistency — DB stays valid. Isolation — concurrent txs don't interfere. Durability — committed data survives crashes. Required for banking, e-commerce.
BASE (NoSQL): Basically Available, Soft state, Eventual consistency — relaxes strict consistency for performance and horizontal scale. NoSQL scales horizontally across commodity nodes; SQL scales vertically (one powerful server — expensive). This is why tech giants (Google, Amazon, Facebook) adopted NoSQL for web-scale applications.

Relational Data Model

  • Data organized into relations; each relation is a set of tuples
  • A table is the visual representation of a relation
  • Each row = one tuple
  • Relations are unordered — you can shuffle rows or columns
  • Uses SQL (Structured Query Language)
  • Connects tables via primary keys and foreign keys (ER model)
💡 Prof. Bhutada's note

RDBMS gives 100% accurate results — SELECT query either returns the correct data or says "no results." Unlike search engines which give approximate results.

Graph Data Model

🔮 Research — Neo4j 2025 (SiliconAngle / Neo4j.com)

Neo4j is used by 50%+ of Fortune 500 companies. In 2025, Neo4j launched Infinigraph — running operational + analytical workloads on one graph at 100TB+ scale. GraphRAG (Graph Retrieval-Augmented Generation) is a major 2025 trend: combining knowledge graphs with LLMs for better AI answers. Key use cases: fraud detection, drug discovery (millions of scientific documents), supply chain, customer 360. Gartner: "graph technologies will be used in 80% of data innovations by 2025." Cypher is now ISO GQL-compliant (2024).

  • Built around nodes (entities) and edges (relationships)
  • Relationships are the priority — not the document content
  • Faster retrieval because relationships are modeled structurally
  • Query language: Cypher (Neo4j), Gremlin
💡 LinkedIn / Facebook

Person A IS_FRIEND_OF → Person B → LIKES → Restaurant → SERVES → Location
This is why LinkedIn finds "mutual connections" so fast — the relationship graph is pre-stored.

Mapping projects to business objectives · Detecting project dependencies · Finding redundancy across projects

Sample DBs: Neo4j, Amazon Neptune, TigerGraph · NoSQL

  • Each document has a unique key
  • Collection of documents ≈ table in relational DB

Normalization

Why un-normalized tables are a problem — and how to fix them

🔮 Research — 3 anomaly types (DigitalOcean / freeCodeCamp 2024)

Un-normalized tables suffer from three anomalies:
Insertion anomaly: Can't add a new Subject without assigning it to a Student first (they share the same table).
Update anomaly: If a student's name changes, must update every row they appear in — miss one and data becomes inconsistent.
Deletion anomaly: Deleting the last student enrolled in a course accidentally deletes the course information.
Normalization was invented by Edgar F. Codd (1970) — the father of the relational model. All modern RDBMS follow his rules.

The problem with un-normalized data

In the lecture, Prof. Bhutada showed a flat table with StudentID, StudentName, CourseName, and SubjectName all in one table. A student enrolled in multiple subjects appeared on multiple rows with their name and course repeated.

⚠️ Problems with un-normalized tables

1. Data Integrity — "DataScience" vs "DataSciences" (inconsistent SubjectName in the lecture example)
2. Highly redundant values — Same StudentName and CourseName repeated for every subject
3. Difficult to interpret — Hard to see clean relationships
4. Not expandable — Adding a new field means updating many rows

Solution: Normalization

📖 Normalization

The process of organizing a relational database to reduce data redundancy and improve data integrity. Split one flat table into multiple related tables.

After normalization, the un-normalized table becomes 4 clean tables:

👤

Student Table

StudentID, StudentName

📚

Course Table

CourseID, CourseName, IsValid, Start/End Year

📖

Subject Table

SubjectID, SubjectName, Type (Core/Elective)

🔗

Mapping Table

StudentID, CourseID, SubjectID

Normal FormRule (simplified)
1NFEach column contains atomic (indivisible) values; no repeating groups
2NFMust be in 1NF; every non-key attribute fully depends on the whole primary key
3NFMust be in 2NF; no transitive dependencies (non-key attributes not dependent on other non-key attributes)
BCNFBoyce-Codd Normal Form — stronger version of 3NF; every determinant is a candidate key
5NFMust be in 4NF; every join dependency is implied by the candidate keys
⭐ What normalization achieves

✅ Reduced data redundancy   ✅ Improved data integrity   ✅ Flexible for expansion

Query Languages

Declarative vs Imperative — the fundamental dichotomy

⚙️ Imperative (Procedural)

Focuses on HOW to do it

  • Step-by-step instructions
  • Developer must define the optimal path
  • System cannot optimize execution
  • Parallel execution is complex
  • More complex to write
  • Output ordering is assured

💬 Declarative

Focuses on WHAT you want

  • State the desired outcome
  • DB query optimizer decides HOW
  • DB can introduce performance improvements without code changes
  • Often enables parallel execution
  • Easier to write; hides implementation details
  • Output order not guaranteed unless specified
Real-world analogy (from lecture)

Imperative version

1. Open the door
2. Go to car parking
3. Open the car
4. Check if bag is not empty
5. If so, take the bag
6. Close the car, come inside
7. Close the door
8. Give the bag to me

Declarative version

"Bring the bag from the car to me if bag is not empty."


Don't care HOW — just tell me WHAT you want done.

Technical code example — retrieving sharks

Imperative (JavaScript)

function getSharks() {
  var sharks = [];
  for (var i = 0; i < animals.length; i++) {
    if (animals[i].family === "Sharks") {
      sharks.push(animals[i]);
    }
  }
  return sharks;
}

Declarative (SQL)

SELECT * FROM animals
WHERE family = 'Sharks';

Same result. The DB optimizer decides which index to use, join order, and parallel execution strategy.

⭐ Key exam insight

SQL is declarative. Procedural/scripted code (JavaScript loops, Python for-loops) is imperative. In SQL you say WHAT; the query optimizer handles HOW — which is why DBs can improve performance invisibly without code changes.

Data Storage Layouts

Row-major vs Column-major — why it profoundly matters for ML workloads

🔮 Research — Parquet internals (MotherDuck / Starburst 2025–2026)

Parquet was created in 2013 (Twitter + Cloudera, inspired by Google's Dremel). Structure: row groups → column chunks → pages. Each column chunk stores min/max statistics — enabling predicate pushdown: query engines skip chunks where values can't match a WHERE clause without reading them. For 1TB / 1000 columns, querying 3 = ~3GB read instead of 1TB (333× I/O reduction). All cloud warehouses (BigQuery, Snowflake, Redshift, Databricks) use columnar storage internally. 2025 challenge: GDPR deletion is expensive for columnar storage since each deleted row's data is spread across separate column chunks.

📄 Row-major (Row format)

  • All columns of a row stored sequentially in memory
  • Data retrieved row by row
  • Good for accessing complete records / samples
  • Faster data writes
  • Suited for OLTP (Online Transaction Processing)
  • Format example: CSV
  • DB example: MS SQL Server

📊 Column-major (Column format)

  • All values of a particular column stored sequentially
  • Data retrieved column by column
  • Good for accessing features (e.g. all values of one attribute)
  • Faster data reads
  • Suited for OLAP (Online Analytical Processing)
  • Format example: Parquet
  • DB example: Amazon Redshift
💡 Why columns matter for ML

In ML feature engineering, you often select specific columns (features) from a dataset with millions of rows. Column-major format lets you read only the columns you need without scanning every row — this is a huge performance win. Prof. Bhutada: "A feature is a column. Column-major stores each column separately — perfect for reading one feature across all records."

Data Serialization

Converting data from one form to another — with real performance benchmarks

🔮 Research — Serialization tax (Medium / AutoMQ 2024–2026)

In microservices, services spend 30–50% of CPU time marshaling/unmarshaling data. Format choice has direct infrastructure cost impact. Protobuf: Google services, gRPC, TensorFlow TFRecord, IoT, financial high-frequency trading — maximum performance. Avro: Kafka (industry standard with Schema Registry), Hadoop, Spark — best for big data pipelines with evolving schemas. JSON: APIs, configs, human-readable exchange — for <100K events/sec, performance difference is negligible. Pickle: Python-only + security risk (arbitrary code execution on deserialization) — never use for cross-system exchange.

📖 Serialization

Converting data from one form to another (e.g., text → binary) to store it (size constraints) or transfer it (bandwidth constraints). Like encryption: serialization and deserialization must use the same algorithm.

OLTP vs OLAP

Online Transaction Processing vs Online Analytical Processing

⚡ OLTP — Online Transaction Processing

  • Purpose: Save transactions immediately
  • Data type: Current, operational data
  • Volume: Many small transactions
  • Optimization: Faster writes
  • Storage: Row format
  • Model: RDBMS / DBMS
  • Example: GPay transactions; office swipe records

📈 OLAP — Online Analytical Processing

  • Purpose: Analytical insights from OLTP data
  • Data type: Historical, consolidated data
  • Volume: Fewer but large datasets
  • Optimization: Faster reads
  • Storage: Column format
  • Model: Data Warehouse (dimensional modeling)
  • Example: GPay spending analysis; attendance calculation
💡 Prof. Bhutada's analogy

OLTP is like a bank completing transactions every day (opening balance → closing balance). OLAP takes that accumulated data and answers "What was our profitability in EU last 3 months?" — the CRED app is a good example: it uses your transaction history (OLTP) to give you spending analysis (OLAP).

OLAP Operations

⬆️

Roll-up

Summarize and aggregate upward

Collapse detail into higher-level summaries. Like rolling up your sleeves — everything gets grouped. E.g., Monthly sales → Quarterly sales → Annual sales.

⬇️

Drill-down

Go from summary to detail

Navigate from high-level to the lowest granularity. E.g., Annual → Quarterly → Monthly → Daily. The profitability investigation example in the lecture: Enterprise → Region → Country → Product → Cost type.

🔄

Pivot / Rotate

Swap rows and columns

Change the axis orientation. E.g., Rows = Product Lines, Columns = Years → Rotate → Rows = Years, Columns = Product Lines. Same data, different perspective.

🔪

Slice & Dice

Filter one or multiple dimensions

Slice: Select a single value for one dimension (like one slice of bread). Dice: Filter on multiple dimensions simultaneously. Prof. Bhutada: "Like slicing a bread loaf — each slice has its own data."

Drill-down investigation example (from slides)

Sharp enterprise-wide profitability dip ↑
→ Sales OK, but profitability down last 3 months
→ Sharp reduction in European region
→ Increase in few countries, flat in others, sharp decline in some
→ Sharp decline in EU countries, last 2 months
→ Direct costs OK, indirect costs UP → Additional tax on some EU products
⭐ Data Warehouse definition (from lecture)

A subject-oriented, integrated, time-variant, non-volatile, historic collection of data for decision-making. Uses dimensional modeling (not ER model like RDBMS). Prof. Bhutada: "Data warehouse is centralized; data fabric is decentralized architecture."
