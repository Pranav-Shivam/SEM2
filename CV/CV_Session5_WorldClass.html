Canny Edge Detection

Slides 4–50 Lab 3
Source note

Canny is the core of the CV5 slide deck (slides 4–50) and the single most exam-likely topic. It was carried over from Session 4 as Prof flagged. Intuition first — why one operator isn't enough

Run a raw Sobel gradient on a real photo and you get a disappointing mess: edges are thick (several pixels wide), noisy (grain fires as fake edges), and broken (a single contour appears as dashes). John Canny's 1986 insight was that edge detection is not one operation but a pipeline of four, each fixing a specific failure of the last.

The analogy: think of it like cleaning up a hand-drawn pencil sketch. First you blur away the smudges (Gaussian — kill noise). Then you find every dark stroke (gradient — candidate edges). Then you thin each stroke to one crisp line (non-maxima suppression — keep only the ridge top). Finally you ink the confident lines and follow faint ones only if they touch a confident one (hysteresis — two thresholds + connectivity). The output is a clean, single-pixel-wide, connected edge map.

Slide 49 calls this "our first computer vision pipeline!" — and that framing is the whole point. Canny is the template for every multi-stage CV algorithm you'll meet after it.

The 4-step pipeline (Slide 15 & Slide 48)

  1. Gaussian smoothing

    Convolve the image with a Gaussian kernel to suppress noise before differentiating. Slide 48 fuses this with step 2 as a single derivative-of-Gaussian filter (associative property of convolution, Slide 30). σ is the key knob: bigger σ = less noise but worse localisation.

  2. Gradient magnitude & angle

    Apply Sobel/Prewitt to get Gx, Gy. Magnitude |G| = √(Gx²+Gy²), direction θ = arctan(Gy/Gx). Slide 23: the gradient "points in the direction of maximum rate of change."

  3. Non-Maxima Suppression (NMS)

    For each pixel, check if its magnitude is a local maximum along the gradient direction. If a neighbour across the edge is stronger, zero this pixel. Thins edges to 1px (Slides 35–38).

  4. Double thresholding + hysteresis linking

    Two thresholds TL, TH (ratio 2:1 to 3:1, Slide 39). Strong (≥TH) = edge. Weak (TL..TH) = edge iff 8-connected to a strong one. Below TL = discarded (Slides 39–47).

Slide 4 — Objectives of Edge Detection (verbatim)

"Good Detection: filter responds to edge, not noise. Good Localization: detected edge near true edge. Single Response: only one point for each true edge point." These three criteria are exactly what the four steps are engineered to satisfy — map each criterion to the step that delivers it (see table below).

Slide 39 — Hysteresis threshold ratio (exam gold)

"Use two thresholds TL and TH which results in two images; Choose the ratio of high to low threshold in the range of 2:1 to 3:1." This exact ratio is a classic one-mark recall question.

Slide 44 — Edge-linking rule (verbatim)

"Strong edges are edges! Weak edges are edges iff they connect to strong. Look in some neighborhood (usually 8 closest)." The "iff connect to strong" + "8-connected" pair is the most testable detail of step 4.

Why This Matters — OpenCV & the CNN link

Canny is one OpenCV call — but knowing the two thresholds are TL, TH (and that order is threshold1=low, threshold2=high) is the Lab-3 trap:

import cv2
img = cv2.imread('road.jpg', cv2.IMREAD_GRAYSCALE)
# threshold1 = T_low, threshold2 = T_high  (ratio ~2:1 to 3:1)
edges = cv2.Canny(img, threshold1=100, threshold2=200, apertureSize=3, L2gradient=True)
# L2gradient=True uses sqrt(Gx^2+Gy^2); False uses |Gx|+|Gy| (faster, less accurate)

DL connection: a CNN's first conv block learns oriented gradient kernels — i.e. it relearns Canny's step 2 by backprop. But CNNs replace the hand-tuned NMS + thresholds with learned non-linearities, which is why deep edge detectors (HED, 2015) beat Canny on cluttered scenes. Real use: lane detection in ADAS, document scanning, medical organ boundary extraction.

Research Enhanced

Canny is still a strong baseline in 2024. A recurring finding (e.g. surveys on edge detection 2022–2024) is that learned detectors like HED / RCF / EDTER (2022) win on semantic boundaries but Canny remains preferred for low-power, real-time geometric tasks (calibration targets, AR markers) because it is parameter-light and deterministic.

Failure modes with evidence: Canny is acutely threshold-sensitive — on texture-rich regions (foliage, hair) a fixed TL/TH floods false edges; under shadows the same physical contour splits because illumination changes the gradient. The modern fix is adaptive/automatic thresholding (Otsu-driven Canny, median-based "auto-Canny", 2015→): set TH=1.33·median, TL=0.66·median per-image.

Worked ExamplesSELECT DIFFICULTY
Given TL = 50 and TH = 120, classify three pixels by gradient magnitude: A = 140, B = 80, C = 30. (Strong / Weak / None)

  1. Rule: |G| ≥ TH → strong; TL ≤ |G| < TH → weak; |G| < TL → none.

  2. A = 140 ≥ 120 → STRONG (definitely an edge).

  3. B = 80: 50 ≤ 80 < 120 → WEAK (edge only if 8-connected to a strong pixel).

  4. C = 30 < 50 → NONE (discarded).

AnswerA = Strong, B = Weak (conditional), C = None. Ratio check: TH/TL = 120/50 = 2.4:1 ✓ (in 2:1–3:1 range).
Key Insight

"Weak" is not a final answer — its fate depends on connectivity. Writing "B is an edge" with no connectivity caveat loses the mark.

A horizontal edge runs left–right. At pixel P the gradient direction θ = 90° (pointing vertically, i.e. across the edge). P's magnitude is 100; its top neighbour is 130, bottom neighbour is 60. After NMS, is P kept? Then with TL=70, TH=150, classify the surviving pixels.

  1. NMS compares along the gradient direction (θ=90° → vertical). P's competitors are top (130) and bottom (60).

  2. Is P (100) ≥ both? 100 ≥ 130? No. So P is suppressed → set to 0. The top neighbour (130) is the ridge maximum and survives NMS instead.

  3. Now threshold the survivor 130 with TL=70, TH=150: 70 ≤ 130 < 150 → WEAK.

  4. So this ridge yields one weak pixel at the old top-neighbour location. It becomes a final edge only if 8-connected to some strong (≥150) pixel further along the contour.

AnswerP is NOT kept (130 > 100 across the edge). The kept pixel (130) is WEAK under TH=150 → survives only via hysteresis link to a strong edge.
Key Insight

NMS and thresholding are independent gates: a pixel can survive NMS (it's the local ridge max) yet still be killed or made conditional by thresholding. Order matters — NMS first, threshold second.

Prove with numbers why two thresholds beat one on a faint but real edge corrupted by noise. Consider a true contour whose magnitudes along its length are: 160, 90, 85, 200, 75 — interspersed with isolated noise spikes of 95 sitting alone in flat regions.

  1. Single threshold @ 100: contour pixels 90, 85, 75 fall below 100 → the real edge breaks into disconnected dots (160 … 200 only). Localisation of a continuous contour is destroyed (violates "single response").

  2. Single threshold @ 70: now the contour survives (all ≥70) — but every isolated 95 noise spike also passes → false edges everywhere (violates "good detection"). No single value wins.

  3. Hysteresis, TL=70, TH=150: 160 and 200 are strong seeds. 90, 85, 75 are weak but each is 8-connected along the contour to a strong seed → promoted to edges. The whole contour is recovered intact.

  4. The isolated noise spike of 95 is weak (70≤95<150) but sits alone — no strong neighbour → discarded. Both criteria satisfied simultaneously.

AnswerNo single threshold can both keep the faint contour (needs low T) and reject lone noise (needs high T). Hysteresis decouples the two: high T finds trustworthy seeds, low T + connectivity grows them while orphan noise dies.
Key Insight

Hysteresis = "trust, then extend trust by contact." It is the same logic as flood-fill from confident seeds — and it is why Canny produces connected contours where Sobel+threshold produces dashes.

Hough Transform for Line Detection

Slides 51–66 Lab 3
Source note

Hough fills slides 51–66 of CV5 and carries a fully worked numerical on Slide 65. Slides only — no live transcript coverage in the May 30 recording. The reading is Gonzalez & Woods 4th Ed, pp. 737–742 (Slide 66).

Intuition first — voting instead of searching

After Canny you have a scatter of edge points, not lines. Slide 51 asks the sharp question: "why aren't we done just by running edge detection?" Because edge points are fragmented (clutter, gaps, noise — Slide 52) and you still don't know which points belong to which line.

The analogy is an election (Slide 53). Rather than testing every possible line against every subset of points (combinatorially hopeless), you let each edge point vote for all lines that could pass through it. Lines that collect many votes are real; noise points scatter their votes inconsistently and never accumulate. Democracy finds the line.

The trick is a duality: a point in image space maps to a line in parameter space, and vice-versa. So "find points that are collinear" becomes "find the parameter-space cell where many image-point lines cross" — a peak in an accumulator.

The image ↔ Hough duality (Slides 55–57)

  1. A line in image space → a point in (m,b) space

    Line y = mx + b is fully described by its slope-intercept pair (m, b) — one point in parameter space.

  2. A point in image space → a line in (m,b) space

    Fix (x0, y0). All lines through it satisfy b = −x0m + y0 — a straight line in (m,b) space (Slide 55).

  3. Two image points → intersection of two Hough lines

    The (m,b) where both Hough lines cross is the single line through both image points (Slide 56).

  4. Accumulate votes & find peaks

    Discretise (m,b) into bins; each edge point increments every bin on its Hough line. Peak bins = most prominent image lines (Slide 57).

Hough — slope-intercept vs polar form
Image line: y = m·x + b Point→line in Hough space: b = −x0·m + y0 (Slide 55) Problem: vertical lines need m = ∞ → slope-intercept blows up Fix — polar/normal form (Slides 61–63): ρ = x·cos θ + y·sin θ ρ = ⊥ distance from line to origin θ = angle the ⊥ makes with the x-axis → bounded, no infinities A point now votes along a sinusoid in (ρ, θ) space.
Slide 53 — Why voting (verbatim)

"It's not feasible to check all combinations of features by fitting a model to each possible subset. Voting is a general technique where we let the features vote for all models that are compatible with it… Noise & clutter features will cast votes too, but typically their votes should be inconsistent with the majority of good features."

Slide 65 — Worked numerical (Prof's own example)

"Determine if (1,2), (2,3) and (3,4) are on the same line using Hough transformation." Using c = −mx + y: for (1,2) → c = −m+2; for (2,3) → c = −2m+3; for (3,4) → c = −3m+4. Solve any two: −m+2 = −2m+3 → m = 1, c = 1. Check the third: c = −3(1)+4 = 1 ✓. All three Hough lines intersect at (m=1, c=1) → the points are collinear on y = x + 1.

Why polar form is mandatory in practice

FormParametersVertical line?Vote shape in param space
Slope-intercept(m, b)m → ∞ (breaks)Straight line

θ ranges over [0°, 180°), ρ over [−D, +D] where D is the image diagonal — both bounded, so the accumulator is a finite 2-D array. That is why every real implementation (and OpenCV) uses (ρ, θ), never (m, b).

Why This Matters — OpenCV & beyond lines

Hough generalises from lines to circles and arbitrary shapes. The two line variants in OpenCV:

import cv2, numpy as np
edges = cv2.Canny(img, 50, 150)           # Hough needs an edge map first
# Standard Hough: returns (rho, theta) peaks
lines = cv2.HoughLines(edges, rho=1, theta=np.pi/180, threshold=200)
# Probabilistic Hough: returns actual segment endpoints (faster, used in Lab)
segs  = cv2.HoughLinesP(edges, 1, np.pi/180, threshold=80,
                        minLineLength=50, maxLineGap=10)
circles = cv2.HoughCircles(img, cv2.HOUGH_GRADIENT, dp=1, minDist=20)

Real use: lane-line detection (ADAS), barcode/grid detection, detecting the iris in eye-tracking, rectifying scanned documents. Pipeline note: Hough always runs after Canny — it consumes the binary edge map, it does not find edges itself.

Research Enhanced

Hough lives on inside deep nets. Deep Hough Transform (TPAMI 2021/2022) bakes the Hough voting operation into a differentiable layer for semantic line detection, and HoughNet (ECCV 2020) uses voting for object detection — evidence that the "let features vote" principle outlived hand-engineered CV.

Failure modes: the classic Hough is memory- and compute-heavy (the accumulator scales with resolution × angular bins) and its peaks blur when many near-parallel lines exist. Probabilistic Hough (random subset of edge points) is the standard 2024 speed fix; coarse-to-fine accumulators handle the memory blow-up.

Worked Examples
What does a single image point (x0, y0) map to in (m, b) Hough space?

  1. Every line through (x0,y0) obeys y0 = m·x0 + b. Rearrange for b: b = −x0·m + y0.

  2. Treating m as the variable, this is the equation of a straight line in (m,b) space with slope −x0 and intercept y0.

AnswerA straight line: b = −x0·m + y0. (One image point ↔ one Hough line — the core duality.)
Key Insight

Point↔line duality is symmetric: an image line is a Hough point, an image point is a Hough line. Memorise the swap.

Reproduce Slide 65: are (1,2), (2,3), (3,4) collinear via Hough? Show the intersection.

  1. Use c = −mx + y. Point (1,2): c = −m + 2. Point (2,3): c = −2m + 3. Point (3,4): c = −3m + 4.

  2. Intersect the first two: −m + 2 = −2m + 3 → m = 1. Then c = −1 + 2 = 1.

  3. Test the third Hough line at m=1: c = −3(1) + 4 = 1 ✓. It passes through (1,1) too.

  4. All three lines meet at the single point (m=1, c=1) → the image points lie on y = x + 1.

AnswerYes — collinear. Common Hough point (m=1, c=1) ⇒ image line y = x + 1.
Key Insight

"Collinear in image ⟺ concurrent in Hough." Three intersecting Hough lines = three collinear image points. That single sentence answers the whole slide.

You must detect a perfectly vertical wall edge (x = 5). Show why slope-intercept Hough fails and how polar form rescues it. Give the (ρ, θ) of the line.

  1. A vertical line x = 5 has slope m = ∞. In (m,b) space its vote-line has infinite slope — the accumulator can't represent it, votes never land. Detection fails.

  2. Switch to ρ = x·cosθ + y·sinθ. For a vertical line the perpendicular from the origin is horizontal → θ = 0°.

  3. At θ = 0°: ρ = x·cos0 + y·sin0 = x·1 + y·0 = x = 5. So every point on x=5 votes for the same cell (ρ=5, θ=0°).

  4. That cell accumulates a sharp peak → the vertical line is detected cleanly. No infinity anywhere.

Answer(ρ, θ) = (5, 0°). Polar form bounds both parameters (θ∈[0,180°), ρ∈[−D,D]), so vertical lines that break the (m,b) form are represented trivially.
Key Insight

The infinity in slope is not a corner case you can ignore — vertical/near-vertical edges are everywhere (walls, poles, doc borders). Polar form isn't an "alternative," it's the only production-grade choice.

Harris Corner Detector

Transcript 0:11–1:02
Source note

Harris is the first half of the actual May 30 lecture — Prof spent ~50 minutes deriving it live, including a full 4×4 numerical. It is genuine CV5 content (Prof: "from your curriculum perspective we are gonna study Harris corner detector", ~29:30) even though it isn't in the handout's 5-topic list. Transcript-grounded; the static slide text for these slides was image-heavy.

Intuition first — slide a window and watch the change

Prof's definition, repeated many times: "a corner is a point where intensity changes in multiple directions" — and equivalently "a corner is where two edges intersect." A flat region changes in no direction; an edge changes in one direction (across it, not along it); a corner changes in every direction.

The test: take a small window (think 3×3) and shift it by a tiny amount (u,v). On a flat patch the contents barely change for any shift. On an edge they don't change if you slide along the edge, but do if you slide across. On a corner, any shift changes the contents a lot. Harris turns "how much does the windowed content change for a shift" into an eigenvalue test.

Why it matters (Prof, ~14:00): corners drive apparent motion / parallax — as you move toward a corner its image position shifts, which is the backbone of ADAS Level-1, structure-from-motion, camera calibration, AR and 3D reconstruction. Harris (Harris & Stephens, 1988) reports corners, edges, AND flats all from one response function.

The derivation Prof did live (Transcript 23:00–52:00)

  1. Window shift & intensity change

    E(u,v) = Σ w(x,y)·[ I(x+u, y+v) − I(x,y) ]². Sum the squared intensity difference over the window for a shift (u,v).

  2. Taylor-expand the shift

    I(x+u,y+v) ≈ I(x,y) + Ixu + Iyv (drop higher orders since u,v are small — Prof stressed this). The I(x,y) terms cancel, leaving E ≈ Σ w·(Ixu + Iyv)².

  3. Write it as a quadratic form

    E(u,v) ≈ [u v] · M · [u v]ᵀ where M = Σ w·[[Ix², IxIy],[IxIy, Iy²]]. Prof labelled A=Ix², B=IxIy, C=Iy².

  4. Eigen-decompose M (the structure tensor)

    λ1, λ2 = eigenvalues = axes of the response ellipse. λmin = slowest-change direction, λmax = fastest.

  5. Corner response R (avoid computing eigenvalues directly)

    R = det(M) − k·trace(M)² with k ∈ [0.04, 0.06]. det = λ1λ2, trace = λ1+λ2. R>0 → corner, R<0 → edge, |R|≈0 → flat.

Change for shift (u,v): E(u,v) = Σ w(x,y)·[I(x+u,y+v) − I(x,y)]² Taylor → E(u,v) ≈ [u v] M [u v]ᵀ Structure tensor: M = Σ w · [ Ix² IxIy ] [ IxIy Iy² ] = [ A B ; B C ] det M = A·C − B² = λ1λ2 trace M = A + C = λ1 + λ2 Response: R = det(M) − k·trace(M)², k ∈ [0.04, 0.06] R > 0 → corner · R < 0 → edge · |R| ≈ 0 (1e-9) → flat
Prof. Dhruba — Transcript (~31:00)

"Corners are basically areas where intensity changes along multiple directions at the same time and they help us in detecting the apparent motion of an object. Corners are also places where two edges intersect."

Prof. Dhruba — Transcript (~53:00) — eigenvalue logic

"If both lambda one and lambda two are small, you are at a flat area. If one of them is large, you are at an edge. And if they are both equally large, you are at a corner… so rather than calculating the gradient, we directly take det and trace — determinant is λ₁·λ₂, trace is λ₁+λ₂."

Prof. Dhruba — Transcript (~57:00) — the Sobel-squared trick

"Why am I taking the gradient of the Sobel operator and not the image? It is just the associative property of convolution. Rather than taking the derivative then squaring, I take the square of the Sobel filter and multiply with the input signal — almost equivalent, cheaper."

Steps to remember (Prof, ~1:00): (1) Gaussian-smooth the image — "if you have noise, the partial-order derivative will be garbled." (2) Compute Ix, Iy (Sobel). (3) Form M = [A B; B C]. (4) Compute R = det − k·trace². (5) Threshold & non-max-suppress R to mark corners.

Why This Matters — OpenCV & motion
import cv2, numpy as np
gray = np.float32(cv2.cvtColor(img, cv2.COLOR_BGR2GRAY))
# blockSize=neighbourhood, ksize=Sobel aperture, k=Harris free param 0.04-0.06
R = cv2.cornerHarris(gray, blockSize=2, ksize=3, k=0.04)
img[R > 0.01 * R.max()] = [0, 0, 255]   # threshold on R, mark corners red
# Sub-pixel refinement: cv2.cornerSubPix(...)
# Modern alternative: cv2.goodFeaturesToTrack (Shi-Tomasi, uses min(λ1,λ2))

Prof's industry link (~17:00): ADAS Level-1 uses exactly this — corner + edge + line detection, "no fancy deep-learning stuff" because cars can't mount a GPU. Corners feed parallax estimation (are we moving toward or away?), camera calibration, monocular/stereo SLAM, and AR overlay anchoring.

Research Enhanced

Shi-Tomasi vs Harris. The widely used Shi & Tomasi "Good Features to Track" (1994) replaces R = det − k·trace² with R = min(λ1, λ2) — no free parameter k, and empirically more stable corners for tracking. It's cv2.goodFeaturesToTrack and is the default front-end of the Lucas-Kanade optical-flow / VO pipelines still shipping in 2024.

Failure modes: Harris is rotation-invariant but NOT scale-invariant — a corner viewed from far away becomes a smooth blob and stops being detected. This exact limitation is what motivates SIFT's scale-space (Section 08). It is also sensitive to the k and threshold choices, as Prof showed live by tuning the threshold to reveal/hide corners (~1:01).

Worked Examples
At a pixel the eigenvalues of M are λ1 = 1200, λ2 = 1100. Using k = 0.04, compute R and classify the region.

  1. det(M) = λ1λ2 = 1200 × 1100 = 1,320,000.

  2. trace(M) = λ1 + λ2 = 2300. trace² = 5,290,000.

  3. R = det − k·trace² = 1,320,000 − 0.04 × 5,290,000 = 1,320,000 − 211,600 = 1,108,400.

  4. R ≫ 0 and both eigenvalues are large & comparable → CORNER.

AnswerR ≈ +1.11×10⁶ > 0 → corner. (Both λ large & similar = corner, per Prof's rule.)
Key Insight

You never actually need λ₁, λ₂ separately — det and trace give R directly from A, B, C. That's the whole computational point of Harris.

From Prof's live example: at the corner pixel Ix = 150 and Iy = 150 (so Ix² = 22500, Iy² = 22500, IxIy = 22500). With a single-pixel window, build M, then compute det, trace and R (k=0.04).

  1. M = [[A, B],[B, C]] = [[22500, 22500],[22500, 22500]].

  2. det(M) = A·C − B² = 22500·22500 − 22500² = 0.

  3. trace(M) = A + C = 45000. R = det − k·trace² = 0 − 0.04·(45000)² = −0.04·2.025×10⁹ = −8.1×10⁷.

  4. R < 0 → this single-pixel tensor reads as an EDGE, not a corner — because with Ix=Iy the tensor is rank-1 (det=0). A true corner needs the window sum over neighbours with different gradient directions so det>0.

Answerdet=0, trace=45000, R=−8.1×10⁷ (<0). A rank-1 single-pixel tensor → edge-like. Corners emerge only after the windowed Σ mixes multiple gradient directions (det becomes >0).

Explain — using the structure tensor M — why Harris is rotation-invariant but not scale-invariant. Tie it to what SIFT must add.

  2. Scale: the window is a fixed pixel size. Zoom out and a sharp corner spreads across fewer pixels than the window — the gradients average out, λ's shrink, R drops below threshold. The corner "disappears."

  3. Conversely a large smooth curve, viewed close up, can momentarily look corner-like inside a small window. The response is tied to the absolute pixel scale of the window.

  4. What SIFT adds: a scale space (Gaussian pyramid + DoG) so features are detected at the scale where they are extremal — making the detector scale-invariant, which Harris fundamentally is not.

AnswerRotation-invariant: R = f(det, trace) = f(λ₁λ₂, λ₁+λ₂), and eigenvalues don't change under rotation. NOT scale-invariant: a single fixed-size window can't track a feature across zoom — gradients wash out. SIFT fixes this with scale-space/DoG.

Source note

Intuition first — we never see objects, only reflected light

Slide 69 states it bluntly: "We do not see objects, but the light that has been reflected by or transmitted through the objects." Colour is therefore not a property of an object alone — it's the product of (1) the light source's spectral power distribution and (2) the object's reflectance (Slide 70). Change the lamp, change the colour.

Human colour vision is trichromatic (Slide 87): three cone types — S (short, ≈430 nm), M (medium, ≈560 nm), L (long, ≈610 nm) — sometimes called B, G, R cones. Every perceivable colour is just a 3-number combination of these cone responses. That biological fact is exactly why a 3-channel RGB image suffices to fool the eye, and why colour spaces are all 3-coordinate systems.

The reason there are many colour spaces (Slide 74): some match devices (RGB for screens, CMYK for printers, YCbCr for video), others match perception (HSI/HSV for human description — "blue for cold water," Slide 72). You pick the space whose coordinates align with your task.

Intuition first — un-blur by adding back the edges

Smoothing (Gaussian) removes high-frequency detail. Sharpening does the opposite: it amplifies high frequencies so edges and fine texture pop. The clean trick: a blurred image is "the original minus its detail," so detail = original − blurred. Add that detail back (scaled up) and the image looks crisper.

Two routes reach the same place. The Laplacian route uses the second derivative — it's zero on flat/ramps and spikes at edges, so subtracting it from the image accentuates edges. The unsharp masking route (a darkroom technique from film photography — print through a blurred "unsharp" negative mask) computes detail = f − blur(f) and adds it back. They're the same idea in two languages.

The analogy: turning up the "clarity" slider. You're not adding new information — you're exaggerating the contrast that already exists at boundaries so the eye reads them as sharper.

Sharpening — Laplacian, unsharp, high-boost
Laplacian (2nd derivative): ∇²f = ∂²f/∂x² + ∂²f/∂y² 4-neighbour kernel: [0 1 0 ; 1 −4 1 ; 0 1 0] 8-neighbour kernel: [1 1 1 ; 1 −8 1 ; 1 1 1] (includes diagonals) Sharpen: g(x,y) = f(x,y) + c·∇²f(x,y) c = −1 if centre is negative (−4, −8 kernels) c = +1 if centre is positive (+4, +8 kernels) Unsharp mask: g = f + k·(f − blur(f)) k = 1 → standard unsharp · k > 1 → high-boost filtering

Why the sign of c flips, and why k>1 = high-boost

The Laplacian's centre coefficient sets the sign. With the [1,−4,1] (negative-centre) kernel, edges produce a result of the opposite polarity, so you must subtract it (c = −1) to brighten the bright side of the edge. With a positive-centre [−1,4,−1] kernel you add it (c = +1). Get the sign wrong and the image looks embossed/inverted instead of sharpened.

For unsharp masking, k controls how much detail you re-inject. k = 1 restores the lost detail exactly (standard sharpening). k > 1 ("high-boost") adds more detail than was lost — it both sharpens AND keeps the original brightness, used when the image is dark and you want edges to pop without losing the base. Too large a k → over-sharpening halos.

Why This Matters — OpenCV three ways

                   [-1, 5,-1],
                   [ 0,-1, 0]])                 # = identity - Laplacian

DL connection: sharpening is a fixed high-pass conv; modern image-restoration / super-resolution nets (SRCNN→Real-ESRGAN, 2021) learn content-adaptive sharpening that avoids halos. Real use: medical X-ray/MRI enhancement, satellite imagery, every phone camera's "detail" pipeline.

Research Enhanced

Sharpening amplifies noise — and LoG fixes it. Because the Laplacian is a second derivative, it doubles down on noise (noise has high frequency too). The standard remedy is the Laplacian-of-Gaussian (LoG / "Mexican hat"): smooth with Gaussian first, then take the Laplacian — one combined kernel. This is the same "smooth-before-differentiate" lesson as Canny Step 1 and Harris Step 1.

Failure modes: ringing / halo artefacts (bright/dark fringes hugging strong edges) appear when k or c is too large — the classic over-sharpened look. Modern deblurring (2022–2024) uses learned priors to suppress these halos while still recovering detail.

Derive, in the frequency domain, why g = f + c·∇²f sharpens. Then explain why high-boost (k>1) preserves brightness where plain unsharp (k=1) can darken.

  1. The Laplacian is a high-pass operator: in the frequency domain ∇² ↔ −(u²+v²), which scales up high frequencies and zeros DC. So ∇²f contains only the image's high-frequency (edge) content.

  2. g = f − ∇²f (taking c=−1) = (all frequencies) + (high frequencies) → high frequencies are boosted relative to lows. Boosting highs = sharper edges. QED.

  3. Unsharp: g = f + k(f − blur). The mask (f − blur) is also pure high-pass (a low-pass blur subtracted from the original). k=1 adds one copy of detail.

  4. High-boost rewrite: g = (1+k)·f − k·blur. The (1+k) gain on f means the DC/brightness term is amplified, so dark images keep their base level while edges are boosted — whereas an aggressive pure high-pass (no f term) would suppress the low-frequency brightness and darken flat regions.

Answer∇²f ↔ −(u²+v²) is high-pass; adding it to f re-weights high frequencies up ⇒ sharper. High-boost g=(1+k)f − k·blur keeps a (1+k) gain on the full image, preserving brightness; a pure high-pass (no f) would strip the DC term and darken flats.
Key Insight

"Sharpening = original + a scaled high-pass." Whether you phrase it as Laplacian or unsharp mask, you're always re-injecting the high-frequency residual. The k (or c) is just the gain on that residual.

RANSAC — Robust Model Fitting

Preview · not in CV5 deck
Preview note — research-sourced

RANSAC is not in the CV5 deck or transcript — it belongs to a later session (it follows SIFT in the standard CV pipeline). Included as research-sourced revision (Fischler & Bolles 1981 + OpenCV). It connects directly to Hough (Section 02): both are voting/consensus methods for fitting models amid outliers.

Intuition first — one bad point ruins least squares

Fit a line to points with least squares and a single wild outlier drags the whole line toward it — the squared error gives outliers enormous pull. RANSAC's idea is gloriously simple: don't trust all the data; sample a tiny random subset, fit a model, and see how many other points agree. Repeat many times; keep the model with the biggest agreement (consensus) set.

The analogy is voting / jury selection. You can't poll everyone honestly (outliers lie), so you pick a few at random, propose a model from them, and count how many of the rest "vote" for it (land within ε). A model built from inliers attracts a big crowd; a model built from outliers attracts almost no one. After enough random juries, the popular model wins.

Why minimum samples matter: you fit from the smallest set that defines the model — 2 points for a line, 4 point-pairs for a homography (a homography is a 3×3 matrix with 8 degrees of freedom; each point-pair gives 2 equations, so 4 pairs = 8 equations = solvable).

The RANSAC loop (homography example)

  1. Randomly select the minimal sample

    For a homography, pick 4 random point correspondences between the two images.

  2. Fit the model

    Solve for the 3×3 homography H (8 DOF) from those 4 pairs.

  3. Score by counting inliers

    For every other pair, measure reprojection error: inlier if ‖p′ − H·p‖ < ε (SSD/reprojection threshold).

  4. Keep the best consensus

    Remember the H with the most inliers so far.

  5. Refit on all inliers

    After N iterations, re-estimate H using every inlier of the best model (least squares now safe — no outliers left).

RANSAC — iteration count & sample size
Iterations needed for confidence p: N = log(1 − p) / log(1 − (1 − ε)s) p = desired confidence (e.g. 0.99) ε = outlier fraction s = sample size (min points) (1 − ε)s = P(all s sampled points are inliers, one trial) Minimum sample size s: line: s = 2 · affine: s = 3 · homography: s = 4 (8 DOF ÷ 2 eqns/pair) Inlier test: ‖ p′ − H·p ‖² < ε²

What a homography is — and when it applies

A homography H is a 3×3 projective transform (8 DOF, defined up to scale) mapping one plane to another: [x′ y′ w]ᵀ = H·[x y 1]ᵀ. It is exact only when the scene is planar (a wall, a floor, a poster) OR the camera undergoes pure rotation (no translation) — that's why panorama stitching works (you only rotate the phone). For a general 3-D scene with translation, a single homography is wrong and you need the fundamental/essential matrix instead.

The statistics behind it. One trial picks s points; chance all are inliers = (1−ε)s. Chance the trial is "bad" = 1−(1−ε)s. After N independent trials, chance all are bad = [1−(1−ε)s]N. Set that ≤ 1−p and solve → N = log(1−p)/log(1−(1−ε)s). Worked: p=0.99, s=4, ε=0.5 → (1−0.5)⁴=0.0625 → N = log(0.01)/log(0.9375) ≈ 72 iterations. Cheap.

Failure modes (NOT fixable by more iterations): (1) degenerate configurations — 4 collinear points can't define a homography no matter how often sampled. (2) non-planar scenes — a single homography is simply the wrong model. (3) outliers > 50% blows N up super-linearly. (4) The formula assumes ε is known; in practice it's estimated adaptively (MLESAC, PROSAC, MAGSAC++).

Derive N algebraically for general (p, ε, s), then identify which failure mode adding iterations does NOT solve.

  1. P(one sampled point is inlier) = 1−ε. P(all s independent points inliers) = (1−ε)ˢ.

  2. P(a given trial fails to be all-inlier) = 1 − (1−ε)ˢ. Over N independent trials, P(every trial fails) = [1−(1−ε)ˢ]ᴺ.

  3. We want P(at least one good trial) ≥ p ⟺ 1 − [1−(1−ε)ˢ]ᴺ ≥ p ⟺ [1−(1−ε)ˢ]ᴺ ≤ 1−p.

  4. Take logs (argument <1, so log negative — inequality flips): N ≥ log(1−p) / log(1−(1−ε)ˢ).

  5. Not solvable by more N: a degenerate / non-planar configuration. If the model itself is wrong for the data (e.g. fitting one homography to a non-planar scene), no number of iterations produces a correct fit — the inlier set is structurally bounded. More N only helps when the issue is finding a good sample, not when no good model exists.

AnswerN = ⌈log(1−p)/log(1−(1−ε)ˢ)⌉. Increasing N defeats outliers (sampling luck) but CANNOT fix degenerate samples or a wrong model class (non-planar scene). Those need a different model or a degeneracy check, not more iterations.
Key Insight

RANSAC's N formula answers "how long until I get lucky." It says nothing about whether the model you're fitting is even appropriate — a separate, more dangerous failure.

SIFT — Scale-Invariant Feature Transform

Preview · next lecture
Preview note — Prof defers this

SIFT is explicitly the next lecture's topic, in Prof's own words: "SIFT stands for Scale Invariant Feature Transform… we will be studying about it" (~15:30) and "Google uses something called SIFT… which we will study in our next lecture" (~1:53). It directly answers the scale-invariance gap left by Harris (Section 03). Content below is research-sourced (Lowe 1999/2004 + OpenCV) and given so your mid-sem syllabus is complete.

Intuition first — a fingerprint that survives zoom and rotation

Harris finds corners but dies under scale change (Section 03). SIFT's whole purpose is a feature that you can re-find in another photo even if it's bigger, rotated, or differently lit — and describe it with a number-vector ("descriptor") distinctive enough to match across images.

The analogy is fingerprinting a landmark. For each interesting spot SIFT records: (1) where it is, (2) at what scale it looks most distinctive, (3) its dominant orientation (so you can rotate it upright), and (4) a 128-number summary of the local gradient pattern (the fingerprint). Two photos of the same building produce near-identical fingerprints → match; everything else differs.

The cleverness is achieving invariance constructively: detect at the scale where the feature is extremal (scale invariance), rotate the patch to its dominant gradient (rotation invariance), and normalise the descriptor (illumination invariance). David Lowe's 1999/2004 papers made wide-baseline matching, panoramas and early visual search practical.

The 4-stage pipeline (Lowe 2004)

  1. Scale-space extrema detection

    Build a Gaussian pyramid (octaves × scales). Compute DoG = G(kσ) − G(σ) per level. A keypoint = a pixel that is a local max/min among its 26 neighbours (8 same scale + 9 above + 9 below) — extremal in space AND scale.

  2. Keypoint localisation

    Sub-pixel refine via Taylor expansion of DoG. Reject low-contrast points (|D| < 0.03). Reject edge responses using a Harris-like curvature ratio (principal-curvature test, ratio threshold r=10).

  3. Orientation assignment

    Build a 36-bin (10° each) gradient-orientation histogram in the keypoint's neighbourhood; the peak = dominant orientation. Rotating the descriptor to this → rotation invariance.

  4. Descriptor generation

    Take a 16×16 window around the keypoint → 4×4 sub-blocks, each an 8-bin orientation histogram → 4×4×8 = 128-D vector. Gaussian-weight magnitudes, L2-normalise, clip at 0.2, renormalise (illumination invariance).

SIFT — DoG, descriptor, ratio test
Scale space: L(x,y,σ) = G(x,y,σ) ∗ I(x,y) DoG (≈ scale-normalised LoG): D = L(x,y,kσ) − L(x,y,σ) relation: G(kσ) − G(σ) ≈ (k−1)·σ²·∇²G (DoG approximates σ²∇²G) Descriptor length: 4 × 4 grid × 8 orientation bins = 128 Lowe's ratio test (matching): keep match if d1 / d2 < 0.7 – 0.8 d1 = nearest-neighbour dist, d2 = 2nd-nearest. Ambiguous → reject. Rejection thresholds: low-contrast |D| < 0.03 · edge ratio r = 10

Why DoG instead of LoG, and why 128

DoG ≈ LoG, but cheaper. The Laplacian-of-Gaussian (σ²∇²G) is the ideal scale-normalised blob detector, but expensive. The heat equation gives ∂G/∂σ = σ∇²G, and a finite difference G(kσ)−G(σ) ≈ (k−1)σ²∇²G. So a subtraction of two Gaussians you already computed for the pyramid approximates the LoG for free — no explicit second-derivative kernel.

Why 128 dimensions? A trade-off. 4×4 spatial bins keep coarse geometry (where in the patch each gradient is), 8 orientation bins keep enough angular detail — 128 numbers is distinctive enough to disambiguate millions of keypoints, yet small enough to match fast. Fewer dims (e.g. 36) lose distinctiveness; more (e.g. 512) barely help matching but cost memory/speed.

Show why DoG is a valid approximation to the scale-normalised LoG, and state the scale-factor relationship.

  1. The Gaussian satisfies the heat/diffusion equation: ∂G/∂σ = σ∇²G (the scale derivative equals σ times the Laplacian).

  2. Approximate the left side by a finite difference across two nearby scales σ and kσ: ∂G/∂σ ≈ [G(kσ) − G(σ)] / (kσ − σ).

  3. Combine: G(kσ) − G(σ) ≈ (k−1)·σ·(∂G/∂σ)·... → G(kσ) − G(σ) ≈ (k−1)·σ²·∇²G.

  4. So the DoG image is (k−1) times the scale-normalised LoG (σ²∇²G). The constant (k−1) is the same at every pixel, so it doesn't shift the location of extrema — DoG extrema = LoG extrema. You get the expensive LoG blob detector for the price of one subtraction.

AnswerG(kσ)−G(σ) ≈ (k−1)σ²∇²G, from the heat equation ∂G/∂σ = σ∇²G. The constant (k−1) scale factor doesn't move extrema, so DoG and scale-normalised LoG peak at the same keypoints — DoG is a free, accurate LoG approximation.
Key Insight

DoG isn't a hack — it's the heat equation's finite-difference form. That's why it works so well: it's a principled approximation, not a coincidence.

Conceptual Deep Dives

Six cross-cutting ideas that connect the session's algorithms. These are the "why" questions that separate a distinction answer from a pass.

A · The Derivative Duality — 1st vs 2nd derivative

Edge detection lives between two derivatives. The first derivative (gradient) is large on an edge — it peaks where intensity changes fastest. The second derivative (Laplacian) is zero on flat AND on linear ramps, but crosses zero exactly at the edge centre. So: gradient finds "where is the slope steep" (a maximum to threshold), Laplacian finds "where does the slope stop increasing and start decreasing" (a zero-crossing to locate).

1D edge (ramp up): f rises from 0 → 255 f′ (gradient): a single bump — maximum at the edge → threshold it f″ (Laplacian): + then −, passing through zero at the edge centre → find zero-crossing

Why it matters here: Canny & Sobel use the first derivative (threshold the gradient max). Sharpening & Marr-Hildreth/LoG use the second derivative (zero-crossings). Prof connected the second derivative to corner detection (Harris uses second-order structure via Ix², Iy², IxIy). Zero-crossings give sub-pixel localisation but amplify noise twice as hard — hence always Gaussian-smooth first (LoG).

B · RANSAC — the statistics behind the magic

RANSAC works because finding one clean sample is exponentially easier than cleaning all the data. Probability a single random minimal sample is all-inliers = (1−ε)ˢ. The chance you've failed every time after N tries is [1−(1−ε)ˢ]ᴺ, which decays geometrically. Demand it drop below 1−p and solve:

N ≥ log(1−p) / log(1 − (1−ε)ˢ) Worked table (s = 4, p = 0.99): ε = 0.10 → (0.9)⁴=0.656 → N ≈ 5 ε = 0.30 → (0.7)⁴=0.240 → N ≈ 17 ε = 0.50 → (0.5)⁴=0.0625 → N ≈ 72 ε = 0.70 → (0.3)⁴=0.0081 → N ≈ 588 (blows up past 50%)

Limitation: the formula assumes ε is known. In practice you don't know the outlier fraction up front, so real implementations estimate it adaptively (recompute N as the best inlier count grows) — MLESAC, PROSAC, MAGSAC++.

D · HoG → CNN — the feature-learning revolution

This is Prof's central thesis across the whole lecture. Dalal & Triggs hand-designed in 2005 what AlexNet learned automatically in 2012. HoG's pipeline — oriented gradients → cell histograms → block normalisation → linear classifier — is structurally a shallow CNN with fixed first-layer filters. A trained CNN's conv-block-1 converges to oriented edge/gradient detectors (verifiable by feature visualisation), i.e. it relearns HoG. The difference: CNNs then learn blocks 2–4 (textures → patterns → parts) too, while HoG stops at one hand-tuned layer.

That's why Prof quotes the numbers: HoG classification ~36%, AlexNet/VGG ~40% in 2012 "built on top of this only," and modern nets >90%. The receptive field grows, the filters specialise, and orientation selectivity (the Schrödinger-cat edge neurons from Session 4) emerges by gradient descent — the same oriented-gradient idea, now learned end-to-end.

D is unrelated to any Canny step.Common mistakeThinking NMS removes noise — that's Step 1's job. NMS is purely about thinning/localisation.

With TL=40, TH=100, a pixel with gradient magnitude 70 that is NOT 8-connected to any strong pixel is classified as:

  • BWeak edge, kept

Lowe's ratio test (d₁/d₂ < 0.7) primarily rejects:

  • BAmbiguous matches where best and 2nd-best are nearly equidistant
  • CLow-contrast keypoints

Q8MediumSharpening
In unsharp masking g = f + k·(f − blur(f)), setting k > 1 instead of k = 1:
  • ABoosts detail beyond what was lost AND preserves base brightness (high-boost)
  • BBlurs the image more
  • CHas no effect
  • DConverts it to grayscale
k>1 is "high-boost": g=(1+k)f − k·blur, so the original's brightness is amplified (1+k gain) while extra detail is added — sharper without darkening. B is the opposite (more blur). C only k=0 does nothing. D is unrelated.Common mistakeThinking k=1 and k>1 are the same. k=1 just restores lost detail; k>1 over-emphasises it (and preserves brightness).
Q9MediumHarris · Transcript
In Harris, the corner response uses R = det(M) − k·trace(M)². For a corner we expect:
  • AR > 0 (both eigenvalues large)
  • BR < 0 (one eigenvalue large)
  • CR ≈ 0 (both eigenvalues small)
  • DR exactly = k
Corner ⇒ both λ large ⇒ det large, R>0. B (R<0) is an edge (one λ large, one small). C (R≈0) is a flat region (both λ small). D is meaningless — k is just the 0.04–0.06 sensitivity constant.Common mistakeFlipping the sign convention. Memorise: R>0 corner, R<0 edge, R≈0 flat.
Q10HardRANSAC
Which RANSAC failure is NOT solvable by increasing the iteration count N?
  • AHigh outlier fraction (60%)
  • BNeeding higher confidence p
  • CA non-planar scene fitted with a single homography
  • DBad luck in early sampling
A non-planar scene is the wrong model — no sample can produce a correct single homography, so more N can't help (a degeneracy/model-class failure). A, B, D are all "sampling luck" problems the N formula directly addresses — bigger N buys higher confidence even at 60% outliers.Common mistakeAssuming more iterations cure everything. Q11HardHoG · Dalal-Triggs
Why does HoG use unsigned orientation (0–180°) rather than signed (0–360°) for pedestrians?
  • AIt halves computation only, no accuracy reason
  • B360° is impossible to compute
  • CGradients are always positive
  • DContrast polarity flips at edges (dark-on-light vs light-on-dark) describe the same contour, so folding to 180° removes polarity noise
A body contour produces opposite gradient directions depending on whether clothing is darker/lighter than the background — 180° apart but the same shape. Folding to unsigned removes this contrast-polarity noise (Dalal & Triggs ablation). A understates it (there IS an accuracy gain). B/C are false — signed gradients are perfectly computable and can be negative.Common mistakeSaying "to save computation." The real reason is invariance to contrast polarity.
Q12HardHough · Slides 55/61
Why does the standard Hough transform use the polar form ρ = x·cosθ + y·sinθ instead of y = mx + b?
  • APolar form is faster to compute
  • BVertical lines need m=∞ in slope form, which the accumulator can't represent; (ρ,θ) bounds both parameters
  • CPolar form detects circles
  • DIt removes the need for an edge map
A vertical line has infinite slope; the (m,b) accumulator can't hold m=∞, so vertical edges (walls, poles) are undetectable. Polar (ρ,θ) bounds θ∈[0,180°) and ρ∈[−D,D] — no infinities. A is false (sinusoid voting is slightly costlier). C is HoughCircles, a separate variant. D is wrong — Hough always needs a Canny edge map first.Common mistakeCiting speed. The real reason is representational: m=∞ breaks the slope-intercept accumulator.
