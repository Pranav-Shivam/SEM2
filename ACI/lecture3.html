01

Uninformed search — the blind algorithms

blind

Only the problem definition is visible: initial state, goal, transitions, costs. No sense of which unexplored state looks promising.

The instructor's words. BFS is "cautious" — it finishes an entire level before going one step deeper. DFS is "aggressive" — it grabs one path and rides it to the end, backtracking only once stuck.

Two applications, worked small

Same two algorithms, but here's what they're actually for outside of pathfinding — the exam likes asking "which algorithm, and why" for problems like these.

BFS → checking bipartiteness

Colour each node the opposite colour of its parent as BFS visits level by level. If any edge ever connects two same-coloured nodes, the graph is not bipartite.

MEHAR (blue) → SUNANDA, DEV (orange, depth 1) → HARISH, JENITTA (blue, depth 2). Two colours, no clash — bipartite.

DFS → topological sort

Run DFS; the moment a node finishes (all its children done), push it onto a stack. Popping the stack gives a valid build/task order — every dependency before its dependent.

visited = set(); order = []
def dfs(node):
    visited.add(node)
    for child in children(node):
        if child not in visited: dfs(child)
    order.append(node)   # finishes last → goes in stack

# topological order = reverse(order)

Same idea behind npm install resolving package dependencies, or a build system ordering compile steps.

02

Uniform Cost Search

blind + cost

f(n) = g(n) only. BFS with its queue replaced by a priority queue sorted on accumulated cost — always expand the cheapest path so far.

The rule that decides this trace. The goal is tested on expansion, never on generation. A goal can be generated looking cheap and still get beaten by a better path discovered afterward — watch it happen below.
Romania map · Sibiu → Bucharest Step 0 of 5

Optimal path: Sibiu → Rimnicu Vilcea → Pitesti → Bucharest = 278. Fagaras generated Bucharest at 310 first — it got overwritten the instant Pitesti found 278. That overwrite is the entire reason UCS waits to test the goal until expansion.
UCS pseudocode
frontier = priority_queue()   # sorted by path cost
frontier.add(start, cost=0)
explored = set()

while frontier is not empty:
    node = frontier.pop_min()      # cheapest node
    if node == goal: return path   # tested on EXPANSION
    if node in explored: continue
    explored.add(node)
    for each child of node:
        frontier.add(child, cost=g(node) + edge_cost)

From blind to informed — what is h(n)?

informed

An estimate of remaining distance to the goal, supplied from outside the raw problem definition.

The city-guide analogy. Uninformed search tries every street. Informed search asks a local: "the station is roughly 3km east." Not exact — but it slashes the search space. Something is better than nothing.

05

Greedy Best-First Search

f(n) = h(n)

Trusts the heuristic alone. Cost paid so far is invisible to it. Fast — and easily betrayed.

Romania map · h(n) only Step 0 of 3

GBFS reaches Bucharest via Arad → Sibiu → Fagaras → Bucharest = 450km — not the optimal 418. It was lured by Fagaras's low h(n) and never reconsidered the real road cost.
GBFS pseudocode
f(n) = h(n)   # ONLY the heuristic, no actual cost

frontier = priority_queue sorted by h(n)
frontier.add(start)

while frontier is not empty:
    node = frontier.pop_min_h()   # smallest h(n)
    if node == goal: return path
    for each child of node:
        frontier.add(child, priority=h(child))

The exam-safe takeaway. Whichever h(3) you're handed, the pattern is identical: GBFS commits to whatever neighbour has the smallest h(n) and never reconsiders; A* weighs g(n)+h(n) together and finds 150 < 225. 06

A* Search

f(n) = g(n) + h(n)
f(n) = g(n) + h(n)g = actual cost from start  ·  h = estimated cost to goal  ·  g(start)=0  ·  h(goal)=0
Romania map · full trace Step 0 of 5

The trap. At one point Pitesti is generated with f=417. But Fagaras (f=415) was already sitting in the open list from an earlier step, and 415 is lower. A* scans the entire open list before picking — never only the newest arrivals, and never assumes the newest arrival wins just because it was generated last. Miss this and every A* trace in the exam goes wrong.
Optimal path: Arad → Sibiu → Rimnicu Vilcea → Pitesti → Bucharest = 418km.
A* pseudocode
open_list = priority_queue sorted by f(n)
open_list.add(start, f=h(start))

while open_list is not empty:
    node = open_list.pop_min_f()   # lowest f(n)
    if node == goal: return path
    for each successor n' of node:
        new_g = g(node) + cost(node, n')
        new_f = new_g + h(n')
        if n' not in open_list or new_f < f(n'):
            open_list.add(n', f=new_f, g=new_g)

What makes a heuristic "good"?

Three properties, and one rule for comparing two candidate heuristics against each other.

PropertyWhat it means
AdmissibleNever overestimates the true cost — required for optimality
ConsistentSatisfies the triangle inequality across every neighbouring pair — required for graph-search optimality, and avoids re-opening nodes
InformativeClose to the actual h*(n) — the closer, the fewer nodes A* needs to explore
h₁(n) ≤ h₂(n) ≤ h*(n) for all nif both are admissible and h₂ dominates h₁, A* with h₂ expands no more nodes than A* with h₁
Weak heuristic: h₁(n) = 0

f(n) = g(n) + 0 = g(n). This is exactly UCS — admissible (0 never overestimates) but gives zero guidance.

Stronger heuristic: h₂(n) = straight-line distance

Still admissible, but h₂(n) ≥ h₁(n) everywhere — strictly more informative. A* explores far fewer nodes to reach the same optimal answer.

The dominance rule. If h₁ and h₂ are both admissible and h₂(n) ≥ h₁(n) for every node, h₂ dominates h₁ — and dominance is good news, not a tie. A* using the dominating heuristic never expands more nodes than A* using the dominated one, for the same problem. When you have a choice of heuristics, pick the largest one that's still guaranteed admissible.

Admissibility & consistency — interactive checker

exercise
h(n) ≤ h*(n)admissible — never overestimates the true cost-to-goal
h(n) ≤ cost(n,n′) + h(n′)consistent — triangle inequality, h can't drop faster than the real step cost
Consistent ⟹ admissible, given h(goal)=0. Not every admissible heuristic is consistent. Most heuristics in this course — straight-line, Manhattan — satisfy both.
Consistent — worked example

h(n)=10, cost(n→n')=3, h(n')=8

10 ≤ 3+8 = 11 — consistent

Not consistent — worked example

h(n)=10, cost(n→n')=2, h(n')=5

10 > 2+5 = 7 — not consistent

Exercise 2 — separate exercise
Trace A* on this graph

Nodes: S, A, B, C, D, G. Start = S, goal = G.
Edges: S→A(1), S→G(10), A→B(2), A→C(1), B→D(5), C→D(3), C→G(4), D→G(2)
Heuristics: h(S)=5, h(A)=3, h(B)=4, h(C)=2, h(D)=6, h(G)=0
For each expansion: show the open list, the node chosen, its f(n)=g(n)+h(n), and the final path. Then check admissibility.

Hint: always compare ALL nodes in the open list, not just recently generated ones. Watch out for the direct S→G(10) edge — it looks tempting but isn't on the optimal path.
Step 1: Expand S (f=0+5=5). Generate A: g=1, h=3, f=4. Generate G: g=10, h=0, f=10. Open: [A 4, G 10]
Step 2: Expand A (f=4, lowest). Generate B: g=3, h=4, f=7. Generate C: g=2, h=2, f=4. Open: [C 4, B 7, G 10]
Step 3: Expand C (f=4, lowest). Generate D: g=5, h=6, f=11. Generate G via C: g=6, h=0, f=6 — replaces the f=10 entry found via S directly. Open: [G 6, B 7, D 11]
Step 4: Expand G (f=6, lowest). Goal reached. Path: S→A→C→G, cost 6.
Admissibility: h(S)=5 vs the true cheapest cost from S (6, via S-A-C-G). 5 ≤ 6 ✓. Spot-check the rest the same way — h(A)=3 vs true cost 5 (A-C-G) ✓, h(C)=2 vs true cost 4 (C-G) ✓. This heuristic is admissible.
