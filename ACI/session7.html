Adversarial Search & Minimax

In normal search the world is passive — you just find a path. Here, an opponent actively works against you, so you must choose a move while anticipating their best reply. The core algorithm is Minimax.

01 Adversarial search

Two or more agents with conflicting goals: one player's gain is another's loss. You can't only ask "what gets me closer to my goal?" — you must also ask "what will my opponent do to reduce my chances?"

AspectNormal searchAdversarial search
Agentsusually oneat least two
Goalreach a goal statewin / maximise utility
Environmentpassivestrategic, competitive
Next state depends onour actionour action and the opponent's
Exampleroute findingchess

Think of GPS navigation vs. chess. With GPS, the road doesn't try to dodge you — it just sits there. In chess, every time you move, a thinking rival moves back, specifically trying to ruin your plan. So you can't just plan "my best path" — you have to plan "my best path assuming the other guy plays his best counter every single time."

02 A game, formally

A game is defined by six components — an extension of the problem-solving formulation you already know from search:

Initial state S₀
The starting board.
PLAYER(s)
Whose turn it is.
ACTIONS(s)
Legal moves.
RESULT(s,a)
State after a move.
TERMINAL-TEST(s)
Is the game over?
UTILITY(s,p)
Final value to player p.
⚔ Two-player zero-sum

Two players with exactly opposite goals. MAX maximises the score, MIN minimises it. Typical utilities: Win = +1, Draw = 0, Loss = −1. If MAX gets +1, MIN effectively gets −1.

"Zero-sum" just means the scoreboard always adds to zero — there's one pizza, and whatever slice you don't get, your opponent does. There's no way for both of you to "win" — every point you gain is a point they lose. That's why we can use a single number line: positive favours you (MAX), negative favours them (MIN).

03 Game tree & the Minimax principle

A game tree enumerates move sequences; levels alternate between MAX and MIN. At MAX nodes choose the largest backed-up value; at MIN nodes choose the smallest. Values propagate up from the leaves.

💡 The key insight

MAX does not pick the branch containing the best leaf — because after MAX moves, MIN replies! MAX picks the branch whose worst-case outcome is best ("maximise the minimum"). Minimax = minimise the maximum possible loss.

Imagine choosing a restaurant for a date where your date secretly hates exactly one dish on whatever menu you pick. You don't ask "which restaurant has my favourite dish" — you ask "after my date crosses off the worst possible thing for me, which restaurant leaves me the best remaining option?" That's minimax: you're not optimizing for the best case, you're optimizing for the best worst case, because you don't get the last move.

04 The Minimax algorithm

Pure minimax searches the whole tree to the leaves — fine for tic-tac-toe, impossible for chess. So in practice we use a depth limit + a static evaluation function: look a few moves ahead, then estimate how good the resulting position is.

MINIMAX(s) =
  UTILITY(s)                  if TERMINAL-TEST(s)
  max_a MINIMAX(RESULT(s,a))  if PLAYER(s) = MAX
  min_a MINIMAX(RESULT(s,a))  if PLAYER(s) = MIN

Time O(b^m), Space O(bm) (depth-first), where b = branching factor, m = max depth.

The formula is just a recursive "ask the question one level down, then flip who's choosing." Read it like a function in code: if the game is over, just return the score. Otherwise, ask every possible next move what its value is (by calling this same function on it), and then — depending on whose turn it is — either keep the biggest answer (MAX's turn) or the smallest answer (MIN's turn).

The complexity is the painful part: if a tree has on average b choices per turn and goes m turns deep, you're examining b multiplied by itself m times — it explodes fast, which is exactly why real engines (and you, in Session 8) need pruning and cutoffs.

05 Static evaluation functions

A static evaluation function estimates the utility of a non-terminal position "as is", without searching further. A good one: gives higher values to MAX-favourable states, lower to MIN-favourable ones, is fast, and correlates with the real chance of winning.

  • Chess example: Eval = material advantage + king safety + mobility + board control = 5 + 2 + 1 + 2 = +10 (good for White/MAX).
  • Simple version: Score = (MAX pieces) − (MIN pieces). With 8 vs 5 ⟹ Score = +3. Positive = good for MAX, negative = good for MIN, 0 = balanced.

You can't always play a game out to the very end to know who's winning — chess has way too many possible games. So instead you take a "snapshot guess": like glancing at a half-finished Monopoly board and going "okay, they own three railroads and I'm low on cash, this looks bad for me" without simulating the rest of the game. A static eval function is just a formula that turns that gut feeling into a number.

06 Worked examples

NumericalTiny game tree▸

Root = MAX, two MIN children, leaves left = (4,7), right = (2,6).

1.Left MIN node: min(4, 7) = 4.
2.Right MIN node: min(2, 6) = 2.
3.Root MAX: max(4, 2) = 4.

Best move = left branch, guaranteed value 4. MAX takes the left branch because even after MIN's best reply it still gets 4.

TrapWhy can't MAX just pick the +5 leaf?▸

Leaves: left = (+3, +5), right = (−2, +4). Students often say "MAX should choose the branch with +5." But after MAX picks the left branch, MIN moves and will choose +3 (the worse value for MAX), not +5.

  • Left MIN = min(+3, +5) = +3
  • Right MIN = min(−2, +4) = −2
  • Root MAX = max(+3, −2) = +3 → left branch.
⚠ The trap

So the left branch is worth +3, not +5. Always evaluate the opponent's best reply.

This is the single most common mistake in exams. It's like picking a vending machine because it has your favourite candy bar in it — except your little brother gets to press the button, and he's not picking your favourite. You only get the value AFTER the opponent has had their say. Never grade a branch by its best leaf — grade it by what survives the opponent's reply.

07 Self-test

Q1. At a MIN node, the backed-up value is the…
MIN minimises MAX's score, so it backs up the smallest child value. MAX backs up the largest.
Q2. Minimax time complexity for branching b, depth m:
Minimax explores the whole tree: O(bᵐ) time, O(bm) space (depth-first).
Q3. Why use a static evaluation function?
Big game trees can't be fully searched; we cut off at a depth limit and estimate the position with a static eval.
Q4. The minimax principle says MAX should choose the branch whose…
Because MIN replies optimally, MAX maximises the minimum — "best worst-case".
