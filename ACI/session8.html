Alpha-Beta Pruning & Monte-Carlo Tree Search

Minimax wastes effort on branches that can't change the decision. Alpha-Beta pruning gives the identical answer while skipping those branches. MCTS takes a different route: it samples random play-outs to focus search where it matters — the idea behind modern Go engines.

01 Alpha-Beta pruning

Alpha-Beta is minimax + a heuristic that stops evaluating a move once it's proven worse than an already-examined option. Same output, far fewer nodes. Maintain two bounds through the search:

α (alpha)
The best (highest) value MAX can already guarantee so far. Lower bound on the result. Starts at −∞.
β (beta)
The best (lowest) value MIN can already guarantee so far. Upper bound. Starts at +∞.
✂ General pruning principle

At a node n, if a player already has a better option at n's parent (or higher up), then n will never be reached in optimal play — so the entire subtree under n can be pruned. The cut-off condition is β ≤ α.

α and β are basically two players' "asking prices" written on a sticky note. α is the best deal MAX has found so far ("I can already get at least 6 elsewhere"); β is the best deal MIN has found so far ("I can already limit my loss to at most 4 elsewhere"). The moment α catches up to or passes β, it means: "Whatever I find deeper in this branch can't change my mind — I already have something at least this good guaranteed elsewhere." So why keep looking? You prune.

02 The algorithm

  1. At the root, set α = −∞, β = +∞.
  2. Descend to the depth limit and read static values.
  3. For each value VAL, until all leaves are done or β ≤ α:
    • If player is MAX: if VAL > α, set α = VAL. Then if β ≤ α, prune the remaining siblings (a β-cutoff).
    • If player is MIN: if VAL < β, set β = VAL. Then if β ≤ α, prune the remaining siblings (an α-cutoff).

In plain steps: walk the tree exactly like minimax, but carry two running "records" downward — the best guarantee MAX has found (α) and the best guarantee MIN has found (β). Every time you finish a child, check: "does this break the deal for whoever owns the parent?" If yes — stop checking that parent's remaining children entirely. You already know the parent will never be chosen in that direction.

03 Move ordering matters

Alpha-Beta's savings depend entirely on the order children are examined. Examine the best moves first → more cut-offs.

  • Worst-case (bad ordering): O(b^m) — no better than minimax.
  • Best-case (perfect ordering): O(b^(m/2)) — effectively branching factor √b, so you can search twice as deep in the same time.
🧭 Intuition

If the first move you check is already great, the opponent's good replies establish tight α/β bounds early, and most of the remaining junk branches fail the β ≤ α test immediately.

Imagine house-hunting and the very first house you tour happens to be excellent. Now every later house gets compared against that great one and rejected in seconds — you don't need to tour every room. But if the first house you see is the worst one on your list, you have nothing to compare against, and you end up touring every single house fully. That's the difference between best-case and worst-case ordering: a strong early bound lets you reject the rest fast.

04 Monte-Carlo Tree Search (MCTS)

A heuristic search that uses random simulations to decide where to grow the tree, instead of a hand-built evaluation function. It only searches a few layers and simulates the rest — keeping evaluations cheap. Four phases per iteration:

1 · Selection
From the root, repeatedly pick the most promising child, balancing exploitation (high win-rate) and exploration (rarely tried).
2 · Expansion
At a node with untried moves, add one new child node.
3 · Simulation / rollout
Play to the end from the new node using random (or lightweight) moves; record win/loss/draw.
4 · Backpropagation
Send the result back up the visited path, updating each node's visit count & win score.

UCB / UCT (used in Selection)

UCB_i = (w_i / n_i)  +  c · sqrt( ln(N) / n_i )
         └─ exploit ─┘    └──── explore ────┘

Probability rises with a node's value (exploit) and falls with its visits (explore); every option is tried at least once.

Applies to combinatorial games: two-player, sequential, perfect information, finite well-defined moves (chess, checkers, Go, tic-tac-toe).

Why not naïve Monte-Carlo as an eval? A single simulation is a noisy 0/1 signal and running many per node is too slow (Go: ~25 evals/sec) — MCTS fixes this by reusing simulations to guide tree growth.

Forget the formula for a second. You're at a casino with 3 slot machines and limited coins. Early on, you try each machine a few times just to get a feel (explore). Once one machine seems clearly better, you start feeding it more coins (exploit) — but you still occasionally test the others, just in case you got unlucky/lucky early and your first impression was wrong. UCB is the formula that decides, each round, exactly which machine to try next — by adding a small "but I haven't tried this much" bonus to each machine's average payout. As you play more, that bonus shrinks, and the system trusts the averages more.

The 4 phases are just: (1) walk down picking the best-looking path so far, (2) try one brand-new move you haven't tried, (3) "fast-forward" the rest of the game randomly to see who wins, and (4) walk back up updating everyone's scorecard with that result.

05 Stochastic games (brief)

Games with chance (dice, shuffled cards) add chance nodes between MAX and MIN levels. We back up the expected value at chance nodes (= Σ probability × child value) — this generalises minimax to expectiminimax. Example: backgammon.

Normal minimax assumes both players always make the smartest possible move. But what if a dice roll decides part of the outcome? You can't "out-strategize" a die — you can only know the odds. So at a chance node, instead of picking max or min, you just average all outcomes weighted by how likely each one is — same way you'd compute the "expected payout" of a casino game from its odds table.

06 Worked examples

NumericalAlpha-Beta on leaves 6,5,8,10,2,1,9,12▸

Root MAX → two MIN nodes (B, C) → each has two MAX nodes → each two leaves. Left-to-right:

1.B's left MAX sees 6 then 5 → value 6. So B (MIN) gets β = 6.
2.B's right MAX (called with β = 6) sees 8: as a MAX node α becomes 8 ≥ β 6 → β-cutoff, prune leaf 10. Returns 8.
3.B (MIN) = min(6, 8) = 6. Root MAX sets α = 6.
4.C's left MAX (called with α = 6) sees 2 then 1 → value 2. So C (MIN) gets β = 2.
5.Now at C: α 6 ≥ β 2 → α-cutoff, prune C's entire right MAX subtree (leaves 9 and 12).
6.C (MIN) = 2. Root MAX = max(6, 2) = 6.

Result = 6 (same as minimax). Pruned leaves: 10, 9, 12. 3 of 8 leaves never evaluated.

ConceptMake a tree where Alpha-Beta cannot prune at all▸

Pruning fails when every node's best option is discovered last — i.e. the worst-ordered case. For a MAX-root tree, arrange leaves so values increase exactly when a higher value would be needed to trigger a cut-off (e.g. each MIN node's first child is its smallest, so β never gets tight enough early). Then α-β degenerates to full minimax, O(b^m), evaluating all leaves. This is the worst case; perfect (best-first) ordering gives the O(b^(m/2)) best case.

It's the "worst house first" version of the house-hunting analogy above — if every tour you do is slightly better than the last, you never get a strong-enough early bound to skip anything, and you end up touring every single house.

ConceptMCTS vs Alpha-Beta — when to use which▸
  • Alpha-Beta needs a good static evaluation function and a modest branching factor (chess). Exact within the depth limit.
  • MCTS needs no hand-crafted evaluation — it estimates value by random play-outs. Wins when branching is huge or a good eval is hard to design (Go). It's anytime: stop whenever and return the best-sampled move.
🎯 Exam tip

If a question says "branching factor is enormous" or "no good evaluation function exists" → that's your MCTS cue. If it gives you a hand-built eval formula and asks you to prune → that's Alpha-Beta.

07 Self-test

Q1. Compared with plain minimax, alpha-beta pruning gives:
Alpha-Beta returns identical results to minimax; it only skips branches that can't affect the decision.
Q2. The alpha-beta cut-off condition is:
When β ≤ α, the current player already has a better option elsewhere, so the rest of the subtree is pruned.
Q3. With perfect move ordering, alpha-beta's time complexity is:
Best-case α-β effectively halves the exponent — branching factor √b — letting you search twice as deep.
Q4. Which is NOT one of the four MCTS phases?
MCTS phases are Selection, Expansion, Simulation (rollout), Backpropagation. Pruning belongs to alpha-beta.
