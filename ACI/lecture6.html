Genetic Algorithms, Ant Colony Optimization & NAS

Three nature-inspired methods, one underlying idea: keep many candidate solutions instead of one, and improve them in parallel.

Module 2 · Ant Colony Optimization

Pheromone-guided search

Ants leave trails. Shorter paths get reinforced faster. The colony "discovers" good routes — without any single ant being smart.

StoryAnts explore randomly at first. A shorter path means a faster round trip — so that ant lays pheromone sooner, and more ants follow it, reinforcing it further. Unused trails fade. But if a longer-path ant gets lucky early, the colony can be misled.

General pseudocode

Procedure ACO
  Schedule Activities
    Initialization
    Construction
    Update Pheromone
    Daemon Actions {optional — local search, elitism}
  End Schedule Activities
End ACO

Symbol decoder

SymbolMeaning
Ntotal number of ants; N > 1
τ₀initial pheromone amount
τijpheromone deposited while traversing i→j
ηij1 / cost(i,j) — shorter edge = bigger η
αimportance coefficient of pheromone intensity
βimportance coefficient of route cost
ρevaporation coefficient; 0 < ρ < 1
visitkvisited-nodes table for ant k
Qconstant for pheromone trail importance
fkroute cost obtained by ant k
⚠
Don't confuse η (eta) with ρ (rho). η = 1/cost. ρ = evaporation. Different letters, different jobs.

The two core formulas

NTPij = ( τijα · ηijβ ) / Σh∉visitk ( τihα · ηihβ )
NTP = next transition probability. numerator = this edge's pull · denominator = sum over all unvisited next edges from i
τij(new) = (1 − ρ) · τij(old) + Δτijk
Δτijk = Q / fk  if ant k passes edge (i,j), else 0
✓
Every edge evaporates every step. Only the edge the ant actually walked gets the extra deposit — reinforcement (exploitation) vs. fading (exploration) are two separate mechanisms.
⚠
Stopping criterion: reaching a predetermined number of iterations. Watch for the "ant drop" problem — if the limit is reached before the destination, the ant's incomplete tour is simply dropped.

Module 3 · Neural Architecture Search

NEAT → DeepNEAT → CoDeepNEAT

Treat a neural network like an individual in a population — let GA evolve the architecture instead of a human hand-designing it.

StoryInstead of trial-and-error by a human, treat each architecture as an individual. Fitness = task performance. Selection, crossover, and mutation evolve better designs automatically.

Why decompose into modules + blueprints?

Evolving one entire deep network as a single structure is expensive and hard to optimize. CoDeepNEAT splits the search into two smaller, separately-evolving populations instead.

  • Modules — reusable subnetwork components (e.g. Conv→ReLU→Pool), each with its own topology + hyperparameters, evolved in their own subpopulation.
  • Blueprints — high-level structural graphs where each node references a module species (placeholder), not a concrete layer.
  • Smaller subnetworks (feature extractors) and their arrangement evolve separately, instead of evolving one giant CNN end to end — this is what makes deeper, more complex architectures tractable to search.
✓
Assembly, 3 steps: (1) select a blueprint, (2) for each node, randomly pick a module from the matching species, (3) replace nodes with the chosen modules and connect them as the blueprint specifies. The assembled network is then trained and evaluated.

CoDeepNEAT mechanics

  • Blueprint — macro-architecture graph; each node points to a module species (a placeholder).
  • Module — a small reusable sub-network (e.g. Conv→ReLU→Pool), evolved in its own subpopulation.
  • Phenotype construction — pick a blueprint → fill each node with a chosen module → connect per blueprint → train.
  • Shared fitness — the assembled network's score credits both the blueprint and every module used, so good building blocks spread.

Inside a forward pass — cat vs dog, layer by layer

Once CoDeepNEAT has evolved an architecture, here's what actually happens when an image goes through it.

StepWhat happens
1. Input → numbersA 32×32 cat image becomes a 32×32×3 = 3072-value tensor. Pixel (R,G,B) = (120,80,60) normalizes to (0.47, 0.31, 0.23). At this point the system has no concept of "cat" — only numbers.
2. Conv + ReLUA 3×3 filter detects edges (ear outlines, body silhouette). ReLU, f(x) = max(0,x), zeroes out negative activations, keeping only the strongest signals.
3. Conv + PoolA second convolution combines edges into textures (fur, eye regions). Max pooling (2×2) shrinks the map while keeping the strongest features — "is this feature present," not "exactly where."
4. Higher-level featuresPatterns combine into object parts: triangular ears, small nose, whisker lines — features specific to "cat."
5. Fully connectedAll features flatten into one vector, e.g. [0.2, 0.8, 0.1, 0.9, …], and the layer learns relationships like "whiskers + triangular ears → likely cat."
6. Softmax outputP(yi) = ezi / Σezj. Example output: Cat = 0.92, Dog = 0.08 → final prediction "Cat," 92% confidence.
7. Backprop (training only)If the true label was actually "Dog," loss = predicted − actual, and gradient descent updates the weights to reduce that error next time.
✓
Key insight: the network never "sees" a cat the way a human does — it climbs a hierarchy: edges → textures → parts → object. CoDeepNEAT's job is discovering which combination of modules builds that hierarchy best, automatically.

“Stuck at 60% after 50 generations” — model exam answer

  • Richer module pool — add residual / skip-connection and batch-norm modules so deeper nets train without vanishing gradients.
  • Train longer per evaluation — too few epochs makes fitness noisy and unreliable.
  • Increase population size / mutation rate — escape a converged local optimum in architecture-space.
  • Better data — augmentation or more labels; the bottleneck may be data, not architecture.
  • Tune the fitness metric — validation accuracy with regularization, to avoid overfitting masquerading as fitness.
⚠
Vague answers (“use a better algorithm”) get zero marks. Always tie each suggestion to a specific mechanism.
Module 4 · Game Playing (Adversarial Search)

Minimax & static evaluation

Two players, opposing goals. One side's gain is the other's loss — so every move has to anticipate the opponent's best reply.

StoryIn normal search, the only question is "what gets me closer to the goal?" In adversarial search, there's a second question: "what will my opponent do to stop me?" Chess, tic-tac-toe, Go — someone is actively working against you, not just standing still.

Normal search vs. adversarial search

AspectNormal searchAdversarial search
Number of agentsUsually oneAt least two
GoalReach a goal stateWin or maximize utility
EnvironmentPassiveStrategic and competitive
Next state depends onOur actionOur action and the opponent's
ExampleRoute findingChess

A game, formally — six parts

  • Initial state — the starting board or situation
  • Player(s) — whose turn it is at state s
  • Actions(s) — the legal moves from state s
  • Result(s, a) — the new state after action a
  • Terminal-test(s) — checks whether the game has ended
  • Utility(s, p) — the final value of outcome s for player p
✓
Two-player zero-sum convention: Win = +1, Draw = 0, Loss = −1. MAX wants the score as high as possible; MIN wants it as low as possible. If MAX gets +1, MIN effectively gets −1 — one's gain is exactly the other's loss.

The minimax principle

MAX does not simply pick the branch with the best-looking leaf. MAX picks the branch whose worst-case, guaranteed outcome is best — because MIN moves next, and MIN will always pick the worst outcome for MAX within whatever branch MAX commits to.

function MINIMAX-DECISION(state) returns an action
  return argmaxa∈ACTIONS(s) MIN-VALUE(RESULT(state, a))
function MAX-VALUE(state):
  if TERMINAL-TEST(state): return UTILITY(state)
  v ← −∞; for each a: v ← MAX(v, MIN-VALUE(RESULT(s,a))); return v
function MIN-VALUE(state):
  if TERMINAL-TEST(state): return UTILITY(state)
  v ← +∞; for each a: v ← MIN(v, MAX-VALUE(RESULT(s,a))); return v
✓
Backing up values: leaf values propagate upward — MIN nodes take the min of their children, MAX nodes take the max of theirs, alternating all the way to the root.

Worked example — small tree

MAX (root)
max(4, 2) = 4
MIN (left)
min(4, 7) = 4
MIN (right)
min(2, 6) = 2
✓
MAX picks the left branch — guaranteed value 4 — because even after MIN's best response, MAX still walks away with 4. The right branch only guarantees 2.

The single most common mistake

Tree: MAX → two MIN children → leaves +3, +5 (left) and −2, +4 (right).

⚠
Wrong reasoning: "MAX should pick the branch containing +5, since that's the best leaf." Why it's wrong: MAX doesn't get to pick a leaf directly — after MAX commits to the left branch, MIN gets a turn, and MIN will pick min(+3, +5) = +3, not +5. The left branch's real value is +3, not +5. Minimax means choosing the best move after accounting for the opponent's optimal reply, every single time.

Multiplayer games — worked example

With 3+ players, each leaf becomes a vector of utilities — one value per player (A, B, C) — instead of a single number. At their own turn, each player maximizes only their own component of the vector, ignoring the others' scores entirely.

A to move → chooses between two branches leading to B's turn
B to move → chooses between branches leading to C's turn
C to move → chooses the leaf vector that maximizes C's own value
Each player only cares about their own slot in the (A, B, C) vector — there's no single "winner," just everyone optimizing their own outcome along the path that actually gets played.
⚠
Don't try to find one "best" leaf for everyone — there isn't one. Work backward one player-turn at a time, and at each turn that player picks their own best component, just like MAX does in a 2-player tree.

Why not search the whole tree?

Small games (tic-tac-toe) can be fully searched to terminal states. Large games (chess) cannot — the tree is too enormous. The fix: a depth limit + a static evaluation function that estimates how good a non-terminal position is, without searching further.

PropertyWhy it matters
Higher value = better for MAXThe function must point the right direction
Lower value = better for MINSame scale, opposite sign convention
Fast to computeIt runs at every cut-off node — speed matters more than perfection
Correlates with true winning chancesA fast-but-wrong estimate is worse than a slow-but-right search

Designing an evaluation function

Eval(S) = w₁f₁(s) + w₂f₂(s) + ⋯ + wₙfₙ(s)
a weighted linear combination of board features, each scaled by an importance weight

Chess example: Eval = material advantage + king safety + mobility + board control. Suppose White (MAX) scores +5 material, +2 king safety, +1 mobility, +2 board control → Eval = 5+2+1+2 = +10, a strong position for MAX.

Simplest possible version: Score = MAX pieces − MIN pieces. If MAX has 8 pieces and MIN has 5, Score = 8−5 = +3 (positive = good for MAX, negative = good for MIN, zero = balanced). Crude, but useful when full search isn't possible.

Other evaluation domains

The same weighted-feature idea applies outside chess — anywhere a position isn't a clean win/loss yet.

DomainFeatures used
Tic-Tac-ToeMAX's share of potential winning lines vs. MIN's share, plus overall board value
N-QueensPer-queen conflict/attack counts feed directly into the same weighted-sum style evaluation
8-puzzle (N-Tile)Number of tiles out of place compared to the goal configuration — fewer out of place = better state
✓
Whatever the domain, the recipe is the same: pick a handful of measurable features, weight each by importance, sum them — fast to compute, and roughly tracks who's winning.

Exam method — minimax with static evaluation

  • Identify which level is MAX and which is MIN — alternates every ply down from the root.
  • If leaves are terminal states, use the actual utility (+1/0/−1). If leaves are cut off early, use the static evaluation formula given in the question.
  • Work bottom-up: MIN nodes take the min of their children, MAX nodes take the max — one level at a time.
  • State both the backed-up root value and which branch MAX should actually choose.
⚠
Watch for single-child branches at any level — their "min" or "max" is just that one value, nothing to compare. Don't skip writing the step out; it's an easy place to lose a mark by jumping ahead.

GA in one line
Random population → fitness → roulette wheel selection → crossover → mutation → new generation → stop condition.

Ant at node → compute transition probabilities → choose next node → update pheromone → repeat until tour is built.

Common mistakes to avoid

  • Saying GA always improves every generation — children can be worse
  • Thinking roulette wheel always picks highest fitness — it only raises probability
  • Forgetting 4-Queens max fitness = 6
  • Confusing η (1/cost) with ρ (evaporation rate)
  • Adding Δτ to every edge — only the used edge gets it
  • Claiming ACO guarantees the shortest path — it doesn't
  • Vague NAS improvement answers — always name the specific mechanism
🧠
Final recap: GA = evolution of solutions. ACO = pheromone-guided path building. NAS = GA designing neural nets. All heuristic — good answers expected, perfect global optimum not guaranteed.
