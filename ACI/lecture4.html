The big idea in one sentence

Instead of wandering blindly through every possible solution, we use heuristics — smart educated guesses — to focus only on paths that look promising. And when the path doesn't matter, we use local search to directly sculpt the best answer.

f(n) = g(n) + h(n) ← A* : actual cost + estimated remaining cost f(n) = h(n) ← GBFS : only look at estimated cost f(n) = objective() ← Local Search : score the state itself

Foundations

What is a Heuristic?

A heuristic is like a GPS estimate — it's not perfectly accurate, but it's close enough to point you in the right direction without exploring every possible road.

Definition (in plain language)

A heuristic is a practical rule-of-thumb or educated guess that estimates "how far am I from the goal?" It comes from the Greek word heuriskein — "to find" or "to discover."

h(n) = estimated cost from node n to the goal

The key insight: instead of exploring every path, we prioritise paths that seem most promising. A good heuristic makes AI dramatically faster.

Real-world examples
🗺️

GPS Navigation

Uses straight-line distance to estimate remaining route. Not exact, but guides you correctly.

♟️

Chess Engines

Evaluate board positions using piece values, control of centre — not by checking all futures.

🤖

Robots

Estimate Euclidean distance to target when planning movement paths.

🏥

Medical Diagnosis

Doctors guess likely diseases from symptoms — an expert heuristic built from experience.

Why uninformed search fails

15-Puzzle state space

1.3 trillion

possible states to explore blindly. BFS would take years.

With a good heuristic

~1,000

nodes actually explored. Orders of magnitude faster.

The trade-off: Perfect heuristic h(n)=h*(n) → explores only O(d) nodes (solution depth). Zero heuristic h(n)=0 → explores O(b^d) nodes (exponential). The better your heuristic, the less you explore.

Core concept

Admissibility

Think of it as the "optimism rule" — a heuristic must always be optimistic (or exact), never pessimistic. It can underestimate the cost but never overestimate.

The rule

🧠 Layman version: Your GPS estimate of "5 mins to destination" is admissible if the actual drive always takes 5 mins or more. It's optimistic — never pessimistic.

h(n) ≤ h*(n) for every single node n h(n) = your heuristic estimate h*(n) = the TRUE minimum cost from n to goal

If even one node violates this, A* might discard the optimal path permanently. You must fix it before deploying.

What goes wrong when it fails?

When h(n) > h*(n), A* thinks a path is more expensive than it really is — and permanently throws it away. It can't go back. The optimal answer gets lost forever.

Fix: Return to your domain expert. Correct any node where h(n) > h*(n) to a value at or below h*(n).

Core concept

Consistency

Also called Monotonicity. Stronger than admissibility. Think of it as: "your estimate must not make sudden suspicious jumps between neighbouring nodes."

The triangle inequality

🧠 Layman version: If I estimate 10 km to the destination, and the next town is 3 km away, then from that next town I should estimate at most 7 km (not 20 km). The estimate must shrink sensibly.

h(i) ≤ cost(i → j) + h(j) for every edge i→j Meaning: my estimate at i must not be more than "the actual step cost" + "my estimate at j"

Consistency implies admissibility (but not vice versa). A consistent heuristic means A* never needs to re-expand a node — it's always moving forward efficiently.

Heuristic design

What Makes a Good Heuristic?

A heuristic that always returns 0 is technically admissible — but completely useless. It turns A* into uniform cost search. Good heuristics need more than just "not overestimating."

The four core properties

1. Admissible

h(n) ≤ h*(n) always. Never overestimates. Without this, A* can't guarantee the optimal path.

2. Consistent

Triangle inequality on every edge. Prevents A* from re-expanding already-explored nodes.

3. Informative

Close to h*(n). The tighter your estimate, the fewer nodes A* has to explore. Informativeness = efficiency.

4. Fast to compute

A heuristic that costs as much as the search itself is pointless. Speed of evaluation matters.

What if you have multiple heuristics? (4 cases)
SituationWhat to doWhy
h₂ covers everything h₁ does, and moreChoose h₂h₂(n) ≥ h₁(n) for all n → h₂ dominates → fewer nodes expanded
Both unique, both admissible+consistentCompare branching factorsLower effective branching factor = more efficient search
Both pass, but different expert reliabilityWeighted combinationf(n) = g(n) + α·h₁(n) + β·h₂(n) where α,β = trust weights
Same meaning, different formulaEquivalent — pick simplerSame node ranking = same search behaviour. Prefer faster to compute.
Dominance rule
h₂ dominates h₁ ↔ h₂(n) ≥ h₁(n) for ALL n, both admissible Safe combination: h(n) = max(h₁(n), h₂(n), ...) → Always admissible, always dominates each individually

Taking the maximum of multiple admissible heuristics is always safe and always better than using any single one alone.

Heuristic quality metric

Effective Branching Factor

A way to measure how "focused" your heuristic is. The lower b* is, the less your algorithm wanders around exploring useless paths.

The formula

🧠 Layman version: If A* explores N nodes to find a solution at depth d, then b* is the branching factor a perfect uniform tree of depth d would need to have exactly N+1 nodes.

N + 1 = 1 + b* + (b*)² + (b*)³ + ... + (b*)^d N = number of nodes generated d = depth of the solution b* = effective branching factor (solve for this)

A good heuristic keeps b* close to 1. A bad heuristic (h=0) means b* = actual branching factor b.

What b* tells you

b* ≈ 1.0

Nearly perfect heuristic. Explores almost only the optimal path. Very fast.

b* ≈ 1.5

Good heuristic. A* on 8-puzzle with Manhattan distance achieves around this.

b* ≈ b

Heuristic provides no useful information. Same as uninformed search.

Worked example

N-Queens Problem

Place N queens on an N×N board so no two attack each other. Queens attack along the same row, column, or diagonal. Click any cell to move a queen.

Interactive 4-Queens board — click a cell to move that column's queen
Three heuristics explained

H1: Non-conflicting pairs

Count queen pairs NOT attacking each other. C(4,2)=6 possible pairs total.

Goal = 6 (max) · Maximise

H2: Conflicting pairs

Count pairs that ARE attacking (same row, col, or diagonal). Want zero.

Goal = 0 (min) · Minimise

H3: Safe queens

Count queens not attacked by any other queen.

Goal = 4 (max) · Maximise

H1 and H3 are equivalent in meaning. If all 6 pairs are non-conflicting (H1=6) then all 4 queens are safe (H3=4). Using both gives no new information — pick the simpler one (H3).

Relaxed Problems

🧠 Layman version: If solving the puzzle with some rules removed is easy, use the cost of that easy version as your heuristic for the hard version. It's always admissible — and you get it for free.

The formal proof (in plain language)
Original Problem P (strict rules) Relaxed Problem P' (some rules removed) Key logic: Any solution to P works in P' too (P' has fewer restrictions) → optimal_cost(P') ≤ optimal_cost(P) → h(n) = optimal_cost(P') ≤ h*(n) = optimal_cost(P) → h(n) is ADMISSIBLE by definition ✓ Bonus: State space of relaxed problem is a SUPER-GRAPH of the original — more states allowed.
8-Puzzle — the three relaxations, step by step

Original: tile only moves into adjacent EMPTY cell

Most restricted. Hardest to solve. 9! states.

Relax 1: Remove "must be empty" → tile moves to any adjacent cell

A tile can now slide into an occupied cell too.

→ Manhattan distance (H2)

Relax 2: Also allow diagonal moves

Tile can move in any direction including diagonals.

→ Chebyshev distance

Relax 3: Remove ALL movement restrictions → tile teleports

Each wrongly-placed tile can jump to its goal in one step.

→ Misplaced tiles (H1) — weakest but fastest
Pattern databases

🧠 Layman version: Solve every possible arrangement of just 4 tiles (ignoring the others). Save the answer in a lookup table. When A* is searching, just look up the table instead of computing anything.

1
Pick a sub-problem — e.g. only tiles 1–4 of the 8-puzzle
2
Pre-compute ALL configurations of that sub-problem. Offline, once.
3
Store in a hash table keyed by sub-problem state
4
During A* search — just do a table lookup, no computation
5
Always admissible because sub-problem cost ≤ full problem cost
Landmarks

🧠 Layman version: GPS doesn't calculate every street corner to every other street corner. It picks major cities as "landmarks," pre-computes distances to them, then estimates your route via the nearest landmark.

Rubik's cube masters do the same: from any scrambled state, manoeuvre to a memorised "landmark" configuration, then apply the memorised solve sequence from that landmark to finished.

Advanced topic

Learning Heuristics

What if you can't write the heuristic yourself? Let the AI learn it. Three approaches — and one famous reference paper from the lecture.

When manual design fails

No domain expert

Problem is too novel — no SME can give you reliable h(n) values.

State space too large

Even relaxed-problem heuristics are too loose to be useful.

Complex constraints

Rules intertwine in ways too complex to express as a clean relaxation.

Two learning approaches

Metalevel learning

The AI "thinks about its own thinking." It studies which paths it took and learns to avoid dead-ends and unhelpful patterns in the future.

Feature learning

Solve thousands of random instances. Notice patterns — e.g. "tiles out-of-order" correlates with cost. Fit a formula (or neural net) to those features.

Caveat: Learned heuristics are usually NOT guaranteed admissible. You trade optimality guarantee for better practical performance on large problems.

Reference paper — Ferber, Helmert & Hoffmann (2020)

Learning Heuristic Functions for Large State Spaces

Patrick Ferber, Malte Helmert, Jörg Hoffmann · arXiv:2107.12945 · 2020

What they did: Trained CNNs on Sokoban, maze-with-teleports, and sliding tile puzzles to output a heuristic value directly from the grid state.

Network architecture
Grid state s → 7 Conv layers → 4 CoAt blocks → Mean pool → FC → h(s,θ)

CoAt = Convolution + Attention. Captures both local spatial patterns and global context.

A* formula with learned heuristic: f(n) = g(n) + h(s, θ) where θ = trained network weights

Result: Learned heuristics outperformed manually designed ones on large state spaces. GBFS with learned h was often faster (though not always optimal).

Local search

What is Local Search?

🧠 Layman version: Instead of finding a path from A to B, you start somewhere random and keep nudging the current solution until it can't get any better. The final state IS the answer — the path doesn't matter.

Path search vs Local search

Traditional search (A*, BFS...)

The path from start to goal IS the solution. You need to remember how you got there. Like following a recipe step-by-step.

Local search

Only the final state matters. Like sculpting — you keep reshaping until it looks right, and no one cares what cuts you made.

Classic example: 8-Queens problem. We don't care how we rearranged the queens — we only care that the final board has no conflicts.

Advantages of local search

Very low memory

Only stores the current state (and maybe a few neighbours). No open/closed lists, no path tracking.

Works on infinite spaces

Can handle continuous or infinite state spaces where systematic search is impossible.

Local search algorithm

Hill Climbing

🧠 Layman version: Stand somewhere on a hilly landscape. Always step to the highest neighbouring spot. Stop when all neighbours are lower. Simple — but you might get stuck on a small hill, never finding the mountain.

Algorithm (plain language + pseudocode)

Step 1: Pick a random starting state.
Step 2: Look at all neighbours. If any is better → move there.
Step 3: If no neighbour is better → stop. You're at a local maximum.

function HILL-CLIMBING(problem): current ← problem.INITIAL_STATE while true: neighbor ← best successor of current if VALUE(neighbor) ≤ VALUE(current): return current ← local maximum found current ← neighbor ← keep climbing
Interactive 4-Queens Hill Climbing

Using h(n) = number of non-conflicting pairs (maximise). Start state: [1,4,2,2].

Hill Climbing with Random Restart

When Hill Climbing gets stuck at a local maximum, restart from a new random position. Run many times and keep the best result found across all restarts.

Stop condition 1

No neighbour has a better fitness value → local optima reached

Stop condition 2

Reached the global maximum or minimum (h=0 for N-Queens conflicts)

Stop condition 3

Reached the predefined number of restarts k

How many restarts? For 8-Queens, random restart usually finds a solution in just a few attempts. For harder problems, k may need to be large.

Problems with Hill Climbing

Local maxima

Gets stuck on a small peak. Thinks it found the best when a better peak exists elsewhere.

Ridges

A sequence of local maxima that's hard to traverse — each step makes things worse.

Plateaus

A flat region where all neighbours have equal value. No gradient to follow.

Key landscape features

Global maximum

The best possible solution in the entire state space. What we're looking for.

Local maximum

Better than immediate neighbours, but not the global best. Hill Climbing gets trapped here.

Plateau

A flat region — all neighbours have equal fitness. No clear direction to move. Algorithm stalls.

Local search variant

Stochastic Hill Climbing

🧠 Layman version: Sometimes deliberately take a worse move — with a probability that depends on how much worse it is and a "temperature" that cools over time. This helps escape local maxima.

The acceptance rule
next ← randomly selected successor of current ΔE ← next.VALUE − current.VALUE if ΔE > 0: → always move (it's better) current ← next else: → sometimes move even if worse current ← next with probability e^(ΔE / T) Where T = temperature parameter (controls risk tolerance)

When ΔE is very negative (much worse) → probability is near 0, rarely accepted. When ΔE is mildly negative and T is high → probability is reasonable, may accept. As T → 0, behaviour approaches standard Hill Climbing.

Stochastic HC steps (from lecture example)
1
Select a random state — e.g. [1, 4, 2, 2]
2
Evaluate all successors — compute h(n) for all 12 neighbours
3
Calculate probabilities — based on fitness scores (proportional selection)
4
Select next state probabilistically — higher fitness = higher chance, but not guaranteed
5
Repeat from step 2 — until termination condition met

The lecture example: From N={4,2,2,3,3,2,1,3,2,1,3,2}, probabilities were: P(4)=1/12, P(3)=4/12, P(2)=5/12, P(1)=2/12. State B with h=2 had highest probability 5/12 = 0.42.

Exercise 1 — GBFS and A* on Weighted Graph

Start = B (h=19), Goal = G (h=0). Numbers on edges are costs. Numbers beside nodes are heuristic values.

B(h=19) —5→ C(h=13) —6→ D(h=6) —3→ G(h=0)
B(h=19) —7→ F(h=4) —6→ H(h=3) —3→ G(h=0)
B(h=19) —10→ E(h=10) —6→ F(h=4)
C(h=13) —4→ F(h=4)
A(h=12) —6→ B(h=19) & A —6→ D(h=6)
D(h=6) —4→ H(h=3)
(a) Apply GBFS
Use f(n) = h(n) only. Always expand the node with smallest h. Trace the path.
(b) Apply A*
Use f(n) = g(n) + h(n). Track actual cost g(n) at each node. Compare with GBFS result.
(c) Compare
Which found the optimal path? Which was faster (fewer nodes)? Is GBFS path always optimal?
(d) Verify admissibility & consistency
For each node, check h(n) ≤ h*(n). For each edge, check h(i) ≤ c(i,j) + h(j).
Hint: For GBFS trace: B(19) → C(13)? F(4)? E(10)? Expand lowest h first. For A*: track cumulative g(n) at each node. The path B→F→H→G costs 7+6+3=16; B→C→D→G costs 5+6+3=14.
