Heuristic Design, Admissibility & Relaxed Problems

Session 4 of 16 — 25% through the course. This session formalises what makes a heuristic good, gives tools to generate admissible ones automatically, and introduces learning-based heuristics.

Core principle

The closer your heuristic values are to the actual costs, the better the heuristic. Admissibility and Consistency are the mathematical formalisation of this principle.

h(n) ≤ h*(n) for all n → A* finds the optimal path

Core concept

Admissibility

The minimum requirement for A* optimality in tree search. A heuristic must never overestimate the true cost to reach the goal.

Definition

A heuristic h(n) is admissible if for every node n:

h(n) ≤ h*(n) where h*(n) = the TRUE minimum cost from n to the goal

Even a single node where h(n) > h*(n) breaks admissibility for the entire graph. The procedure: enumerate all paths from each node to the goal, take the minimum cost, then compare with h(n).

What happens when admissibility fails?

When h(n) > h*(n), A* sees that path as inflated and permanently discards it. Once discarded, A* cannot recover — the optimal solution is lost. Admissibility is the MINIMUM requirement for A* optimality in tree search.

Fix: Return to the domain expert. Correct any node where h(n) > h*(n) to any value ≤ h*(n).

Core concept

Consistency

Also called Monotonicity. Stronger than admissibility — ensures f(n) = g(n)+h(n) is non-decreasing along any path, preventing A* from re-expanding nodes in graph search.

Definition — triangle inequality

For every node i and its successor j:

h(i) ≤ cost(i → j) + h(j)

Consistency implies admissibility, but admissibility does NOT imply consistency. Consistency prevents "sudden jumps" in heuristic values — A* with a consistent heuristic never needs to re-expand a closed node.

What makes a good heuristic?

Heuristic Quality & Dominance

Four core properties, how to choose between multiple heuristics, and the formal dominance rule. A heuristic that always returns 0 is admissible but useless — it turns A* into UCS.

The four core properties

Admissible

h(n) ≤ h*(n) for every node. Never overestimates the true cost to goal. The minimum requirement for A* optimality.

Consistent

h(i) ≤ cost(i→j) + h(j) for every edge. Stronger than admissibility. Prevents re-expansion of nodes in graph search.

Informative

Close to the true cost h*(n). A heuristic closer to h* expands fewer nodes. Dominates weaker alternatives.

Fast to compute

A heuristic that costs as much to compute as the search itself is useless. The trade-off between quality and speed matters.

When you have multiple heuristics — 4 situations

Domain experts often provide different heuristics. Here's how to choose:

SituationDecisionReasoning
h₂ is a superset of h₁ — covers everything h₁ covers plus moreChoose h₂More information = more guidance. h₂ dominates h₁ if h₂(n) ≥ h₁(n) for all n.
Both unique, both admissible + consistentCompare branch factorsWhichever produces a smaller effective branching factor (expands fewer nodes) is more efficient.
Both unique, both pass, but different expert reliabilityWeighted combinationf(n) = g(n) + α·h₁(n) + β·h₂(n) where α,β reflect expert reliability/trust.
Both give same meaning but different formulationThey are equivalentCheck if they produce the same ranking of nodes. If so, choose the simpler one to compute.
Dominance — which heuristic is stronger?

h₂ dominates h₁ if h₂(n) ≥ h₁(n) for all n, and both are admissible.

h₂(n) ≥ h₁(n) for ALL n, both admissible → h₂ dominates h₁

A* with h₂ will always expand fewer or equal nodes compared to A* with h₁. Taking the max of multiple admissible heuristics is always safe: h(n) = max(h₁, h₂, …) is admissible and dominates each individually.

The simple rule: the closer your heuristic values are to h*(n), the better. A heuristic that always returns 0 is admissible but completely useless — it degrades A* into Uniform Cost Search.

Heuristic design

N-Queens Problem

Place N queens on an N×N board so no two attack each other. Queens attack in the same row, column, or diagonal. Click any cell to move a queen in that column.

Interactive board — 4-Queens (click to move queens)
Three heuristics compared

H1: Non-conflicting pairs

Count queen pairs NOT attacking each other. C(4,2)=6 possible pairs total.

Goal = 6 (max) · Maximise

H2: Conflicting pairs

Count queen pairs that ARE attacking each other (same row, column, or diagonal).

Goal = 0 (min) · Minimise

H3: Safe queens

Count queens not being attacked by any other queen. Simpler per-queen check.

Goal = 4 (max) · Maximise

H1 and H3 are equivalent in meaning. Both answer "how far from goal?" in the same direction. If all 6 pairs are non-conflicting (H1=6), all 4 queens are safe (H3=4). Using both gives no extra information — prefer the simpler one (H3: count safe queens individually).

Automatic heuristic generation

Relaxed Problems

When no domain expert is available — or the problem is too novel — remove constraints from the original problem. The relaxed solution cost is always ≤ the true cost: admissibility guaranteed by construction.

Formal proof of admissibility

Let P = original problem (all constraints). Let P' = relaxed problem (fewer constraints).

Any solution to P is also a solution to P' (P has MORE constraints, so P' accepts everything P does) → optimal_cost(P') ≤ optimal_cost(P) → h(n) = optimal_cost(P') ≤ h*(n) = optimal_cost(P) → h(n) is ADMISSIBLE by definition ✓
8-Puzzle: three relaxations

Original: Tile moves to adjacent empty cell only

Fully constrained. Hard to solve optimally. State space: 9! = 362,880 states.

Relax 1: Remove "empty" requirement → tile moves to any adjacent cell

Each tile can slide regardless of whether the adjacent cell is occupied.

→ Manhattan distance (H2) · Admissible ✓

Relax 2: Also allow diagonal moves

Tile can now move in any direction including diagonals.

→ Chebyshev distance · Admissible ✓

Relax 3: Remove ALL movement restrictions → tile teleports

A tile can move to any position in one step. Maximally relaxed.

→ Misplaced tiles (H1) · Admissible ✓ — weakest but fastest
Other heuristic sources

Pattern databases

Pre-compute exact costs for sub-problems (e.g. tiles 1–4 only). Store in a hash table. During A* search, look up the cached value. Always admissible: solving the sub-problem ≤ solving the full problem.

Landmarks

Pick key "landmark" nodes and pre-compute distances to them. From any state, navigate via the nearest landmark. Rubik's cube masters use this — from any scrambled config, manoeuvre to a known landmark, then apply memorised moves to the solved state. The instructor's visually impaired friend who holds the Asia Book of Records uses the same technique with braille-based landmarks.

Learning heuristics

Neural Networks as h(n)

When manual heuristic design fails, train a neural network on thousands of solved problem instances and use its output as h(n). The A* formula becomes f(n) = g(n) + h(s,θ).

Three scenarios where manual design fails
1
No domain expert available — the problem is too novel or specialised for any SME to provide reliable h(n) values.
2
State space too large — even relaxed-problem heuristics become too loose to be useful in enormous spaces (e.g. Sokoban, sliding-tile variants).
3
Complex implicit constraints — domain constraints are too intertwined to express as a clean mathematical relaxation.
Two approaches to learning heuristics

Metalevel learning

The agent "thinks about its own thinking." It learns to avoid dead ends and recognise promising states from past searches — improving the search strategy itself, not just the estimate.

Feature learning

Solve thousands of instances. Identify features that predict cost (e.g. # out-of-place tiles). Fit a formula or neural network to those features. The result: h(s,θ) where θ are the learned parameters.

Caveat: Learned heuristics are usually NOT guaranteed admissible — optimality can be lost. But they are often far more informative and faster in practice on large state spaces.

Reference paper (shown by instructor in lecture)

Learning Heuristic Functions for Large State Spaces

Patrick Ferber, Malte Helmert, Jörg Hoffmann · 2020 · arXiv:2107.12945

Uses convolutional neural networks to learn heuristic functions for grid-world problems including Sokoban, maze-with-teleports, and sliding tile puzzles. Key formula: f(n) = g(n) + h(s,θ) where θ are the trained network parameters. Results show learned heuristics outperform manually designed ones in large state spaces.

arxiv.org/abs/2107.12945
Pattern databases — learning by pre-computation

Pattern databases are a form of "offline learning" — the system solves many sub-problems before the main search begins and caches the results.

1
Identify a sub-problem — e.g. only tiles 1–4 in the 8-puzzle, ignoring tiles 5–8 and the blank.
2
Pre-compute optimal cost for EVERY configuration of those 4 tiles. Exhaustive but offline.
3
Store in a hash table / database — keyed by the sub-problem state.
4
During A* search — look up the current sub-problem configuration to get its cost instantly.
5
Use as h(n) — always admissible because solving the sub-problem ≤ solving the full problem.
Heuristic design summary

The Heuristic Evaluation Cycle

What to tell a domain expert, how to validate the heuristic, and what to do when it fails. This is the complete deployment workflow from the lecture.

What to give the domain expert (SME)

Provide the complete problem specification:

·
PEAS description (Performance, Environment, Actuators, Sensors)
·
Environment type (6-dimension classification from Session 2)
·
All 5 PSA components (Initial State, Actions, Transition, Goal Test, Path Cost)
·
A sample problem instance with known graph structure for validation
The 8-step evaluation cycle
1
Give SME the full problem spec (PEAS + environment + PSA components)
2
SME provides h(n) values for all nodes in sample problem
3
Check admissibility: h(n) ≤ h*(n) for every node
4
Check consistency: h(i) ≤ cost(i→j) + h(j) for every edge
5
If BOTH pass → deploy the heuristic
6
If EITHER fails → return to SME with the specific failed nodes/edges
7
SME corrects the values → repeat from step 3
8
No SME available → use relaxed problems or learning heuristics

The heuristic design process is iterative, not one-shot. Build validation of admissibility and consistency into your workflow before deployment.

Heuristic design process — 5 stages
1
Problem Spec — Provide PEAS, environment type, PSA 5-components to domain expert
2
Get Heuristic — Domain expert or SME provides h(n) values for all nodes
3
Validate Admissible — Check h(n) ≤ h*(n) for every node in sample problem
4
Validate Consistent — Check h(i) ≤ c(i,j)+h(j) for every edge in sample problem
5
Deploy / Refine — If both pass → deploy. If either fails → go back to SME

Exercise 1 — Admissibility & Consistency Check
Mid-sem exam

A graph has nodes {A,B,C,D,E} where E is the goal.

Edges: A→B(3), A→C(6), B→D(4), C→D(2), B→E(8), D→E(3)

Heuristic values: h(A)=7, h(B)=5, h(C)=4, h(D)=2, h(E)=0

(a)
For each node, find h*(n) — the minimum actual cost to goal E.
(b)
Check admissibility for each node: is h(n) ≤ h*(n)?
(c)
Check consistency for every edge: is h(i) ≤ cost(i→j) + h(j)?
(d)
Is A* guaranteed to find the optimal path? Justify your answer.
Hint: For h*(n): enumerate all paths from each node to E, take the minimum cost. For consistency: apply h(i) ≤ cost(i→j) + h(j) for every directed edge. Remember: consistency implies admissibility but not vice versa.
Exercise 2 — 8-Puzzle Heuristic Design
Mid-sem exam

Current state: [1,2,3 / 4,_,5 / 7,8,6] where _ is blank

Goal state: [1,2,3 / 4,5,6 / 7,8,_]

(a)
Calculate H1 (misplaced tiles) — how many tiles are in the wrong position?
(b)
Calculate H2 (Manhattan distance) — compute the Manhattan distance for each tile individually.
(c)
Calculate H3 (empty tile Manhattan) — where is the blank, where should it go?
(d)
Which heuristic dominates? Is H2 ≥ H1 for this configuration?
(e)
Which heuristic would A* expand fewer nodes with? Why?
Hint: For Manhattan distance, count |row_current − row_goal| + |col_current − col_goal| for each tile independently, ignoring all other tiles.
Exercise 3 — Heuristic Design for a New Problem: Delivery Robot
Mid-sem exam

A robot must deliver packages to 5 offices (A,B,C,D,E) starting from a depot (D0). It can carry one package at a time. The goal is all packages delivered.

(a)
Write the PSA 5-component specification (Initial State, Actions, Transition, Goal Test, Path Cost).
(b)
Propose THREE different heuristics for this problem.
(c)
Argue why each heuristic is admissible.
(d)
Which heuristic would you use in A* and why?
(e)

Hint: Think about what information is available at each state: which packages are delivered, which are not, current robot location, distances between locations. A relaxed version might ignore the one-at-a-time constraint.
Exercise 4 — N-Queens Extended
Mid-sem exam

4-Queens problem. Queens are at row-column positions: Q1=(0,0), Q2=(1,2), Q3=(2,0), Q4=(3,2).

(a)
How many conflicting pairs are there? List them all.
(b)
Compute H1 (non-conflicting pairs), H2 (conflicting pairs), H3 (safe queens) for this configuration.
(c)
Is this state promising (close to goal) or not?
(d)
Propose one move (change position of one queen) that improves all three heuristics.
(e)
Design a fourth heuristic for N-Queens not mentioned in class.
Hint: Two queens conflict if they share a row, column, or diagonal. For diagonals: |row1−row2| == |col1−col2|. For part (e): think about what other geometric properties indicate how "spread out" queens are.
