Multi-Armed Bandits

Prof. Sangeetha Viswanathan  ·  Sutton & Barto Ch. 2

Chapter one-liner

How do we find the best option while occasionally trying new ones? Every algorithm here is a different answer.

Food court analogyBiryani, Momos, Pizza, Dosa. Goal = max happiness. Always eat Biryani? Safe but you might miss something better. Always try something new? You learn a lot but never earn. This tension is the chapter.

Reward vs Value function

Reward = Present

Immediate feedback. The slap/high-five right now. "What happened this step?"

Value = Future

Long-term goodness. "Where does this path lead?"

Chess example — Prof.'s setup: win=+1, loss=−1, all other moves=0 Both move A and move B give reward=0 right now. Reward says: tie, can't decide.

V(S2)=7 → move A leads to checkmate in ~5 moves.
V(S3)=3 → move B leads to losing the queen in 3 moves.

Value picks move A. Reward was completely useless here.
Prof. Sangeetha (~23:41) "Value function is a long-term decision-making signal which will decide how good is it for you to be in a state and do a particular action."
Two flavours — both examinable V(s) — how good is it to be in state s.
Q(s,a) — how good is it to take action a in state s.

When you cannot model the environment (e.g. autonomous vehicle — can't write clean transition dynamics), use Q directly. Q "directly evaluates your policy." This is model-free RL's seed.
Exam mistake Value ≠ sum of rewards already received. Value = expected future return. Also: "fewer moves to reward" ≠ "higher value."

Exploration vs Exploitation

Exploitation — Earn

Cash in on what you know is good. Go to the best-known stall every day.

Exploration — Learn

Gamble on something unproven. Try the new stall even though it's risky.

Noodle stall example (Prof.'s) Biryani scores best on day 1 → you go only to Biryani forever. But noodle stall's chef was on leave that day and gave you a bad experience. You'll never discover it's actually better. "Always sticking with a greedy choice will never give you the options to learn."
The trap — both extremes fail Pure exploitation → locked in, misses the actual best.
Pure exploration → never cashes in on what's good.
Memory trick Explore = Learn  ·  Exploit = Earn

K-Armed Bandit setup

Simplest RL. One state. K actions. Pick one lever, get reward, repeat. Find which lever pays best over T steps.

Stationary ≠ Constant (Prof. spent ~15 min on this) Stationary = reward distribution stays fixed (machine always pays 10–100 range).
The actual reward varies each pull — that's noise, NOT non-stationarity.
Non-stationary = distribution shifts (Week 1: 10–100 → Week 2: 200–300).
Memory trick One-Armed Bandit = one slot machine. K-Armed Bandit = K slot machines.

True value q*(a) vs estimated value Q(a)

q*(a) — True value (unknown)

Expected reward if action a is taken forever. Hidden. What we're trying to approximate.

Q(a) — Estimated value (computed)

Our best guess from experience. Acts as proxy. Improves over time.

Goal: Q(a) → q*(a) as samples grow [Law of Large Numbers]

Sample average estimate

Q(a) = Σ {rewards from a} / (times a was taken)
Prof.'s live 4-machine example A1: rewards=[2], N=1 → Q=2.0
A2: rewards=[3,3], N=2 → Q=3.0 ← greedy picks this on step 6
A3: rewards=[1], N=1 → Q=1.0
A4: rewards=[1], N=1 → Q=1.0

Note: Prof. first said Q(A2)=2.5 assuming one reward, then corrected to 3.0 once both rewards were counted.
Common exam mistake Divide by per-action count only — never by total time steps. If A2 was pulled 3 times out of 10 total pulls, divide by 3, not 10.

Incremental update rule

Avoids storing all past rewards. Equivalent to sample average but O(1) memory.

Q_{n+1} = Q_n + (1/n)(R_n − Q_n) Derivation: Q_{n+1} = (1/n)(R_n + (n−1)Q_n) = Q_n + (1/n)(R_n − Q_n)
Universal RL one-liner — appears everywhere in RL! New estimate = Old estimate + Learning rate × Error
Error = (R_n − Q_n) = "how wrong was my guess." Positive → increase. Negative → decrease.

Non-stationary fix: constant step size α

For non-stationary problems, replace 1/n with a fixed α. Recent rewards matter more.

Q_{n+1} = Q_n + α(R_n − Q_n) [constant α] Unrolled: Q_{n+1} = (1−α)^n · Q_1 + Σ α(1−α)^(n−i) · R_i Weight on R_i = α(1−α)^(n−i) → older rewards decay exponentially
Recency-weighted Q computation — exam type! (α=0.5, Q init=0) Step 1, A=S, R=7: Q(S) = 0 + 0.5(7−0) = 3.5
Step 2, A=T, R=5: Q(T) = 0 + 0.5(5−0) = 2.5
Step 5, A=S, R=8: Q(S) = 3.5 + 0.5(8−3.5) = 3.5 + 2.25 = 5.75
Bias comparison — exam-critical α = 1/n (sample avg): satisfies Robbins-Monro Σα=∞ AND Σα²<∞ → converges, initial bias fully removed.

α = constant: satisfies Σα=∞ but NOT Σα²<∞ → keeps adapting, initial bias persists (geometrically shrinking but never zero). This is intentional for non-stationary problems.

Greedy action selection

Always pick the action with the highest current estimate. Pure exploitation.

A_t = argmax_a Q_t(a)
Warm-up rule Try every action at least once before starting greedy (otherwise untried actions stay at Q=0 forever and are never chosen).
The lock-in problem One unlucky sample permanently buries a good action. Noodle stall → chef on leave → low rating → greedy never returns → misses the actual best stall. "It's a never-ending loop." — Prof. Sangeetha

ε-Greedy action selection

With probability (1−ε): pick argmax Q (exploit). With probability ε: pick uniformly at random (explore). ε is a tunable hyperparameter.

Crucial subtlety — students often miss this During the explore branch, you pick uniformly over ALL K actions — the greedy action is still in the pool. Exploration does NOT remove the greedy action. This is what makes the probability derivation come out to 0.75, not 0.5.

P(greedy selected) — full derivation (K=2, ε=0.5)

P(greedy) = P(greedy|exploit)·P(exploit) + P(greedy|explore)·P(explore) = (1)·(1−ε) + (1/K)·ε = (1)(0.5) + (1/2)(0.5) = 0.5 + 0.25 = 0.75
Interpretation Exploit branch (prob 0.5) → always picks greedy → contributes 0.5
Explore branch (prob 0.5) → picks each of 2 actions at prob 1/2 → greedy gets an extra 0.25
Total = 0.75
General formulas (K actions) — memorise both P(greedy selected) = (1−ε) + ε/K
P(a specific non-greedy action selected) = ε/K

Quick check K=4, ε=0.2: P(greedy) = 0.8 + 0.05 = 0.85
Exam alert — almost-certain question Prof. promised to open the next class with this exact derivation. Classic wrong answer = 0.5 (forgetting greedy is still in the explore pool). Also know: ε=0 → pure greedy, ε=1 → pure random.

Step size α — three cases (Prof. quizzed the class on α=0)

α valueEffectProf.'s framing
α = 0No update ever"Will not converge, you will not learn anything at all"
α = constantFixed fraction each errorStandard hyperparameter, like in ML
α ↓ slowlyAdapts, never reaches 0"Tuning adaptively based on performance"

UCB — Upper Confidence Bound

Smart exploration: prefer actions we're most uncertain about. Not random — uncertainty-directed.

A_t = argmax_a [ Q_t(a) + c · √(ln t / N_t(a)) ]
TermNameRole
Q_t(a)Action value estimateExploit — how good we think this action is
cConfidence levelTunable knob — larger c = more exploration
√(ln t / N_t(a))Uncertainty bonusExplore — big when rarely tried, shrinks as N grows
Counter dynamics a selected → N_t(a) ↑ → bonus shrinks → "I know this one well now."
Other actions selected → t ↑ but N_t(a) fixed → a's bonus grows → "a is stale, revisit it."

N_t(a) = 0 → bonus = ∞ → UCB always tries every action at least once before comparing.
Numerical example — N=4, t=50, c=2 ln(50) ≈ 3.912 → 3.912/4 ≈ 0.978 → √0.978 ≈ 0.989 → bonus = 2 × 0.989 ≈ 1.98
UCB score = Q(a) + 1.98. Even a mediocre estimate can win the argmax if uncertainty is high.
ε-Greedy vs UCB ε-Greedy: explores randomly (coin flip, any action equally)
UCB: explores by uncertainty (targets least-understood actions)
UCB beats ε-greedy on bandits. But: harder to scale to deep RL with large state spaces.
Common exam mistake Getting the fraction inverted: ln t is in the NUMERATOR, N_t(a) is in the DENOMINATOR. Never swap.

Optimistic Initial Values

Set Q₁(a) = very high (e.g. +5) for all actions when true values ≈ 0. Use ε=0 (pure greedy). Exploration happens automatically through disappointment.

Disappointment mechanism — step by step 1. All Q = 5. Every arm looks equally amazing.
2. Greedy picks any arm (say A1). Expected +5, gets +1.
3. Q(A1) = 5 + 0.5(1−5) = 5 − 2 = 3 (with α=0.5).
4. A1 (Q=3) is now worth less than untried arms (still Q=5).
5. Greedy switches to a fresh arm. Repeat → all arms get visited several times.
Worked calculation — exam-style (Q₁=5, α=0.5, R₁=1) Q₂ = Q₁ + α(R₁−Q₁) = 5 + 0.5(1−5) = 5 − 2 = 3
Estimate dropped. Every untried arm still at Q=5. Greedy switches. That's forced exploration.
Limitation — why it fails for non-stationary (likely exam follow-up) The inflation is a one-time burst. Once all estimates decay toward reality, no mechanism remains to keep exploring. If the environment shifts after the initial phase, the agent is stuck. Stationary only, early training only.
Bias comparison — exam-critical Sample average (α=1/n): initial bias fully removed once each arm is tried at least once.
Constant α: initial bias persists (shrinks geometrically but never reaches 0).
→ This is why optimistic init + constant α doesn't cleanly reset.

From value-based to policy-based

All previous methods estimate Q(a) then derive a policy as argmax. Policy methods skip value estimation — directly learn which action to choose.

Key conceptual shift Value-based: Q(a) → argmax → action (indirect)
Policy-based: preference H(a) → softmax → probability → sample action (direct)
Why this matters long-term Softmax-over-preferences is a differentiable policy. Swap the K scalars for a neural net πθ(a|s) and the gradient-bandit update generalises into REINFORCE → A2C → PPO. The entire policy-gradient branch of deep RL starts here, in the bandit.

Preferences H(a) and Softmax

H(a) = learnable preference score. Not a reward, not a value — just a relative "how much I lean toward this action" knob. Higher H → chosen more often.

Hardmax vs Softmax

Hardmax = argmax H(a)

Deterministic. Winner takes all. No exploration. Not differentiable — gradient is undefined at the boundary.

Softmax (Boltzmann)

Smooth, probabilistic. Every action gets nonzero probability. Differentiable. Both exploratory and gradient-friendly.

π(a) = e^H(a) / Σ_{a'} e^H(a') [Softmax / Boltzmann distribution]
Worked example — H = (2, 0) e² ≈ 7.389, e⁰ = 1. Total = 8.389.
π(a1) = 7.389/8.389 ≈ 0.881  ·  π(a2) = 1/8.389 ≈ 0.119

a1 dominates (~88%) but a2 still gets ~12% — that residual IS the exploration.
Prof.'s example — H = (6, 9, 2) softmax → (0.047, 0.952, 0.00087)
a2 dominates (~95%). a1 gets ~5%. a3 keeps tiny <0.1% share. Not winner-take-all.
Key insight — softmax depends only on differences Adding a constant to every H leaves all probabilities unchanged. Only relative preferences matter.

Gradient Bandit — update rule

Differentiable policy → do gradient ascent on expected reward. Compare reward to a running baseline R̄ (mean of all rewards so far).

Chosen action At: H_{t+1}(At) = H_t(At) + α(Rt − R̄t)(1 − π_t(At)) All other actions a ≠ At: H_{t+1}(a) = H_t(a) − α(Rt − R̄t) π_t(a)
Tug-of-war logic Rt > R̄t (beat the average): push chosen action's preference UP, all others DOWN. "Better than expected — do it more."
Rt < R̄t (below average): push chosen DOWN, all others UP. "Worse than expected — back off."
Baseline R̄t — why it matters (testbed numbers) With baseline, α=0.1: ~85% optimal actions.
Without baseline, α=0.1: ~48% optimal actions.

Same algorithm, same step size. Adding the baseline roughly doubles performance by reducing gradient variance without introducing bias.

"Definitely vs Possibly ε" analysis

This question type appears in every single BITS DRL exam sitting. Given (step, action, reward) sequences, identify which steps definitely used ε and which possibly did.

The two rules Definitely ε Chosen action is NOT the current greedy action (argmax Q). It must have been a random pick — no other explanation.

Possibly ε Chosen action IS the current greedy action. Could be greedy (no ε fired) or got lucky during random branch (ε fired but randomly landed on greedy).
Step-by-step algorithm 1. Before each step: compute current Q(a) for all arms using the update rule.
2. Find argmax Q = the greedy arm. If tie, all tied arms count as greedy.
3. Compare chosen arm to greedy set:
   Chosen NOT in greedy set → Definitely ε
   Chosen IS in greedy set  → Possibly ε
4. Update Q for chosen arm. Proceed to next step.

Worked example — Dec 2025 paper style

4 arms: S, T, R, C. α=0.5 (constant step size). All Q init = 0. Sequence: (1,S,7) (2,T,5) (3,R,6) (4,C,4) (5,S,8) (6,T,6).

1
All Q=0 (all tied). A=S. S is in greedy set. Possibly ε
Q(S) = 0 + 0.5(7−0) = 3.5
2
Q: S=3.5, others=0. Greedy = S. A=T ≠ S. Definitely ε
Q(T) = 0 + 0.5(5−0) = 2.5
3
Q: S=3.5, T=2.5, R=0, C=0. Greedy = S. A=R ≠ S. Definitely ε
Q(R) = 0 + 0.5(6−0) = 3.0
4

Q(C) = 0 + 0.5(4−0) = 2.0

Definitely ε: steps 2, 3, 4, 6 Possibly ε: steps 1, 5 Best arm: S (final Q = 5.75, highest)

Three recurring question patterns (across all past papers)

Pattern 1 — Q computation + best arm Given (step, arm, reward), compute Q using sample-average or constant-α. State the best arm. Seen in: Dec 2025 Q3, Jul 2023 Q2, Jul 2024 Q3.
Pattern 2 — Definitely/Possibly ε trajectory analysis Same sequence as Pattern 1. Which steps definitely used ε? Which possibly? Requires correct Q update as prerequisite. Seen in: Dec 2025 Q3, Jul 2024 Q3.
Pattern 3 — Conceptual contrasts Reward vs value. MAB vs MDP. Optimistic init vs UCB (when each works/fails). Greedy better than ε-greedy (when?). TD update. Seen as short-answer sections in every sitting.
When is greedy better than ε-greedy? After training converges (deployment time, set ε=0). When the environment is stationary and already well-explored. On problems where the cost of exploration is high.

All key formulas

Sample average: Q(a) = ΣR / n Incremental update (stationary): Q_{n+1} = Q_n + (1/n)(R_n − Q_n) Constant step size (non-stationary): Q_{n+1} = Q_n + α(R_n − Q_n) ε-Greedy P(greedy): P(greedy) = (1 − ε) + ε/K P(non-greedy) = ε/K UCB: A_t = argmax[ Q_t(a) + c√(ln t / N_t(a)) ] Softmax policy: π(a) = e^H(a) / Σ e^H(a') Gradient bandit (chosen action At): H_{t+1}(At) = H_t(At) + α(Rt − R̄t)(1 − π_t(At)) Gradient bandit (all other actions): H_{t+1}(a) = H_t(a) − α(Rt − R̄t) π_t(a) TD update (value): V(s) ← V(s) + α[R + γV(s') − V(s)]
