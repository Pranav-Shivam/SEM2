Section 1: MAB Recap & Bridge to MDP 🔥 midsem hot

A1. The Pain That MDPs Exist to Remove

Imagine you're playing chess. You sacrifice a pawn on move 12. The payoff — whether that was genius or a blunder — might not be clear until move 40. How do you even rate that decision?

Why This Matters

Multi-Armed Bandits (MAB) can rate a slot-machine pull instantly — you pull, you see the reward. But in chess, a robot warehouse, or autonomous driving, the reward from one action ripples forward through every future state. The gap between "pull a lever" and "play chess" is the whole reason MDPs exist.

"We can rate one slot machine pull — but how do we rate a decision whose payoff only shows up 50 moves later? That gap is the whole reason MDPs exist."

Prof. Sangeetha Viswanathan — cs3 Session 1 Transcript

To get from the familiar MAB world to the full sequential decision-making world, we climb a three-rung ladder.

A3. Formalising the Full Picture: The MDP 5-Tuple

A Markov Decision Process is defined by five objects, written as the tuple ⟨S, A, P, R, γ⟩:

  • S — the set of all possible states (every situation the agent can be in)
  • A — the set of all possible actions (choices available to the agent)
  • P — transition probabilities: p(s', r | s, a)
  • R — the reward function (immediate payoff signal for each transition)
  • γ — the discount factor ∈ [0, 1]
Prof Annotation — Slide 5–6

"The boundary can be abstract and flexible. Always have clear signals for state, action, and reward. Time steps can be milliseconds (robot control) or days (stock trading). Actions can be voltages or voice commands."

A4. The SAR Sequence & Why Reward is Indexed t+1

Every interaction between agent and environment plays out as an alternating sequence:

S₀, A₀, R₁, S₁, A₁, R₂, S₂, A₂, ···

In English: state, then action, then reward from the transition, then new state, then next action, and so on forever.

Prof Annotation — Why R is Indexed t+1

"Reward is always indexed with t+1 because the state changes before the reward is observed. The agent acts at time t (takes At in state St), the environment responds at t+1 with both the new state St+1 and the reward Rt+1. The reward belongs to the transition, not the moment of action."

One complete (S, A, R, S') tuple is one sample — exactly like one labelled row in a supervised-learning dataset.

A6. The Markov Property — Why the Board Position Is Enough

For the SAR loop to be tractable, we need a crucial property: the future depends only on the current state, not on the full history.

p(St+1, Rt+1 | St, At) = p(St+1, Rt+1 | S0, A0, ···, St, At)

In English: knowing the full game history gives you zero extra information beyond knowing the current board — the present is a complete summary of the past.

Chess analogy: "The current board position tells you everything you need — you don't need to know every move that led to this position to decide your next move."

Prof. Sangeetha Viswanathan — cs3 Session 1 Transcript
Research Enhanced

The Markov Property Is a Design Requirement, Not a Simplifying Assumption

The Markov property is not just mathematical convenience — it is a constraint on how you must define your state. If you choose a state representation that omits relevant history (e.g., representing a chess position without castling rights, or a robot without its battery reading), standard MDP algorithms silently solve the wrong problem — a Partially Observable MDP (POMDP) — with no error message and no convergence failure to alert you. The state must be a sufficient statistic: conditioning on any additional history should give zero extra predictive power. This is why state engineering is often the hardest part of applying RL in practice.

Source: Sutton & Barto 2nd ed, Ch. 3

Self-Check — Click to Reveal

Q: What is the key difference between a contextual bandit and a full MDP? In Arjun's juice stall example, at what point does the stall become an MDP?

In a contextual bandit, the state (context) exists and influences which action is best, but your action does not change the state — the customer's age group is the same before and after you sell them a juice.

In a full MDP, actions cause state transitions. The juice stall becomes an MDP the moment an action changes the world: for example, if a customer's health state improves or declines based on which juice you sold them, and that health state affects what they'll buy next session — now you have a full MDP with sequential consequences.

One-line test: "Did my action alter what state I'll be in next?" If yes → MDP. If no → bandit.

Part B — Worked Examples

Identify the Setting: MAB, Contextual Bandit, or Full MDP?

For each scenario, decide which of the three levels it belongs to. Ask yourself: (1) Is there a state? (2) Does the action change the state?

  1. Scenario A: A doctor gives a patient one of three pain medications. There is no follow-up — just measure which drug gave the most immediate pain relief.
    MAB — one action, one reward, no state, no tomorrow.
  2. Scenario B: An e-commerce site recommends a product based on the user's age group. The recommendation earns a click or not. The user's age doesn't change because of the recommendation.
    Contextual Bandit — state (age group) exists but is unaffected by the action.
  3. Scenario C: A chess engine picks a move. The move changes the board position, which determines all legal future moves.
    Full MDP — the action (move) directly causes a state transition (new board position).
  4. Scenario D: A warehouse robot (battery = High) decides to search for cans. Searching drains battery, so next time it might be in state Low.
    Full MDP — action (search) causes a state transition (High → possibly Low battery).
  5. Scenario E: A gambler repeatedly pulls one of 5 slot machines, each with an unknown fixed payout distribution.
    MAB — classic K-armed bandit. No state, no transition, just repeated pulls.
Key Insight: The single question that separates all three levels is: "Does my action change the state I'm in?" No state at all → MAB. State exists but action doesn't change it → Contextual Bandit. Action changes state → Full RL / MDP.

Numerical: Constant α = 0.5 Update — Step-by-Step Working

This combines two concepts: the incremental update rule and the exponentially decaying weights interpretation.
Given: Q1 = 0, α = 0.5, rewards R1 = 10, R2 = 0, R3 = 5, R4 = 1.

Update rule (constant α — non-stationary / recent-reward form):

Qn+1 = Qn + α ( Rn − Qn )

In English: new estimate = old estimate + α × (what you just saw − what you expected). The term in brackets is your surprise; α controls how much you update on it.

  1. Compute Q2 using R1 = 10:
    Q2 = 0 + 0.5 × ( 10 − 0 ) = 0 + 5 = 5
  2. Compute Q3 using R2 = 0:
    Q3 = 5 + 0.5 × ( 0 − 5 ) = 5 − 2.5 = 2.5
  3. Compute Q4 using R3 = 5:
    Q4 = 2.5 + 0.5 × ( 5 − 2.5 ) = 2.5 + 1.25 = 3.75
  4. Compute Q5 using R4 = 1:
    Q5 = 3.75 + 0.5 × ( 1 − 3.75 ) = 3.75 − 1.375 = 2.375

The Exponentially Decaying Weights Interpretation

The formula can be "unrolled" to reveal that the constant-α update is equivalent to a weighted average with exponentially decaying weights on past rewards:

Qn+1 = (1 − α)n Q1 + α ∑i=1..n (1 − α)n−i Ri

In English: Q₁ gets shrunk by (1−α) raised to the power of n steps. Each past reward Ri gets weight α·(1−α)n−i — most recent reward has the highest weight, oldest reward has the lowest. With α=0.5: R₄ contributes α=0.5, R₃ contributes 0.25, R₂ contributes 0.125, R₁ contributes 0.0625.

Key Insight (Prof's words): "Recent rewards get the highest weight; oldest rewards get the lowest weight. This is why constant α is suitable when we want to 'trust' recent experience more" — especially useful for non-stationary reward distributions.

Design an MDP Formulation from Scratch — Chess Example

This tab walks through the complete reasoning process for formulating a real problem as an MDP, including a common wrong path that many students take. If this clicks, you're at distinction level.

  1. Step 1 — Identify the agent and environment.
    The chess engine is the agent. The chess game (opponent, rules, clock) is the environment. The boundary: the agent controls its own moves, but not the opponent's responses or the dice of luck. Everything outside its direct control is "environment."
  2. Step 2 — Define the state space S.
    First attempt (wrong path):
    Wrong Path — Common Mistake
    State = the sequence of all moves so far (move 1, move 2, ..., move k).
    Why wrong: This violates the Markov property. Storing the full move history makes the state space infinite and sequence-dependent. Two different histories can lead to the same board — you're storing redundant information. Also, MDP algorithms require a fixed-size state for each time step.
    Correct Path
    State = the current board configuration: piece types × squares, whose turn, castling rights, en-passant possibilities. The board position is a sufficient Markov state — given the board, you need no prior move history to decide the optimal next move (this is a direct consequence of the chess rules themselves).
  3. Step 3 — Define the action space A.
    A(s) = all legal moves from current board position s. Note that A depends on s (the set of legal moves changes with the position), so we write A(s), not just A.
  4. Step 4 — Define the reward function R.
    Second common wrong path:
    Wrong Path — Reward Misspecification
    R = +1 each time you capture an opponent's piece.
    Why wrong: The Reward Hypothesis says reward should encode the GOAL (win the game), not an intermediate proxy. Rewarding captures trains an agent to fall for sacrificial traps — it will eagerly take offered pieces even when doing so loses the game. (Prof's exact cautionary example from Slide 15.)
    Correct Path
    R = +1 for checkmate (win), −1 for being checkmated (loss), 0 for all other moves. This cleanly encodes "win the game" without specifying how. The agent discovers opening theory, sacrifices, and endgames on its own.
  5. Step 5 — Define transition probabilities P and discount factor γ.
    Chess against a fixed opponent: P(s'|s, a) is deterministic for the agent's move (given the agent's action, the board changes deterministically), but the opponent's reply adds stochasticity from the agent's perspective. In practice, the opponent's response is modelled as part of the transition.

    γ = 1.0 is appropriate here — chess is episodic (it ends at checkmate/stalemate/draw), so the discounted sum is naturally finite. There is no incentive to win "quickly" unless we add a step penalty.
  6. Step 6 — Write the complete 5-tuple.
    S = { all legal board positions (including castling rights, en passant) }
    A(s) = { all legal moves from board state s }
    P = deterministic for agent's move; stochastic for opponent's reply
    R = +1 (win), −1 (loss), 0 (all other transitions)
    γ = 1 (episodic task — game always terminates)
Key Insight — Why This Is Harder Than It Looks: State design is the hardest engineering decision in RL. Omitting castling rights from the chess state would break the Markov property — now the future (whether castling is legal) depends on history (whether the king has moved), not just the current position. Standard MDP algorithms would silently compute wrong values with no error. The state must be a sufficient statistic.

Common Mistakes (and Why They're Wrong)

  • Mistake 1 — Confusing MAB with MDP: Writing "MAB has states" or "contextual bandit is the same as MDP." Fix: MAB has no state at all. Contextual bandit has a state (context) but the action does not change it. MDP: action changes state.
  • Mistake 2 — Wrong reward index (Rt instead of Rt+1): The sequence S₀, A₀, R₀, S₁ is wrong. Fix: R arrives after the transition — it belongs to time t+1, written Rt+1.
  • Mistake 3 — Using 1/n step-size when α is required: In non-stationary environments, using 1/n means old rewards and new rewards have equal weight at large n. Constant α gives more weight to recent rewards — this is not optional for non-stationary problems. Fix: stationary → can use 1/n or α; non-stationary → must use constant α.
  • Mistake 4 — Forgetting Markov property requires sufficient state: Defining a state that omits history-relevant information (e.g., omitting battery level in a robot MDP) breaks the Markov property and makes the formulation incorrect. Fix: ask "does any relevant history exist that is not captured in my state?"

Section 2: MDP Formalism 🔥 midsem hot

Part A — Theory

Why Do MDPs Exist?

In the Multi-Armed Bandit you rated a single pull: pull arm, get reward, done. Simple. But consider a robot searching for cans: searching burns battery now, and a dead battery three decisions later means an expensive rescue. How do you rate the search decision — a decision whose true payoff only reveals itself 50 steps later?

That gap between action and consequence is the whole reason Markov Decision Processes exist. MDPs give us the mathematical scaffold to credit actions that pay off in the future.

The Markov Property

Before defining an MDP formally, we need its foundational assumption.

Prof. Sangeetha (transcript1.md): "Future depends only on the current state, not the full history." — cs3_1 Transcript, Markov Property section

Chess analogy: You don't need to replay every move that led to the current board position to decide your next move. The board as it stands tells you everything. That is the Markov property in action. Formally:

p(St+1, Rt+1 | St, At) = p(St+1, Rt+1 | S0, A0, … , St, At)
In English: the probability of where you end up and what reward you get depends only on where you are now and what you do now — not on the entire history of how you got here. The state must encode all decision-relevant history.
Research Enhanced State as sufficient statistic (S&B Ch. 3): From an information-theoretic view, a Markov state is a sufficient statistic of the past: conditioning on more history provides zero additional predictive power. This is why Bellman value functions can be defined over states alone — without it, we would need value functions over full histories, making planning intractable.

When this breaks: In a POMDP, the agent receives only a partial observation; observations alone do not satisfy the Markov property, so a memoryless policy is generally suboptimal. The fix is a belief state (a probability distribution over hidden states) — which restores Markov at the cost of a much larger state space. In deep RL, frame-stacking and recurrent networks are practical approximations.

The MDP 5-Tuple: ⟨S, A, P, R, γ⟩

An MDP is a precise mathematical object with five components. Every RL problem you will ever encounter is first translated into this language.

Symbol Name What it means Recycling Robot example
S State space All possible situations the agent can be in {High, Low} — battery charge level
A Action space All choices available to the agent (may be state-dependent: A(s)) High → {Search, Wait}; Low → {Search, Wait, Recharge}
P Transition model How the environment responds to actions — encodes all stochasticity From High + Search: stay High w.p. α, drop to Low w.p. 1−α
R Reward function Immediate numerical feedback; specifies what to achieve, not how rsearch > rwait > 0; −3 for emergency recharge

Agent–Environment Boundary — A Design Choice
The boundary between "agent" and "environment" is not a physical line — it is a design decision. Per Slide 7–8: the boundary marks the limit of the agent's absolute control, not the limit of its knowledge. Battery hardware is "environment" even though the robot knows its charge level, because the robot cannot directly rewrite the battery's physics. Time steps can be milliseconds (motor control) or days (stock trading). States can be raw pixels (Atari) or abstract symbols ("high battery").

Three Dynamics Equations — Why Each Exists

Prof. Sangeetha (transcript1.md): "Why do we need three separate equations? Because each answers a different question." — cs3_1 Transcript, Three Key Equations section

Equation 1 — Full Joint Model Dynamics

p(s', r | s, a)
In English: "What is the chance that action a in state s lands me in s' AND gives me exactly reward r?" This is the most complete description of the environment. It is a joint probability over (next state, reward) pairs.
Why it's needed: It captures everything — stochastic transitions AND stochastic rewards in a single object. All other quantities are derived from it.

Equation 2 — State-Transition Probability (Marginal)

P(s' | s, a) = Σr p(s', r | s, a)
In English: "What is the probability I end up in state s', regardless of what reward I receive?" Sum over all possible reward values to eliminate the reward variable — this is called marginalizing.
Why it's needed: When you only care where you land, not what you earn on the way. Used in state-transition diagrams and tables.
Warehouse robot analogy (transcript1.md): A warehouse robot moves to "reach customer" — it can arrive fast (reward +10) or slowly (reward +5) but still ends in the same next state "customer reached." To find the probability of reaching that state, you sum over both reward outcomes. — Prof. Sangeetha, cs3_1 Transcript, Equation 2 section

Equation 3 — Expected Reward for a Specific Transition

E[R | s, a, s'] = Σr r · p(s', r | s, a) / P(s' | s, a)
In English: "On average, what reward do I get when I go from s via action a and land in s'?" Numerator = weighted sum of all rewards that accompany this transition. Denominator = total probability of making this transition (normalizes correctly).
Why it's needed: Used in the Bellman equation to evaluate action choices — lets you compare "which action gives me the best average reward for each transition I might make?"
Common mistake: Confusing p(s', r | s, a) [joint distribution over both next-state and reward] with P(s' | s, a) [marginal distribution over next-state only]. The capital P is the sum of the lower-case p over all reward values. The lower-case p is the primitive; the capital P is derived from it. On exams: if a question asks for "the probability of reaching s'", use P (capital). If it asks for "the probability of landing in s' and getting reward r", use p (lower-case). Slide 6, S&B Ch. 3.

Why This Matters
Every RL algorithm you will ever study assumes MDP. Q-learning, Policy Gradient, Actor-Critic, DQN — all of them are algorithms for solving MDPs or approximating MDP solutions. Formulating the MDP correctly is Step 1 of any RL project. If you get S, A, R, or P wrong, the algorithm will solve the wrong problem — and give you no error message.
Research Enhanced The Joint Distribution p(s', r | s, a) — Why It Matters (S&B Ch. 3)

Defining reward as a separate function R(s, a) implicitly assumes reward depends only on the (state, action) pair — not on which next state was reached. The joint distribution p(s', r | s, a) is strictly more general: the same action from the same state can yield different rewards depending on which s' results. The warehouse robot example proves this: "reach customer fast" (+10) vs "reach customer slow" (+5) — same s', different r. P(s'|s,a) and E[R|s,a,s'] are both derived from the joint form via marginalization and conditioning.

Reward Hypothesis controversy (S&B p. 53): The hypothesis — that all goals can be formalized as maximizing a scalar cumulative reward — is the foundational bet of RL. Its limit: any fixed reward signal can be "gamed" (Goodhart's law). Recent work in RLHF and Inverse Reward Design represents practical responses to this limit. The debate about whether scalar reward is in principle sufficient remains open.

Section 3: Returns, Discounting & Task Types

Why cumulative reward — not just the next reward?

A chess engine gains a pawn on move 5 — but loses the game by move 40. If we score every decision only by its immediate payoff, the engine happily grabs material and walks into a forced checkmate. What we actually need is a number that captures all future consequences of a decision, appropriately weighted by how far away they are. That number is the return Gt.

Slide 3 — Objective: maximise the return (cumulative rewards) over time, NOT just the next reward.

The Return — Formal Definitions

Formula 1 — Discounted Return (infinite-horizon form)
Gt = Rt+1 + γRt+2 + γ2Rt+3 + … = Σk=0∞ γk Rt+k+1
In English: the return from time t is the sum of all future rewards, each shrunk by γ raised to its distance in the future. A reward two steps ahead counts for γ² of its face value.
Slide 17; Sutton & Barto 2nd ed, Ch. 3, §3.3
Formula 2 — Recursive Form (seed of Bellman — memorise this)
Gt = Rt+1 + γ Gt+1
In English: the value of this decision = the reward you get next, plus a discounted copy of the return from the very next step. The return is self-referential — each Gt contains Gt+1, which contains Gt+2, and so on.
Why this is the seed of Bellman: Replace Gt+1 with the expected return from state St+1 and you get the Bellman expectation equation for vπ — covered in Section 4. Every DP algorithm is just this one line, applied iteratively.
Slide 17 — annotated; Prof. Sangeetha — cs3_2 Transcript §Returns
Formula 3 — Constant Reward Simplification
If Rt+k+1 = R for all k  ⟹  Gt = R + γR + γ2R + … = R · 1 / (1 − γ) = R / (1 − γ)
In English: if you earn the same reward R every step forever, the total discounted payout collapses to R ÷ (1−γ) — a geometric series sum. Example: R = +1, γ = 0.9 → Gt = 1/0.1 = 10.
Slide 18; Prof. Sangeetha — cs3_2 Transcript §Constant Reward
Show geometric-series derivation ▾
Let S = R + γR + γ²R + …      (infinite sum)
Multiply both sides by γ:    γS = γR + γ²R + γ³R + …
Subtract:    S − γS = R
Factor:    S(1 − γ) = R
Divide:    S = R / (1 − γ)  ✓   (valid when |γ| < 1)
Formula 4 — Convergence Bound
|Gt| ≤ Rmax / (1 − γ)
In English: if rewards never exceed Rmax in magnitude, the return is always finite — no matter how long the sequence. This is the guarantee that discounting "tames" infinite horizons.

Two Roles of γ — Mathematical & Economic

Mathematical role: Convergence proof: |Gt| ≤ Rmax/(1−γ) means the infinite sum is always finite when γ<1 and rewards are bounded. More deeply, the Bellman operator T is a γ-contraction in the sup-norm: ‖Tu − Tv‖∞ ≤ γ‖u − v‖∞. Because γ<1, repeated application of T converges geometrically to a unique fixed point (v*) — guaranteed by the Banach fixed-point theorem. Without discounting (γ=1), this contraction property fails and convergence is not guaranteed in general. (Established result, Sutton & Barto 2nd ed, Ch. 3 & Ch. 4)

Economic / probabilistic role: γ can be read as the probability of surviving to the next time step in a task that may terminate randomly — making (1−γ) the per-step termination probability. Under this reading, maximising discounted return is equivalent to maximising undiscounted return in an episodic task with geometrically-distributed episode lengths — a formal duality. Financially, γ = 0.95 per step means tomorrow's reward is worth 95 cents today. (Established result, Sutton & Barto 2nd ed, Ch. 3, §3.3)

When γ=1 is truly desired (average-reward formulation): For continuing tasks where no time preference is appropriate (e.g. network routing), the correct formulation is average reward: r(π) = limT→∞ (1/T) E[Σ Rt]. This avoids the divergence of γ=1 discounted return. In practice, engineers use γ≈0.999 as a pragmatic approximation. (Established result, Sutton & Barto 2nd ed, Ch. 10, §10.3)

Why This Matters — Gt is the Root of Everything

  • Seed of Bellman: The recursive form Gt = Rt+1 + γGt+1 is not just a computational convenience. Take its expectation, condition on state, and you get the Bellman equation for vπ — which is the engine of Dynamic Programming (Section 4).
  • Discounting shapes policy: A small γ makes the agent greedy for immediate rewards; a large γ makes it patient. Choosing γ is a design decision with behavioural consequences, not just a convergence trick.
  • Continuing → episodic conversion: Many real-world tasks are continuing in nature. Understanding why and how to convert them (cart-pole example) is a practical skill tested directly in exams.

Section 4: Value Functions & Bellman Expectation Equation

Part A — Theory

The problem: you already know Gt = Rt+1 + γRt+2 + γ²Rt+3 + …  But simulating the infinite future for every state is computationally impossible. Can you evaluate how good a state is using only one step of lookahead?

Yes — by expressing the value of a state in terms of the value of the next state. That is exactly the Bellman Expectation Equation: evaluate infinite future via a one-step recursive bootstrap.

We will carry a single anchor example through every concept on this page: the 5×5 Gridworld with teleport states A and B, equiprobable policy π(a|s) = 0.25 for all four directions, and γ = 0.9. Intuition → formula → numerical verification — same grid, zero context-switching.

4.1   Policy π(a|s): the Agent’s Decision Rule

A policy π tells the agent what probability to assign to each action in each state:

π(a|s) = Pr{At = a  if  St = s}

In English: the probability the agent picks action a when it finds itself in state s. Must sum to 1 over all actions: Σa π(a|s) = 1 for every s.

Deterministic policy

π(a|s) = 1 for exactly one action, 0 for all others.
Example: “always go North in state s.”

Stochastic policy

Probability spread over multiple actions.
Example (gridworld): π(N|s) = π(S|s) = π(E|s) = π(W|s) = 0.25

4.2   Two Value Functions: vπ and qπ

Once we have a policy, we need to measure how good each state (and each state-action choice) actually is. Two functions do this job:

State-Value Function  vπ(s)
vπ(s) = Eπ[ Gt  |  St = s ]

Question answered: “How good is it to be in this state?”
Averages over all actions the policy might choose.

Gridworld: the number printed in each cell of the 5×5 grid IS vπ(s) under the random policy.

Action-Value Function  qπ(s,a)
qπ(s,a) = Eπ[ Gt  |  St = s ,  At = a ]

Question answered: “How good is it to take action a specifically?”
Fixes the first action, then follows π for the rest.

Gridworld: q(s,“move north”) tells you exactly how much value you get by committing to north from cell s.

Prof. Sangeetha — Session 2

“Action value is embedded inside state value. Some algorithms need only q(s, a) to directly derive a policy without needing the environment model. Others use v(s) but then need an extra step to select actions, which requires the model.”

Practical rule: model available? use vπ. Model unavailable (most deep RL)? use qπ directly — argmax over q gives policy, no model lookup needed.

4.3   The Bridge: v in Terms of q & q in Terms of v

The two value functions are not independent — each can be expressed in terms of the other. Together they give us the Bellman equation.

Equation 1 — v in terms of q  (Slide 22)
vπ(s) = Σa π(a|s)  •  qπ(s,a)

In English: the value of being in state s = the average action-value, where actions are weighted by how likely the policy is to choose them. A policy-weighted average of q values.

Gridworld check: v(s) = 0.25 × q(s,N) + 0.25 × q(s,S) + 0.25 × q(s,E) + 0.25 × q(s,W)

Equation 2 — q in terms of v  (Slide 22)
qπ(s,a) = Σs′,r p(s′,r|s,a)  • [ r + γ vπ(s′)]

In English: the action-value of taking action a from state s = the environment’s average of (immediate reward + discounted value of wherever you land). The environment’s randomness is averaged here via p(s′,r|s,a).

Gridworld check: q(s,“move east”) = 1.0 × [0 + 0.9 × v(east-neighbour)] (deterministic move, so p = 1 for exactly one s′)

4.4   Deriving the Bellman Expectation Equation (3-Step Sketch)

Substitute Equation 2 into Equation 1. Watch what happens:

Step 1 Start from the definition of vπ and expand Gt one level:
vπ(s) = Eπ[ Rt+1 + γ Gt+1  |  St = s ]

Using Gt = Rt+1 + γGt+1 — the recursive return from Section 03.

Step 2 Separate the expectation (linearity of E):
= Eπ[ Rt+1 ] + γ  Eπ[ vπ(St+1) ]

Because E[Gt+1 | St+1] = vπ(St+1) by the definition of vπ.

Step 3 Expand both expectations into explicit sums over actions and next states:
= Σa π(a|s) Σs′,r p(s′,r|s,a) [ r + γ vπ(s′)]

Recognise vπ(s′) on the right — this is the Bellman Expectation Equation. vπ appears on both sides: it is a self-consistency condition, not circular (see Research box below).

4.5   The Full Bellman Expectation Equation  🔥 midsem hot

Bellman Expectation Equation for vπ  (Slide 23; S&B 2nd ed., Eq. 3.14)
vπ(s) = Σa π(a|s) Σs′,r p(s′,r|s,a)  [ r + γ vπ(s′)]

In English: the value of state s (under policy π) = for every action a weighted by π(a|s), for every outcome (s′,r) weighted by the environment dynamics p(s′,r|s,a), take the immediate reward r plus the discounted value of wherever you land. One equation per state — solve the whole system and you have vπ.

4.8   Bootstrapping: the Key Idea

Why Bootstrapping Matters

Bootstrapping means updating an estimate using another estimate — specifically, using vπ(s′) (the next state’s current value estimate) to improve vπ(s) (the current state’s estimate). You never simulate the full infinite horizon; you just look one step ahead, trust your best current estimate of the successor’s value, and update.

Without bootstrapping: you’d need to roll out complete trajectories for every state — computationally impossible for large MDPs. With bootstrapping: one sweep through all states is enough to propagate value information.

This is why the Bellman equation is the engine of all RL algorithms: Policy Evaluation, Value Iteration, Q-Learning, TD-learning — all apply the Bellman operator in some form.

Research Enhanced
The Bellman Operator: γ-Contraction & Unique Fixed Point

The Bellman operator Tπ, defined by (Tπ v)(s) = Σa π(a|s) Σs′,r p(s′,r|s,a) [r + γ v(s′)], is a γ-contraction in the supremum (max) norm:

‖ Tπu − Tπv ‖∞  ≤  γ  ‖ u − v ‖∞

Because 0 ≤ γ < 1, every application of Tπ shrinks the gap between any two value-function guesses by a factor of γ. The Banach Fixed-Point Theorem then guarantees:

  1. A unique fixed point vπ exists such that Tπ vπ = vπ.
  2. Iterating Tπ from any starting point converges to vπ at a geometric rate (error shrinks by γ each sweep).

The self-reference in the Bellman equation (vπ appears on both sides) is not circular reasoning — it is a consistency condition defining a system of simultaneous linear equations whose unique solution is the true value function.

Established result — Sutton & Barto 2nd ed., Ch. 4.1; standard functional analysis (Banach 1922). Confirmed in research/c_d.md §C3.

Why This Matters
  • Bellman = foundation of all RL. Every algorithm you will study — Value Iteration, Policy Iteration, Q-Learning, TD(λ), DQN, Actor-Critic — is some variant of repeatedly applying the Bellman operator. Understand this equation, and the rest is engineering.
  • Without it, evaluation is impossible. Simulating Gt from scratch for every state requires infinite compute. The Bellman equation collapses that to a finite sum — one step, all outcomes, done.
  • v vs q is a model dependency choice. State-values need the model for policy extraction; action-values do not. This single distinction determines which algorithm class (model-based DP vs. model-free Q-learning) your problem calls for.

Section 5: Optimality & Bellman Optimality Equations

The Gap We Still Have

In Section 4 you built the Bellman Expectation Equation: given any policy π, you can compute how good every state is under that policy. But Section 4 never told you which policy to follow. You could evaluate a random policy perfectly and still be stuck going in circles.

The question this section answers: "Among the infinitely many possible policies, which one is best — and how do we describe 'best' mathematically?" That answer is the Bellman Optimality Equation. One tiny change (Σ → max) separates prediction from control, and the rest of modern RL flows from it.

What "Optimal" Means for Policies

Policies are compared through their value functions. We say policy ordering:

π ≥ π' ⇔ vπ(s) ≥ vπ'(s) for ALL s ∈ S

In English: π is "better than or equal to" π' only if it does at least as well in every single state — not just on average, not just in most states.

This is a partial order (not every pair of policies is comparable — π might beat π' in some states but lose in others). Yet a remarkable theorem holds:

Slide 28 — annotated: "There is always at least one optimal policy π*. There could be more than one!"

An optimal policy π* satisfies:

π* ≥ π for ALL policies π

In English: π* beats (or ties) every other policy in every state simultaneously.

Optimal Values: v* and q*

The optimal values are defined as the best achievable values across all policies:

Optimal state-value function:

v*(s) = maxπ vπ(s)

In English: v*(s) is the highest possible expected return from state s, achievable by the single best policy for that state — and it turns out one policy achieves this maximum everywhere at once.

Optimal action-value function:

q*(s, a) = maxπ qπ(s, a)

In English: q*(s, a) is the best expected return when you commit to action a from state s and then follow the optimal policy afterward.

Key relation between v* and q*:

v*(s) = maxa q*(s, a)

In English: the optimal state-value equals the best action-value you can get by choosing the optimal action from that state.

Multiple Optimal Policies — and Why a Deterministic One Always Exists

Prof. Sangeetha — cs3_2 Transcript §Optimality: "There can be multiple optimal policies — e.g., many paths reach the goal with the same maximum return. All such policies share the same optimal value function."

Why can ties happen? In a symmetric gridworld two distinct shortest routes to the goal may give identical total returns. Any policy choosing either route is optimal. They share the same v* and q* even though they differ in which actions they prescribe at some states.

The critical theorem (finite MDPs): Even though multiple π* can exist, at least one of them is deterministic — it picks a single action at each state with probability 1. You never need a randomised optimal policy in a finite MDP.

Why determinism is enough: Randomising among tied optimal actions cannot exceed v* (which is the ceiling). Determinism reaches the ceiling; randomisation only matches it at best. So we lose nothing by committing to one action.

Why Σ → max Is The Most Important Change in This Course

In the Bellman Expectation Equation, the agent averages over actions according to policy π: the coefficient π(a|s) is a fixed probability that does not depend on the current value estimates. That makes the whole expression linear in v: you can write every state's equation as a row in a matrix system Av = b and invert once to get the answer. The values are passive — they don't influence how actions are weighted, only the final output.

The Bellman Optimality Equation replaces that fixed-coefficient sum with maxa. Now the "coefficient" is not fixed at all — it is 1 for whichever action is currently estimated best, and 0 for all others. As value estimates change during iteration, which action wins the max can flip. The equation is no longer linear. Formally:

max(f(u), g(u)) + max(f(v), g(v)) ≠ max(f(u) + f(v), g(u) + g(v)) (in general)

Superposition fails. You cannot decompose the optimal problem into independent sub-problems and add their solutions. The whole system must be solved together, iteratively, until the max selections stabilise.
This is why Value Iteration replaces matrix inversion.

Bellman Optimality Equation for v*

Substitute the definition v*(s) = maxa q*(s, a) into the expression for q* and simplify:

Formula (Slide 29 — annotated):

v*(s) = maxa Σs',r p(s',r |s,a) [r + γ v*(s')]

In English: the optimal value of being in state s equals the expected return from the single best action you could take — where "expected" still accounts for the environment's stochastic transitions, but the agent no longer averages across actions; it picks the one that maximises.

Parsing the equation term by term:

maxa Agent picks the best action — this is the decision maker's choice. The max is what makes this the optimality equation, not the expectation equation.
Σs',r p(...) Environment averages outcomes — for the chosen action, nature randomly lands you in s' with reward r. The agent controls the action; the environment controls where it goes.
r + γ v*(s') One-step lookahead — immediate reward plus the discounted optimal value of wherever you land. This is why v* appears on both sides: it is a self-consistency (fixed-point) condition, not circular reasoning.

Bellman Optimality Equation for q*

For the action-value, you have already committed to action a in state s. The agent's next choice comes one step later, at s':

Formula (Slide 30 — annotated):

q*(s, a) = Σs',r p(s',r |s,a) [r + γ maxa' q*(s', a')]

In English: the optimal value of taking action a in state s equals the expected (immediate reward + discounted best-possible value from wherever you land) — and "best-possible" at s' means the agent again picks optimally using q*.

Parsing the equation term by term:

Σs',r p(...) Environment averages first — action a is already committed; the environment picks the next state s' and reward r stochastically.
r Immediate reward from the transition (s, a) → s'.
γ maxa' q*(s', a') Agent picks optimally at s' — after landing in s', the agent again exercises full control and selects the best available action. The max is INSIDE the expectation sum, not outside it.
Common error to avoid: Some students write maxa' outside the Σ — putting it there would mean the agent first averages all transitions and then picks the best action, which is wrong. The agent picks after the environment reveals s'; max must be inside the Σ.

Policy extraction via q* (model-free — preferred):

π*(s) = argmaxa q*(s, a)

In English: at state s, pick whichever action has the highest q* value. No model needed — q* directly encodes which action wins.

Policy extraction via v* (requires model):

π*(s) = argmaxa Σs',r p(s',r |s,a) [r + γ v*(s')]

In English: simulate each action using the model p(s',r|s,a), compute its one-step return using v*, pick the best one. You need p(s',r|s,a) to do this — which is why q* is preferred when the model is unavailable.

This extraction step is the bridge from "knowing v*/q*" to "acting optimally." Every RL algorithm — Value Iteration, Policy Iteration, Q-learning, DQN — ultimately exists to compute v* or q* so that this argmax can be taken.

Why This Matters

v*, q*, and π* are the final targets of all RL algorithms. Everything in this course — Value Iteration, Policy Iteration, Q-learning, Deep Q-Networks, Policy Gradient, Actor-Critic — is a different strategy for approximating one of these three quantities. Understanding what they mean and how they relate is the foundation every other technique rests on. When you read a research paper claiming "our algorithm converges to the optimal policy," it means converges to this π*.

Formal Guarantees: Contraction, Uniqueness, and Why Non-Linearity Forces Iteration

A deterministic optimal policy always exists in finite MDPs. The proof has two steps: (1) the Bellman optimality operator T* is a γ-contraction in the supremum norm — applying T* twice brings any two value estimates closer by a factor of γ. The Banach fixed-point theorem then guarantees T* has a unique fixed point v*. (2) The greedy policy with respect to v*, i.e., π*(s) = argmaxa Σ p(s',r|s,a)[r + γ v*(s')], achieves v* everywhere — a consequence of the Policy Improvement Theorem. Because argmax selects one action (ties broken arbitrarily), this policy is deterministic. v* and q* are unique even though multiple π* may exist.

Why the max operator breaks linearity — and why iteration is necessary. The Bellman expectation operator Tπ is linear: fixed weights π(a|s) mean the system Av = b can be solved by matrix inversion (for small |S|). T* introduces max: which action "wins" depends on the current value estimates, so the effective weight changes every iteration. The principle of superposition fails (max(f+g) ≠ max(f) + max(g) in general). No closed-form algebraic solution exists. Instead, Value Iteration applies T* repeatedly: each sweep is O(|S|²|A|) and error shrinks geometrically at rate γ per iteration.

Established result, Sutton & Barto 2nd ed., Ch 3.6 + Ch 4.1–4.4; Banach fixed-point theorem (standard functional analysis).

Medium: Write Both Bellman Equations + Explain the Σ → max Transition

This exercise builds the muscle memory for writing both optimality equations and explaining the structural difference in your own words.

  1. Start from the Bellman Expectation Equation for vπ (from S04):

    vπ(s) = Σa π(a|s) Σs',r p(s',r|s,a) [r + γ vπ(s')]

  2. Make the one change — replace Σa π(a|s) with maxa:

    v*(s) = maxa Σs',r p(s',r|s,a) [r + γ v*(s')]

    Note: vπ also changes to v* in the bracket — both sides.
  3. Write the q* equation from scratch: The action a is already fixed at the root. The environment averages over (s', r). Then the agent picks again optimally at s':

    q*(s,a) = Σs',r p(s',r|s,a) [r + γ maxa' q*(s',a')]

  4. Explain the Σ → max structural shift in one sentence: The expectation equation is linear (fixed policy weights π(a|s) are constants → matrix system Av = b → one-shot solution); the optimality equation introduces max (weights change as value estimates evolve → non-linear fixed-point problem → iterative solution required).
  5. Connect to algorithms: Bellman Expectation → Policy Evaluation (linear, iterable or invertible). Bellman Optimality → Value Iteration (apply max update repeatedly until convergence).
Check: In the v* equation the max is outside the Σs',r. In the q* equation the max is inside the Σs',r. These are different placements because the roles differ: for v* the agent picks the best action first and then the environment acts; for q* the action is already committed and the environment acts first, after which the agent picks optimally.
Key Insight: The two equations are not independent — they are two views of the same fixed point. Substitute v*(s) = maxa q*(s,a) into the q* equation and you recover the v* equation. They are mutually consistent.

Wrong Path — Using vπ(random) Instead of v*

Common mistake: A student who forgot to solve for v* first uses the random-policy values from S04 as a substitute:

Suppose the random policy gave: vrand(s1) = 4.0, vrand(s2) = 3.0, vrand(s3) = 0.

Q_wrong(s1, Left) = 1 + 0.9 × 4.0 = 4.6
Q_wrong(s1, Right) = 2 + 0.9 × 3.0 = 4.7 → picks Right ← WRONG

Why it fails: vrand measures how good each state is under the random policy, not optimally. Using it in the argmax produces a policy that is greedy with respect to a suboptimal value function — which may not be π* at all. Here it picks Right at s1 (going to the lower-value terminal quickly) instead of Left (staying in the high-value state).

Common Mistakes

Mistake 1 — Writing π inside the optimality equation:

v*(s) = Σ_a π(a|s) Σ_{s',r} p(...)[r + γ v*(s')]

Why wrong: The moment you take max, you are no longer following any fixed policy — the π(a|s) coefficient has no meaning in the optimality equation. The equation is asking "what is the best you can do?" not "what does π do?"

Mistake 2 — Placing max_{a'} OUTSIDE the Σ in q*:

q*(s,a) = max_{a'} Σ_{s',r} p(...)[r + γ q*(s',a')]

Why wrong: This would mean: average over all next states first, then pick the best action. But the agent sees which state s' they landed in before picking a'. The max must be inside the sum — the agent picks after observing the outcome of the environment's transition.

Mistake 3 — Using vπ instead of v* in policy extraction:

π*(s) = argmaxa Σ p(...)[r + γ vπ(s')] only gives a policy improvement step, not the optimal policy, unless vπ = v*.

Section 6: Dynamic Programming — Value Iteration, Policy Iteration & GPI

The Problem DP Solves

We know what v* looks like (Section 5's Bellman optimality equation). But that equation is self-referential — v* is on both sides. How do we actually compute it?

The answer: start with a guess (all zeros), then repeatedly apply the Bellman equation to update every state's value. Each iteration brings our estimate closer to v*. This is Dynamic Programming — and it only works when we have the full model dynamics.

"DP is a model-based method — it requires full knowledge of the MDP dynamics p(s',r|s,a). The general strategy: break the problem into subproblems, solve them, reuse solutions. The Bellman equation naturally fits DP because it bootstraps from next-state values." Slide annotated — S&B Ch. 4

Value Iteration (VI)

Value Iteration skips separate evaluation — it applies the Bellman optimality update directly from iteration 0:

Vk+1(s) ← maxa Σs',r p(s',r|s,a) [r + γ Vk(s')]
In English: new value of state s = reward + discounted best-reachable next value, for the best action. Uses previous iteration's Vk(s') — not the current one.
VI Algorithm (out-of-place / synchronous)
  1. Initialise V(s) = 0 for all states s (including terminal)
  2. Set Δ = 0 (tracks max value change this iteration)
  3. For each state s: compute Vnew(s) = maxa Σ p(s',r|s,a)[r + γ Vold(s')]
  4. Use Vold for all s' — never mix old and new within one iteration
  5. Update Δ = max(Δ, |Vnew(s) − Vold(s)|)
  6. If Δ < θ, stop. Else set Vold = Vnew and go to step 2
  7. Extract policy: π*(s) = argmaxa Σ p(s',r|s,a)[r + γ V(s')]
⚠️ Out-of-place critical (Prof. Sangeetha): "You must use the previous iteration's state values for V(s') when computing V(s). This is the out-of-place (synchronous) update. If you use the just-updated V(s') for a different s, that is in-place — different algorithm." Prof. Sangeetha — cs3_3 Transcript §VI Algorithm

In-Place vs Out-of-Place vs Asynchronous

Out-of-Place (Synchronous)

  • 2 arrays: Vold and Vnew
  • All updates this iteration use Vold
  • Reliable, predictable convergence
  • More memory, sometimes slower

In-Place (Asynchronous)

  • 1 array: overwrite immediately
  • V(W) may use just-updated V(C) — from same iteration
  • Often converges faster — uses freshest info
  • Order of state updates matters
Prioritised Sweeping: An asynchronous DP heuristic — update states with the largest Bellman error (biggest |Vnew(s) − Vold(s)|) first. Focuses computation where values are changing most. Accelerates convergence significantly. Prof. Sangeetha — cs3_3 Transcript §Async DP

Policy Iteration (PI)

Instead of merging evaluation and improvement, PI alternates them explicitly: fully evaluate a policy, then improve it, then repeat.

PI Algorithm
  1. Init: Choose arbitrary policy π0 (e.g., Cool→Slow, Warm→Slow)
  2. Policy Evaluation: Compute vπ(s) using Bellman expectation (no max), following the current policy's actions. Iterate until convergence (Δ < θ).
  3. Policy Improvement: Run one greedy step using the converged values:
    π'(s) = argmaxa Σ p(s',r|s,a)[r + γ vπ(s')]
  4. If π' = π (policy unchanged): stop — π is optimal. Else set π ← π' and go to step 2.

Generalised Policy Iteration (GPI)

Both VI and PI are instances of one unifying idea: interleave some evaluation with some improvement.

DP Limitations

  1. Requires a complete model — must know p(s',r|s,a) for every (s,a,s',r)
  2. Requires finite, discrete S and A — cannot handle continuous spaces directly
  3. Scales badly — O(|S|² × |A|) per iteration. Chess ~1044 states; Go ~10170 states. Physically impossible to tabulate.
Preview — Model-Free: When the model is unknown, we use Monte Carlo or TD learning to estimate values from experience instead of from transition tables. Still GPI — just with sampled evaluation instead of exact Bellman sweeps.

Value Iteration (NumPy, runnable)

import numpy as np

# Race car MDP: states 0=Cool, 1=Warm, 2=Overheated (terminal)
# P[s][a] = list of (prob, next_s, reward, done)
P = {
    0: {  # Cool
        0: [(1.0, 0, 1,  False)],                      # Slow
        1: [(0.5, 0, 2,  False), (0.5, 1, 2, False)],  # Fast
    },
    1: {  # Warm
        0: [(0.5, 0, 1,  False), (0.5, 1, 1, False)],  # Slow
        1: [(1.0, 2, -10, True)],                        # Fast
    },
    2: {  # Overheated (terminal)
        0: [(1.0, 2, 0, True)],
        1: [(1.0, 2, 0, True)],
    },
}

def value_iteration(P, gamma=0.9, theta=0.01):
    n_states = len(P)
    V = np.zeros(n_states)
    while True:
        delta = 0.0
        V_new = np.zeros(n_states)
        for s in range(n_states):
            action_vals = []
            for a in sorted(P[s].keys()):
                q = sum(
                    prob * (r + gamma * (0.0 if done else V[s_]))
                    for prob, s_, r, done in P[s][a]
                )
                action_vals.append(q)
            V_new[s] = max(action_vals)
            delta = max(delta, abs(V_new[s] - V[s]))
        V = V_new
        if delta < theta:
            break

    # Extract greedy policy
    policy = {}
    for s in range(n_states):
        action_vals = [
            sum(prob * (r + gamma * (0.0 if done else V[s_]))
                for prob, s_, r, done in P[s][a])
            for a in sorted(P[s].keys())
        ]
        policy[s] = int(np.argmax(action_vals))
    return V, policy

V, pi = value_iteration(P)
state_names  = ['Cool', 'Warm', 'Overheated']
action_names = ['Slow', 'Fast']
for s in range(3):

Policy Iteration (NumPy, runnable)

def policy_evaluation(policy, P, gamma=0.9, theta=1e-6):
    """Evaluate a deterministic policy until convergence."""
    n_states = len(P)
    V = np.zeros(n_states)
    while True:
        delta = 0.0
        for s in range(n_states):
            a = policy[s]
            v_new = sum(
                prob * (r + gamma * (0.0 if done else V[s_]))
                for prob, s_, r, done in P[s][a]
            )
            delta = max(delta, abs(v_new - V[s]))
            V[s] = v_new
        if delta < theta:
            break
    return V

def policy_iteration(P, gamma=0.9):
    n_states = len(P)
    policy = {s: 0 for s in range(n_states)}  # init: all Slow
    while True:
        V = policy_evaluation(policy, P, gamma)
        policy_stable = True
        for s in range(n_states):
            old_action = policy[s]
            action_vals = [
                sum(prob * (r + gamma * (0.0 if done else V[s_]))
                    for prob, s_, r, done in P[s][a])
                for a in sorted(P[s].keys())
            ]
            policy[s] = int(np.argmax(action_vals))
            if policy[s] != old_action:
                policy_stable = False
        if policy_stable:
            break
    return V, policy

V_pi, pi_pi = policy_iteration(P)

Research Enhanced Why VI converges — the contraction bound:

The Bellman optimality operator T* is a γ-contraction in the max-norm: ‖T*v − T*u‖∞ ≤ γ ‖v − u‖∞. After k iterations from any V0:

‖Vk − v*‖ ≤ γk × ‖V0 − v*‖

For γ=0.9, ε=0.01, Rmax=1: requires at least 66 iterations to guarantee ‖Vk−v*‖≤0.01. With Rmax=10 (race car's −10 terminal): ≥88 iterations. PI terminates in at most |A||S| outer steps — but in practice the race car needs only 1 outer step (one eval + one improve). Both are GPI instances.

Established result (Sutton & Barto 2nd ed, Ch 4.4)

Why This Matters: The conceptual chain is VI → Q-learning → DQN. Q-learning applies the Bellman optimality update with sampled transitions instead of the full model sum. DQN does the same with a neural network approximating Q*. Every deep RL paper you will read is VI with function approximation and without a model.
Self-Check: Why must VI use the previous iteration's V(s') values rather than the just-updated values (in the out-of-place version)?
Out-of-place VI updates all states simultaneously using a consistent snapshot of values from iteration k. If you used Vk+1(s') for some neighbors (already updated this iteration) but Vk(s') for others (not yet updated), you'd be mixing two different iterations — creating asymmetry depending on state ordering. Out-of-place guarantees that every Vk+1(s) is computed from the exact same Vk snapshot, making the update well-defined and reproducible regardless of state ordering. (In-place deliberately exploits this mixing for speed — but that's a different algorithm.)

  • You can explain the VI/PI tradeoff: VI = fast value convergence (one max sweep), no explicit policy mid-run; PI = full eval per outer step (expensive) but policy available early and fewer outer steps.
  • Watch for: mixing old/new values in out-of-place VI (exam trap), forgetting that DP needs a complete model (if model unknown → MC/TD), and knowing that GPI unifies VI/PI/Q-learning/TD as a single framework.

Complex: Policy Iteration Full Cycle — With Wrong Path

⚠ Wrong Path — Common Exam Mistake

A student runs "Policy Iteration" but during the Policy Evaluation step uses the Bellman optimality update (with max) instead of the Bellman expectation update (follow the policy):

Wrong eval of π0={Slow everywhere}:
V(C) ← max(Slow, Fast) = max(1+..., 2+...) — this is VI, not PE!

Why wrong: Policy Evaluation must use the policy's own action — not the best action. Using max during evaluation is Value Iteration, not Policy Iteration. The two algorithms are only equivalent at convergence; mid-run they produce different V estimates and different intermediate policies.

1Init π0: Cool→Slow, Warm→Slow (arbitrary)
2Policy Evaluation (Bellman expectation, no max):
For Cool (policy says Slow): V(C) ← 1.0×[1+0.9×V(C)]
For Warm (policy says Slow): V(W) ← 0.5×[1+0.9×V(C)]+0.5×[1+0.9×V(W)]
Solve these coupled equations iteratively (≈225 iterations).
Converges to: V(C)=10, V(W)=10
3Policy Improvement (one max step):
Cool: Fast q=0.5×[2+0.9×10]+0.5×[2+0.9×10]=11 > Slow q=1+0.9×10=10 → Fast
Warm: Slow q=0.5×[1+9]+0.5×[1+9]=10, Fast q=−10 → Slow
π1 = {Cool→Fast, Warm→Slow}
4Check stability: π1 ≠ π0 → run another PI cycle
Evaluate π1={Fast, Slow}, improve → same π1. Policy stable → STOP.
Optimal policy: Cool→Fast, Warm→Slow (matches VI result, found in 1 outer PI step)
Key Insight: PI uses Bellman expectation (Σ, follow policy) for evaluation, Bellman optimality (max) only in the improvement step. Mixing these up is the single most common PI exam mistake.
If this clicked: you're at distinction level. You understand the eval/improve boundary — the fundamental split that makes GPI work. If not, re-read Step 2 carefully and compare the Σ vs max operators.
Exam Alert Section 6 — Closed-Book Critical Points

VI Update Rule (memorise):

Vk+1(s) = maxa Σs',r p(s',r|s,a)[r + γ Vk(s')]

Stop when maxs |Vk+1(s)−Vk(s)| < θ. Policy = argmax at convergence.

PI Cycle (memorise):

  1. Eval: V(s) ← Σ p(s',r|s,a)[r+γV(s')] where a=π(s) — no max, follow policy action
  2. Improve: π'(s) = argmaxa Σ p(s',r|s,a)[r+γV(s')] — one max step
  3. Stop when π' = π

Common Mistakes:

  • ❌ Using Vk+1(s') (new values) instead of Vk(s') (previous) in synchronous VI
  • ❌ Applying max during Policy Evaluation (that's VI, not PE)
  • ❌ Forgetting to run PI until policy is stable (not just one cycle)
  • ❌ Saying DP works without a model — DP requires full p(s',r|s,a)

Out-of-Place vs In-Place:

Out-of-place = 2 arrays, synchronous. In-place = 1 array, order-sensitive, often faster. Prioritised sweeping = async + update largest Bellman error first.

GPI:

VI = 1 eval sweep + improve. PI = ∞ sweeps + improve. All RL algorithms (MC, TD, Q-learning, actor-critic) = GPI instances.

Deep Dive A

Two Bellman Equations: Expectation vs Optimality

The two Bellman equations differ by exactly one word — Σ vs max — but this creates a profound mathematical difference.

Unique solution?Yes (for each fixed π)Yes (v* is unique; multiple π* may share it)
AlgorithmPolicy Evaluation (iterative or matrix solve)Value Iteration
v vs qvπ = Σa π(a|s) qπ(s,a)v* = maxa q*(s,a)

Deep Dive B

Why DP Converges: Contraction Mapping & Banach Fixed Point

The Bellman optimality operator T* defined by (T*v)(s) = maxa Σ p(s',r|s,a)[r + γv(s')] is a γ-contraction in the sup-norm:

‖T*v − T*u‖∞ ≤ γ ‖v − u‖∞ for any value functions v, u
Applying T* brings any two value functions closer by factor γ

By the Banach Fixed-Point Theorem: a contraction on a complete metric space has a unique fixed point, and iteration from any starting point converges to it. The fixed point of T* is v*.

Error Bound and Iteration Count

‖Vk − v*‖∞ ≤ γk ‖V0 − v*‖∞
After k iterations, error shrinks by γ^k from initial error
Required iterations for ‖Vk−v*‖ ≤ ε: k ≥ log(ε(1−γ)/Rmax) / log(γ)

Worked Example: γ=0.9, ε=0.01, Rmax=1

k ≥ log(0.01 × 0.1 / 1) / log(0.9) = log(0.001) / log(0.9) = (−6.908) / (−0.10536) ≈ 65.6 → at least 66 iterations
Race car with R_max=10: ≥88 iterations. Both are feasible for small MDPs.

Source: Established result (Sutton & Barto 2nd ed, Ch 4.4)

Deep Dive C

VI vs PI vs Modified PI — The GPI Spectrum

All DP algorithms lie on a spectrum controlled by how many policy evaluation sweeps occur before each improvement step.

AlgorithmEval sweeps per outer stepEval operatorImprovement
Value Iteration1 (merged with max)T* (Bellman optimality)Implicit (max)
Modified PI (k)k (finite, fixed)Tπk (k applications of Bellman expectation)Greedy argmax
Policy Iteration∞ (full convergence)Tπ∞ = (I−γPπ)⁻¹ rπGreedy argmax
Practical implication: For large state spaces, VI is preferred (fast per iteration). For small state spaces where policy is needed quickly, PI is preferred. The k=1 and k=∞ extremes are algorithmic choices; real problems often benefit from intermediate k.
Deep Dive D

Model-Based → Model-Free: The Scaling Problem

DP requires storing and computing with p(s',r|s,a). The curse of dimensionality makes this impossible at scale:

DomainApprox state spaceTabular model?
Race car (course)3 states✅ Trivial
Chess~1044❌ Impossible
Go~10170❌ Exceeds atoms in observable universe (~1080)
Atari pixels~10100,000❌ Definitionally impossible

Two fixes:

  1. Function approximation: Replace tabular V(s) with a parametric function (linear, neural net) — enables DQN, PPO, A3C
  2. Model-free: Abandon the model; learn from sampled (s, a, r, s') transitions — enables MC, TD, Q-learning

Source: Established result (Sutton & Barto 2nd ed, Ch 4.1, 8.1)

Deep Dive E

The Reward Hypothesis — Power and Limits

Hypothesis: "All goals and purposes can be well thought of as maximisation of the expected cumulative sum of a received scalar reward signal." (Sutton & Barto, Ch 3.2)

Why it's powerful

Reduces any goal to a single scalar function — enabling a universal algorithm framework applicable to chess, robotics, trading, language model alignment.

Jan-2024 · Q1(a)2 Marks§1 RL Framework
List the four elements of Reinforcement Learning and give one formal statement on the role of each.
Policy π(a|s) — the agent's behaviour: a mapping from states to action (probabilities); it defines what the agent does.
Reward signal R — the immediate, scalar feedback defining the goal; the agent's sole objective is to maximise cumulative reward.
Value function v/q — long-run desirability: expected return from a state (or state-action), used to choose actions that are good in the long term, not just immediately.
Model p(s′,r|s,a) — the agent's representation of environment dynamics, used for planning (predicting next state & reward). (Optional — model-free agents omit it.)
Jul-2024(EN) · Q1(a,b,c)6 Marks§1 · Appendix D
(a) How do Unsupervised Learning and Reinforcement Learning differ?
(b) How do model-based and model-free RL differ, and when is each appropriate (≤3 statements)?
(c) Give one workplace use-case modellable as RL solvable by Dynamic Programming, with all identifying elements.
(a) Unsupervised learning finds structure/clusters in unlabelled data with no notion of action or reward. RL has an agent taking actions that affect a sequential environment, learning from a reward signal to maximise long-term return — there is feedback and consequence, not just pattern discovery.

(b) Model-based: the agent knows/learns p(s′,r|s,a) and plans (e.g. DP — VI/PI). Model-free: no model; it learns values directly from sampled experience (MC/TD — later chapter). (c) e.g. Datacentre cooling control: States = (temp band, load band); Actions = {raise, lower, hold} fan/coolant; Reward = −(energy + thermal-violation penalty); transition probabilities known/estimable and state space small & discrete → DP (Value/Policy Iteration) applies.

Jul-2024(EN-21) · Q4(b)3 Marks§3 Episodic tasks
Maze robot: reward +1 for escaping, 0 otherwise; treated as episodic. After training, no improvement in escaping. (1) What is going wrong? (2) How to fix?
What's wrong: with reward 0 everywhere except +1 at escape and no discounting (γ=1), every policy that eventually escapes gets the same return +1 — fast and slow escapes are scored identically, so there's no gradient pushing the agent to escape sooner. The reward conveys "escape" but not "escape quickly".

Fix: introduce a sense of time — use γ < 1 (so earlier escape = larger discounted return) or give −1 per time-step until escape (reward shaping that penalises delay). Either makes shorter paths strictly more valuable.

C · Value Functions & Bellman Equations

Dec-2025 · Q4(a)2 Marks§4 v vs q · Appendix D
In a model-free MDP, explain why estimating only v_π(s) is insufficient for selecting actions during control, even if v_π is perfect (2–3 statements).
To pick the best action greedily you must compare actions: a* = argmax_a Σ_{s′,r} p(s′,r|s,a)[r + γ·v_π(s′)]. This requires the model p(s′,r|s,a) to look one step ahead. In a model-free setting that model is unavailable, so a perfect v_π cannot be turned into an action choice. The fix is to estimate the action-value q_π(s,a) directly — then a* = argmax_a q_π(s,a) needs no model.
Jul-2024(EN-21) · Q4(a)2 Marks§4 Bellman · §5 Optimality
Recycling Robot (HIGH/LOW; search keeps HIGH w.p. α else LOW). Write expressions for q_π(high, search) and q_*(high, search), using the class notation.
q_π(high,search) = Σ_{s′,r} p(s′,r|high,search)[ r + γ·v_π(s′) ] = α[ r_search + γ·v_π(high) ] + (1−α)[ r_search + γ·v_π(low) ] = r_search + γ[ α·v_π(high) + (1−α)·v_π(low) ] q_*(high,search) = r_search + γ[ α·v_*(high) + (1−α)·v_*(low) ] where v_*(s) = max_a q_*(s,a) Difference: the expectation form uses v_π (value under the current policy); the optimality form uses v_* (value under the best policy, with the max baked in).
Jul-2024(EN-28) · Q4(b)2 Marks§4 Bellman recursion
Write a recursive expression to evaluate q_π(s,a). What is the significance of its recursive nature?
q_π(s,a) = Σ_{s′,r} p(s′,r|s,a)[ r + γ·Σ_{a′} π(a′|s′) q_π(s′,a′) ] Significance (bootstrapping): q_π is defined in terms of its own values at successor states, so the value of one state-action depends on the values of the next. This recursion (the Bellman equation) lets us solve for all values as a system / by iterative backups — we never have to roll out infinite trajectories; each estimate is built from neighbouring estimates.

Jan-2024 · Q4(a,b,c)6 Marks§6 PI · GPI
(a) What do you perform in Policy Evaluation and Policy Improvement; how do they estimate the optimal policy?
(b) High-level Policy Iteration algorithm (state- or action-values).
