Course Objectives & Learning Outcomes

Why this course exists

The course objectives boil down to three things: understand the conceptual and mathematical foundations of Deep RL (and its classic-to-state-of-the-art algorithms), implement and evaluate DRL solutions for planning/control/decision-making problems, and generalize — be able to model a brand-new real-world problem as a DRL problem.

Understand — conceptual + mathematical foundations of DRL, plus the algorithm landscape (classic and SOTA).
Implement & Evaluate — apply DRL to planning, control, and decision-making problems across domains.
Generalize — recognize new real-world problems and re-cast them as DRL problems (the "model real life problems" goal).

Learning Outcomes (the four exam-mapped pillars)

  • LO1 — Fundamentals: understand RL concepts/algorithms and apply them to control, decision-making, and planning.
  • LO2 — Implementation: implement DRL algorithms and handle training instability and convergence issues.
  • LO3 — Evaluation: judge DRL algorithms on sample efficiency, robustness, generalization — not just final reward.
  • LO4 — Real-world framing: understand challenges/opportunities of applying DRL to real-world problems and modelling new ones.

Why this matters

LO2 and LO3 are easy to skim past in lecture 1, but they describe the actual pain points of every later assignment: DRL training is famously unstable (LO2), and "did it learn?" is the wrong question — "how efficiently did it learn, and does it generalize?" (LO3) is the right one. Keep these four pillars as a checklist when you read any DRL paper or run any experiment this semester.

Exam Alert

Likely question: "List and briefly explain the learning outcomes of this course" or "What distinguishes evaluation of a DRL algorithm from evaluation of a supervised model?" — Answer using LO3: sample efficiency, robustness, and generalization, contrasted with simple accuracy/loss metrics in supervised learning.

What & Why is Reinforcement Learning?

Intuition first

RL is reward-based / feedback-based learning. A common misconception to kill immediately: RL is not a type of neural network, and it is not an alternative to neural networks. It's an approach to learning — a problem framing — that happens to use neural networks (Deep RL) when the state/action spaces get large. Classic applications: autonomous driving, gaming.

"Reinforcement learning (RL) is based on rewarding desired behaviors or punishing undesired ones. Instead of one input producing one output, the algorithm produces a variety of outputs and is trained to select the right one based on certain variables." — Gartner (as quoted in lecture)

When should you reach for RL?

RL is the right tool for large environments in (at least) these three situations:

  1. A model of the environment is known, but no analytic (closed-form) solution exists.
  2. Only a simulation model of the environment is available (this is the home turf of simulation-based optimization).
  3. The only way to gather information about the environment is to interact with it — there is no dataset to fall back on.

Why this matters

This three-way split is a quick diagnostic: if you have labelled data and no sequential decisions → supervised learning. If you have a clean closed-form model → classical optimization/control. RL earns its place specifically when (a) the problem is sequential, (b) feedback is reward-shaped rather than labelled, and (c) either the model is too complex for analytic solutions or doesn't exist at all and must be learned through interaction.

Formal definition

Sutton & Barto-style formal definition RL is an area of machine learning concerned with how intelligent agents ought to take actions in an environment in order to maximize the notion of cumulative reward.
Exam Alert

Likely question: "Is RL a type of neural network? Justify." — No. RL is a learning paradigm/approach defined by an agent maximizing cumulative reward through interaction; neural networks are merely one way to implement the function approximators (policy/value) inside an RL agent — hence "Deep RL".

RL vs. Supervised vs. Unsupervised Learning

The three ML paradigms differ in what signal they learn from and what question they answer:

CriteriaSupervised MLUnsupervised MLReinforcement ML
DefinitionLearns from labelled dataTrained on unlabelled data, no guidanceWorks by interacting with the environment
Type of dataLabelled dataUnlabelled dataNo predefined data
Type of problemsRegression & classificationAssociation & clusteringExploitation or exploration
SupervisionExtra supervisionNo supervisionNo supervision
AlgorithmsLinear/Logistic Regression, SVM, KNN…K-Means, C-Means, AprioriQ-Learning, SARSA
AimCalculate outcomesDiscover underlying patternsLearn a series of actions
ApplicationRisk evaluation, sales forecastRecommendation systems, anomaly detectionSelf-driving cars, gaming, healthcare

The paradigm view from the "(Deep) Reinforcement Learning" slide expresses each as a distribution / function the model represents:

Supervised pθ(y|x) → classification, regression
Unsupervised pθ(x) → inference, generation
Reinforcement πθ(a|s) → prediction, control

Read this as: RL learns a policy — a distribution over actions a conditioned on state s — parameterized by θ (e.g. neural network weights in Deep RL). Everything else in the course is about how to find good θ.

Characteristics of RL (the four things that make it hard)

  • No supervision — only a scalar reward signal, no "correct answer" labels.
  • Sequential decision-making — today's action shapes tomorrow's options.
  • Time matters — order of actions/observations is central, unlike i.i.d. supervised samples.
  • Delayed feedback — the consequence of an action may arrive many steps later (this is exactly the chess/value-function problem you'll meet in Session 2).
Research Enhanced

These four characteristics are precisely why RL cannot simply "borrow" supervised-learning theory: i.i.d. assumptions break (the agent's own policy changes the data distribution it sees — a moving target), and credit assignment across delayed rewards is an open research problem tackled by eligibility traces, TD(λ), and — in deep RL — techniques like reward shaping and n-step returns. — Sutton & Barto, RL: An Introduction, 2nd ed., Ch.1.

Worked examples

Q. A model predicts house prices from labelled (features → price) data. Which paradigm, and why?

Supervised learning — labelled data, the aim is to calculate an outcome (price), evaluated with regression metrics.

Q. A retailer groups customers by purchase patterns with no labels, to design promotions. Which paradigm — and how would the same retailer's "next best offer per customer, learned by trial and feedback over time" differ?

Grouping by patterns with no labels = unsupervised (clustering, e.g. K-Means) — aim is to discover structure. "Next best offer learned by trial/feedback over time" = RL — the system takes an action (an offer), observes a reward (purchase or not), and must balance trying new offers (exploration) vs. repeating known-good ones (exploitation).

Q. Why can't you simply treat an RL agent's experience stream as i.i.d. training data for a supervised model of "best action given state"?

  1. The data distribution depends on the current policy — change the policy, change the data.
  2. Rewards are delayed, so a "label" (was this action good?) isn't available at the time of the action.
  3. Samples are temporally correlated (consecutive states are similar), violating i.i.d.
Because (1) the policy-induced distribution shifts as the agent learns (non-stationary target), (2) the true "label" for an action — its long-term value — isn't known until much later (delayed feedback), and (3) consecutive samples are correlated, not independent. This is exactly why RL needs its own machinery (value functions, TD learning, replay buffers) rather than reusing supervised learning directly.
// 05

The Agent-Environment Loop

The picture you'll redraw all semester

Every RL problem is the same loop: an Agent observes a state, takes an action, and the Environment responds with a new state and a reward.

The loop, step by step (t = 0, 1, 2, …) Agent observes St ∈ S
Agent selects At ∈ A → forms pair (St, At)
Environment transitions to St+1 ∈ S
Agent receives reward Rt+1 ∈ R for taking At in St

The building-block walking example

The slides build intuition with a child learning to walk/ride a bicycle, and with video games:

  • Agent — an entity trying to learn the best way to perform a task (the child).
  • Action (A) — what the agent does at each time step ("walking"); A is the set of all possible moves (in a video game: run left/right, jump high/low, crouch, stand).
  • State (S) — the agent's current situation; after an action the agent moves to a different state (the child takes a step → new position).
  • Reward (R) — feedback on the action: positive if it's good / leads toward winning, negative (and vice versa) if not.
  • Environment — the outside/physical world the agent operates in.
Notation reminder. Be precise about the timing convention: the action At taken from state St produces reward Rt+1 and next state St+1 — the reward/next-state indices are t+1, not t. This off-by-one trips people up in every later derivation.
Exam Alert

Likely question: "Draw and label the agent-environment interaction diagram, defining S, A, R" — make sure your arrows show state/reward flowing from environment to agent, action flowing from agent to environment, and that Rt+1, St+1 are the result of At taken in St.

// 06

Policy, Value Functions & Model

Beyond agent and environment, an RL system has up to four sub-elements: a policy, a reward (already covered), a value function, and optionally a model of the environment.

Policy π — "what do I do here?"

The policy decides which action the agent takes in the current state. Formally, it's the probability that an agent will select a specific action from a specific state — a function mapping states to a probability distribution over actions.

Policy notation π(a|s) = probability that At=a given St=s, under policy π

Value Functions — "how good is this, really?"

A value function is a measure of how good it is for an agent to be in a given state, or to perform a given action in a given state. Two types are introduced (full math next session):

  • State-value function — goodness of being in state s.
  • Action-value function — goodness of taking action a in state s.

Model of the Environment — "what happens if…?"

A model mimics environment behaviour: given a state and action, it predicts the resultant next state and reward. Models are used for planning — deciding a course of action by reasoning ahead, before actually acting.

Why this matters

This four-way split — policy, reward, value, model — is exactly the axis along which the algorithm taxonomy (next section) is organized. "Model-free" methods skip the model entirely and learn policy/value directly from experience; "model-based" methods learn or are given a model and use it for planning. Every algorithm name you'll meet this semester (Q-learning, PPO, DDPG, AlphaZero…) is a specific recipe for combining these four pieces.

Exam Alert

Likely question: "Differentiate between a policy and a value function" — Policy = what to do (action selection rule, π(a|s)); Value function = how good a state/action is (used to derive a good policy, e.g. greedily). They're related but not the same object — a common mix-up is to call the value function "the policy".

Worked Example: Tic-Tac-Toe & Temporal Difference Learning

The setup

Two players alternate X/O on a 3×3 board; three in a row (any direction) wins. Crucially, the slides frame this against an imperfect opponent — one whose play is sometimes incorrect and lets you win. The aim: "How might we construct a player that will find the imperfections in its opponent's play and learn to maximize its chances of winning?"

Step 1 — A table of state values

Every possible board state gets an entry in a value table, initialized as follows (from our perspective):

StateInitial ValueMeaning
Empty / balanced opening board0.5~50% chance we eventually win
Board where we have won1.0Certain win → maximum value
Board where opponent has won0Certain loss → minimum value
… (all other states)… (typically 0.5 initially)Unknown — to be learned

Learning Task: Play many games against the opponent and update these values from experience.

Step 2 — Greedy moves through a game tree

At each "our move" point, we look at the values of the states reachable from each candidate move and pick the one with the highest value — this is the greedy move, marked with a star (★) in the game-tree diagram. The slides define:

  • St — the state before the greedy move
  • St+1 — the state after the greedy move

Step 3 — The Temporal Difference (TD) update

After making the greedy move, we nudge the value of the old state toward the value of the new state:

Temporal Difference Learning Rule V(St) ← V(St) + α [ V(St+1) − V(St) ]

Read it as: "new estimate = old estimate + step-size × error", where the error is the difference between two successive predictions V(St+1) − V(St) — hence "Temporal Difference": you learn from the difference between values at successive time steps, not from a final outcome alone. α is the step-size parameter, controlling how much each update shifts the estimate.

Why this is the seed of everything. Notice there's no "correct label" anywhere — V(St+1) is itself just another estimate, not ground truth. The agent is learning from its own predictions, bootstrapped one step into the future. This bootstrapping idea is the defining feature of TD methods (vs. Monte Carlo, which waits for the actual game outcome).

Discussion: the role of α

The slides pose three open questions about the step-size parameter — work through each:

If α → 0 gradually over many games: updates shrink toward nothing — the value table converges/stabilizes, "freezing" what's been learned. Good once the opponent's pattern is well-estimated and stationary.
If α is reduced but never reaches 0: learning continues indefinitely at a small rate — the agent keeps slowly adapting to any drift in the opponent's play, while being mostly stable.
If α is kept constant for life: every new game has a fixed-size influence — the agent stays maximally responsive to a non-stationary opponent (one whose strategy changes over time), at the cost of never fully "settling".

Key takeaways (as given in the slides)

  1. Learning happens while interacting with the environment (the opponent) — not from a pre-collected dataset.
  2. We have a clear goal (win the game / maximize V).
  3. Our policy makes moves that maximize the chance of reaching the goal — using the values of states most of the time (exploitation), and exploring the rest of the time.

Worked examples

Q. Apply the TD update: V(St)=0.5, V(St+1)=0.6, α=0.1. New V(St)?

V(St) ← 0.5 + 0.1 × (0.6 − 0.5) = 0.5 + 0.01 = 0.51
0.51 — the old estimate moved 10% of the way toward the new estimate.

Q. A board state leads, via the greedy move, to a state valued 1.0 (our win). If V(St)=0.5 and α=0.5, what's the new value, and after another identical update from the new value?

Step 1: V(St) ← 0.5 + 0.5×(1.0 − 0.5) = 0.5 + 0.25 = 0.75
Step 2: V(St) ← 0.75 + 0.5×(1.0 − 0.75) = 0.75 + 0.125 = 0.875
0.75, then 0.875 — each update halves the remaining gap to 1.0. This geometric convergence is why TD learning approaches but never instantly jumps to the "true" value.
Key insight: a large α (0.5) converges fast but is sensitive to noise; a small α converges slowly but smooths over noisy/inconsistent opponent play.

Q. The reading assignment asks: how is this TD-based solution different from (a) minimax and (b) genetic algorithms? Sketch an answer.

  1. Minimax assumes an optimal, adversarial opponent and searches the full game tree to guarantee a worst-case-optimal move — it needs a model of the game (rules/transitions) and typically full tree search at decision time.
  2. Genetic algorithms evolve a population of policies via selection/mutation/crossover across many "generations" (games), with no explicit value function per state — improvement is population-level and not driven by a per-step prediction error.
  3. TD learning (this example) learns per-state value estimates online, from a single agent's sequential experience, explicitly exploits an imperfect opponent (rather than assuming optimal play like minimax), and updates incrementally after every move via bootstrapped prediction error — no full tree search, no population.
Core distinction: minimax = worst-case game-tree search assuming a perfect opponent; genetic algorithms = population-level evolutionary search with no per-state value signal; TD = online, incremental, bootstrapped value learning from one agent's experience against a (possibly imperfect) opponent — explicitly designed to exploit weaknesses rather than assume none exist.
Exam Alert · likely numerical

Likely question: A numerical TD-update computation (given V(St), V(St+1), α — compute new V(St)), plus a conceptual question on the effect of α (constant / decaying / →0) — answer using the three-point discussion above. Also expect the minimax/GA comparison from the reading assignment to resurface.

Worked Example: Reward Design for Autonomous Driving

The slides give a complete (S, A, R, Objective) sketch for a self-driving car — your first taste of reward shaping.

ElementDefinition in this example
ObjectiveReach destination safely
StateImage from the front camera
ActionSteering, acceleration, brake
RewardForward speed (base signal), shaped by the events below
Reward shaping table Drive off road = −10
Hit a person = −1000
Drive at speed limit = +2
Align with lane = +4
Run a stop sign = −5
Erratic driving = −3
…

Why this matters

This table is a masterclass in reward design by relative magnitude. "Hit a person" at −1000 isn't just "bad" — it's orders of magnitude worse than every other penalty combined, so the agent can never learn to trade off small gains (e.g. +2 for speed) against that catastrophic outcome. Meanwhile +2 (speed limit) vs +4 (lane alignment) tells the agent that staying in lane matters more than going fast. Badly scaled rewards are one of the most common real-world DRL failure modes — an agent will ruthlessly optimize exactly the number you give it, including loopholes you didn't intend ("reward hacking").

Worked examples

Q. In one episode, the car drives at the speed limit for 3 steps (+2 each), aligns with lane for 2 steps (+4 each), and runs one stop sign (−5). What's the total reward?

3×(+2) + 2×(+4) + 1×(−5) = 6 + 8 − 5 = +9
+9 — sum the per-step rewards across the episode (this is the "cumulative reward" from the formal RL definition).

Q. Suppose "drive off road" were changed from −10 to −1 (smaller magnitude). What unintended behaviour might emerge, given +2/+4 are still on offer for speed/lane?

If the road-departure penalty (−1) is smaller than achievable positive rewards (+2 speed, +4 lane), the agent could learn to cut corners off-road to gain a better lane-alignment/speed combination on the far side, since the small −1 penalty is outweighed by +2/+4 gains. This is "reward hacking": the agent optimizes the literal reward sum, not the intended objective ("reach destination safely").
Key insight: penalty magnitudes must dominate any combination of rewards that could be earned by the unsafe shortcut — relative scale, not just sign, encodes intent.

Q. Map this example onto the formal RL definition (S, A, R, Environment, cumulative reward) and onto the Objective vs. Reward distinction — why might "forward speed" alone (the base reward) be a poor proxy for the stated Objective ("reach destination safely")?

  1. S = front-camera image, A = {steer, accelerate, brake}, Environment = road/traffic/physics, R = shaped scalar combining speed + event bonuses/penalties.
  2. Cumulative reward = sum of R over the whole drive (the quantity the agent is trained to maximize).
  3. "Forward speed" alone rewards going fast, which is correlated with but not equivalent to "reaching the destination safely" — a purely speed-maximizing agent would have no signal telling it to stay on the road, avoid pedestrians, or stop at signs.
The Objective ("reach destination safely") is the true goal, but it's not directly differentiable/observable per step. The reward function is an engineered proxy for that objective — "forward speed" alone under-specifies safety, so the table adds explicit penalty/bonus terms (off-road, hit person, stop sign, lane alignment, erratic driving) to pull the proxy back toward the true objective. This Objective-vs-Reward gap is exactly the reward-design problem you'll revisit throughout the course.
Exam Alert

Likely question: "For [some new domain — e.g. a delivery drone / warehouse robot], specify the state, action, reward, and objective, and design a reward table with appropriate relative magnitudes." — Practice this pattern: 1 sentence Objective, concrete State/Action definitions, then a reward table where catastrophic outcomes dominate by orders of magnitude and desirable behaviours are positively but moderately weighted relative to each other.

Easy21-Dec-2025 · Q1(a) · 1.5M

PE1. A mobile health-monitoring wristband chooses among three modes each hour (low/medium/high power, increasing accuracy), receiving an immediate benefit (reward) based on the user's activity. State why Reinforcement Learning (not supervised learning) fits this setting, and give a one-line distinction between immediate reward and long-term value.

Why RL fits: there is no dataset of "correct mode per situation" to learn from — the wristband must improve its mode-selection by interacting with the user over time and observing a scalar reward (benefit) for each choice. This is exactly RL's signature: no supervision (only a reward signal, no labels) and sequential decision-making (today's mode can shape tomorrow's situation, e.g. battery state).

Immediate reward = the one-step feedback Rt+1 for the action just taken; long-term value V(s) = the expected cumulative (future) reward from state s onward — it accounts for downstream consequences the immediate reward alone misses.
Key insight: this is the policy-vs-value-function distinction from Section 06, restated in plain terms — "what just happened" vs. "what this leads to".
Hard · Flagship27-Jan-2024 · Q1(a-c) · 6M

PE2. Consider tic-tac-toe. (a) [2M] List the four elements of Reinforcement Learning and give one formal sentence explaining the role of each. (b) [0.5M] Write the Temporal Difference (TD) rule for learning each state's value; explain the elements and how it works. (c) [2M] The current state's value is 4.5, and its successor state's value is 2.7. Using 0.9 as the parameter from part (b), revise the estimate of the current state's value. Explain.

(a) The four elements (Section 06):

  • Policy π(a|s) — the probability the agent selects action a in state s; the rule mapping states to a distribution over actions ("what to do").
  • Reward R — the scalar feedback signal received after each action, defining the goal the agent is trying to maximize.
  • Value function V(s) / Q(s,a) — the expected cumulative future reward from state s (or from taking action a in s); measures "how good", and is used to derive a good policy.
  • Model (optional) — a predictor of the next state and reward given (s,a); mimics environment behaviour and is used for planning.

(b) TD rule:

V(St) ← V(St) + α [ V(St+1) − V(St) ]

α is the step-size parameter (0<α≤1) controlling how big each update is. V(St+1)−V(St) is the TD error — the difference between two successive predictions. The rule nudges the old estimate toward the new one — it bootstraps from its own next-step prediction rather than waiting for the game's final outcome.

(c) Apply the rule with V(St)=4.5, V(St+1)=2.7, α=0.9:

TD update V(St) ← 4.5 + 0.9 × (2.7 − 4.5)
= 4.5 + 0.9 × (−1.8)
= 4.5 − 1.62 = 2.88
New V(St) = 2.88
Key insight: α=0.9 is large — the estimate jumps 90% of the way from 4.5 toward 2.7, landing close to the successor's value. A small α would move it only slightly toward 2.7.
Easy28-Jul-2024 · Q1(a) · 2M

PE3. In what ways are Unsupervised Learning and Reinforcement Learning different?

  1. Data: Unsupervised learning operates on a fixed, unlabelled dataset to find structure within it. RL has no predefined dataset — its "data" (experience) is generated on the fly through interaction with the environment.
  2. Aim: Unsupervised learning's aim is to discover underlying patterns (e.g. clustering, association) — a largely one-shot mapping. RL's aim is to learn a series of actions (a policy) that maximizes cumulative reward, balancing exploration vs. exploitation.
  3. Feedback: Unsupervised learning has no feedback signal at all. RL has a scalar reward signal — often sparse and delayed — that drives learning.
Core distinction: unsupervised = find structure in static unlabelled data with no feedback; RL = learn a sequential policy from a reward signal obtained by acting in an environment.
Medium28-Jul-2024 · Q1(b) · 2M

PE4. How are Model-based and Model-free reinforcement learning algorithms different? What scenarios are these approaches appropriate for (in not more than three statements)?

Difference: Model-free algorithms learn a policy and/or value function (π, V, or Q) directly from experience, without ever representing the environment's transition/reward dynamics — e.g. Q-Learning, SARSA. Model-based algorithms learn (or are given) a model that predicts the next state and reward for a given (state, action), and use that model for planning/lookahead before acting — e.g. Dynamic Programming, AlphaZero.

Scenarios (≤3 statements):

  1. Model-based is appropriate when the environment's dynamics are known exactly or can be learned accurately (e.g. board games with fixed rules) — planning with the model reduces costly real-world trial-and-error.
  2. Model-free is appropriate when dynamics are too complex or unknown to model reliably, but interaction/experience is cheap and plentiful (e.g. video games, simulators, robotics).
  3. Hybrid/actor-critic methods (DDPG, TD3, SAC) bridge both — they learn a model-free critic alongside a policy, without needing an explicit dynamics model.
Medium21-Jul-2024 · Q1(b,c) · 3M+1M

PE5. Consider the RL agent learning tic-tac-toe by playing against different randomly-chosen opponents, using V(St) ← V(St) + α[V(St+1) − V(St)]. (b) [3M] Can α be used to encourage exploration? Why or why not? (c) [1M] Name one Model-Based and one Model-Free RL algorithm covered so far.

(b) No.

α (the step-size) only controls how strongly the value estimate is updated toward a newly observed estimate — it governs the speed/responsiveness of learning, not which states or actions get visited. Exploration is determined by the action-selection mechanism (e.g. ε-greedy, or here, the randomly-chosen opponent's moves), entirely independent of α. A large α makes the agent adapt quickly to a non-stationary opponent (Section 08's discussion), but it does not by itself cause the agent to try new states — it can even amplify noise from rarely-visited states without ever causing more visits to them.

(c) Model-Free: Q-Learning (or TD-learning / Monte Carlo). Model-Based: Dynamic Programming / Value Iteration (or AlphaZero — "Given the Model").

Key insight: don't confuse the learning-rate knob (α) with the action-selection/exploration knob (ε, or the opponent's randomness) — they control entirely different things.
Medium28-Jul-2024 · Q3(a) · 2M

PE6. In the Gridworld example, rewards are positive for goals, negative for running into the edge of the world, and zero most of the time. Are the signs of these rewards important, or only the intervals between them? Explain.

Both matter, but for different reasons:

  • The sign gives a basic qualitative direction — goal=good (positive), wall-hit=bad (negative), neutral=0 — so that comparisons between value estimates push the policy toward goals and away from penalties.
  • What actually drives the learned policy is the relative magnitude/interval between rewards: the TD rule and value comparisons only ever look at differences, so a reward landscape uniformly shifted by a constant largely preserves the same relative ordering of actions (the gaps encode "how much better" one outcome is than another — Section 09's "relative magnitude" reward-design principle).
  • Caveat: for episodic tasks, adding a constant to every step's reward is not entirely free — e.g. a small negative reward on every non-terminal step creates pressure to finish quickly, which can change the optimal policy. So intervals dominate, but the sign/zero-point convention isn't fully arbitrary either.
Mainly the intervals (relative magnitudes) — they encode the preference ordering the agent learns from; signs set the qualitative direction but a uniform shift mostly preserves behaviour (except where it changes episode-length incentives).

EasySample · TD Update

SQ1. Apply the TD update rule V(St) ← V(St) + α[V(St+1) − V(St)] given V(St)=0.6, V(St+1)=0.9, α=0.4. Compute the new V(St). Then explain qualitatively what would change if α were instead 0.05.

V(St) ← 0.6 + 0.4 × (0.9 − 0.6) = 0.6 + 0.12 = 0.72
New V(St) = 0.72

With α=0.05: V(St) ← 0.6 + 0.05×0.3 = 0.615 — a much smaller step. The estimate moves slowly toward 0.9, but is more robust to noisy/inconsistent feedback (smoothing), whereas α=0.4 converges faster but is noisier.

EasySample · Paradigm Classification

SQ2. Classify each scenario as Supervised, Unsupervised, or Reinforcement Learning, with a one-line justification: (1) predicting loan-default risk from a labelled history of past loans (default / no-default). (2) grouping warehouse SKUs into clusters by co-purchase pattern, with no labels. (3) a robot arm learns to grasp objects by trying different grips and receiving a success/failure signal each attempt, improving over thousands of trials.

  2. Unsupervised — unlabelled data; aim is to discover underlying structure (clustering/association).
  3. Reinforcement Learning — no labelled "correct grip"; the agent interacts with the environment and improves via reward (success/failure) feedback across sequential trials (exploration vs. exploitation).
MediumSample · Algorithm Taxonomy

SQ3. Classify each algorithm into the RL taxonomy (Model-Free → Policy Optimization / Model-Free → Q-Learning / Model-Based → Learn the Model / Model-Based → Given the Model) and briefly justify: (a) Vanilla Policy Gradient, (b) DQN, (c) World Models, (d) DDPG.

  • (a) Policy Gradient — Model-Free → Policy Optimization: directly parameterizes and optimizes π(a|s) via gradient ascent on expected return; no environment model.
  • (b) DQN — Model-Free → Q-Learning: learns Q(s,a) from experience (replay buffer), derives the policy greedily; no model.
  • (c) World Models — Model-Based → Learn the Model: learns a predictive model of environment dynamics from experience, then plans/imagines rollouts inside it.
  • (d) DDPG — bridges Policy Optimization and Q-Learning (actor-critic): learns a deterministic policy (actor) and an action-value critic; still Model-Free overall (no learned dynamics model).
HardSample · Reward Design

SQ4. A delivery drone must fly from a depot to a customer's location and drop a package. Objective: deliver quickly and safely. Specify State, Action, and a reward-shaping table with at least 5 entries, ensuring catastrophic outcomes dominate by relative magnitude. Then describe one possible reward-hacking failure mode of your design.

ElementDefinition
StateGPS position, altitude, battery level, obstacle-sensor readings
Actionpitch/roll/yaw adjustment, throttle, release package
Reward-shaping table Collision with obstacle/building = −500
Battery depleted mid-flight (forced landing) = −200
Package delivered at correct location = +100
Each step's net progress toward destination = +1
Flying outside designated air corridor = −20
Erratic flight (excess altitude change) = −2
Key insight: −500 / −200 dwarf any achievable sum of +1-per-step progress rewards, so the agent can never trade safety for speed — same "dominant penalty" principle as the autonomous-driving example (Section 09).

Reward-hacking failure mode: if the +1 "closer to destination" reward is summed without bound and the −20 corridor penalty is small relative to many such steps, the drone could learn to oscillate in/out of the corridor to repeatedly rack up small +1 progress rewards rather than delivering efficiently — or hover near (but not at) the destination to keep collecting proximity rewards without ever releasing the package. Fix: reward progress only on net distance decrease since the last step, and make the terminal success reward dominate the sum of step rewards.

MediumSample · Conceptual Short-Answers

SQ6. Answer briefly: (a) True/False — "RL is a type of neural network architecture." Justify. (b) Distinguish a policy from a value function in one sentence each. (c) True/False — "A constant step-size α (never decaying to 0) is preferable for a non-stationary environment." Justify. (d) Give one example each of a model-free and a model-based algorithm.

(a) False. RL is a learning paradigm — an agent maximizing cumulative reward through interaction; neural networks are merely one way to implement the function approximators (policy/value) inside an RL agent, hence "Deep" RL.

(b) Policy π(a|s) = the rule for what action to take in a state (a probability distribution over actions). Value function V(s)/Q(s,a) = an estimate of how good a state (or state-action pair) is, i.e. expected cumulative future reward.

(c) True. A constant α gives every recent observation a fixed (geometrically-decaying-in-the-past) weight, keeping the agent maximally responsive to drift in a non-stationary environment — at the cost of never fully converging/settling.

(d) Model-free: Q-Learning (or TD-learning). Model-based: Value/Policy Iteration (Dynamic Programming), or AlphaZero ("Given the Model").
